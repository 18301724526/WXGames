## 2026-06-23 战斗相机渲染职责拆分

现象：实体战斗场景缩放或拖拽时，角色单位会跟随相机变化，背景图保持全屏固定 cover 绘制，导致单位看起来在背景上滑动，缩放越大越明显。

边界判断：

- 问题属于前端 canvas 战斗渲染职责拆分，不涉及后端战斗会话、战斗模拟、API 或输入指令流。
- `BattleCameraPolicy` 已经负责相机数学，问题不是相机公式本身，而是背景层没有接入同一套战场世界坐标到屏幕坐标的 transform。
- HUD、底部军令按钮、结算弹层和按钮热区仍属于屏幕 UI 层，不进入 battle camera。

职责拆分结果：

- `BattleCanvasRenderer.getEntityBattleRenderContext` 统一生成实体战斗的 `layout`、`stage`、`arena`、`fit`、`camera` 和 `transform`。
- `drawEntityBattleBackground` 和 `drawEntityBattleField` 共享同一个 render context。
- 背景图在 stage clip 内按照 `arena * transform.scale` 和 `transform.offsetX/offsetY` 绘制。
- 单位继续按照同一个 `transform` 绘制世界坐标，避免背景和单位进入不同坐标系。
- `drawEntityBattleTopHud`、`drawEntityBattleControls`、`drawEntityReplayControls` 和 `drawEntityBattleResult` 保持屏幕坐标。

## 2026-06-23 多语言文本协议首轮收口

目标：把用户可见文案从渲染/投影代码里拆出，建立代码只引用语义 key、语言配置负责各语种展示的统一入口。首轮不做全仓字符串替换，避免把测试 fixture、协议枚举、日志、内容数据名和 vendor 文本误收进 UI i18n 范围。

职责边界：

- `LocaleTextRegistry` 负责维护按 locale 分组的稳定文案表，当前包含 `zh-CN` 默认表和 `en-US` 对照表。
- `LocaleText` 负责运行时语言选择、`t(key, params)` 参数替换和缺失 key fallback。
- 渲染层只通过 `t()` 读取用户可见 UI 文案，不再在 HUD 分支里创造英文 fallback。
- `WorldActorProjection` 对未命名敌军只输出 `nameKey/labelKey`，不再把默认英文名写进投影数据；后端或内容配置明确传来的 encounter `name` 仍作为内容名展示。
- 本轮范围只迁移已确认的可见英文点：行军/进攻 HUD、敌军默认文案、士兵数量、军政世界视图空态/回中按钮/格子计数、地图地点 fallback 标记。

首轮结果：

- 新增 `frontend/js/config/LocaleTextRegistry.js` 与 `frontend/js/domain/LocaleText.js`。
- H5 `frontend/index.html` 和 minigame `frontend/minigame/game.js` 已按加载顺序加入语言 registry/runtime。
- `WorldMarchHudCanvasRenderer`、`WorldMapMilitaryViewRenderer`、`WorldMapStaticEntryRenderer` 已接入 locale runtime。
- `scripts/run-architecture-smoke.js` 已把 locale registry/runtime 和对应测试纳入架构门禁。

验证记录：

- `node --test frontend/js/config/LocaleTextRegistry.test.js frontend/js/domain/LocaleText.test.js frontend/js/domain/WorldActorProjection.test.js`
- `node --test frontend/js/platform/renderers/WorldMarchHudCanvasRenderer.test.js frontend/js/platform/renderers/WorldMapMilitaryViewRenderer.test.js frontend/js/platform/renderers/WorldMapStaticEntryRenderer.test.js`
