## 🔴 P0 - 严重 BUG（必须修）

### BUG-1：`auth.js` 中 `localStorage.removeItem('civilizationFirePhase2')` 位置错误

**位置**：`frontend/auth.js` 第 58 行（`logout` 函数闭合括号之后）

**代码**：
```javascript
game.logout = function() { this.token = null; ... location.reload(); };
localStorage.removeItem('civilizationFirePhase2');  // ← 在 logout 外面！
```

**问题**：每次挂载 auth 模块都会执行这行代码，无条件清除 `civilizationFirePhase2`。这不是 logout 行为，而是每次页面加载都会触发。

**修复**：把这行移进 `logout` 函数内部：
```javascript
game.logout = function() {
    this.token = null;
    this.playerId = null;
    localStorage.removeItem('cf_token');
    localStorage.removeItem('cf_deviceId');
    localStorage.removeItem('civilizationFirePhase2');  // ← 移到这里
    location.reload();
};
```

### BUG-2：`GameStateSync.start()` 没有并发锁，可能导致重复请求

**位置**：`frontend/js/services/GameStateSync.js`

**代码**：
```javascript
start() {
    this.stop();
    this.timer = setInterval(() => this.fetchNow().catch(() => {}), this.intervalMs);
}
```

**问题**：如果 `fetchNow()` 因为网络慢还没有返回，下一次 interval（2秒后）会触发新的 `fetchNow()`，导致两个并发请求。虽然后端可以处理，但会增加服务器负担，且可能导致状态覆盖的竞态条件。

**修复**：加锁：
```javascript
async fetchNow() {
    if (this.fetching) return;
    this.fetching = true;
    try {
        const data = await this.api.getState();
        if (this.onState) this.onState(data);
        return data;
    } catch (error) {
        if (this.onError) this.onError(error);
        throw error;
    } finally {
        this.fetching = false;
    }
}
```

### BUG-3：`startHeartbeat()` 中 auth 错误被吞掉后仍然继续轮询

**位置**：`frontend/app.js`

**代码**：
```javascript
async startHeartbeat() {
    this.gameAPI.setToken(this.token);
    await this.syncService.fetchNow().catch((error) => {
        if (error.payload && error.payload.error && this.handleAuthError) this.handleAuthError(error.payload);
    });
    this.syncService.start();  // ← catch 后继续执行
}
```

**问题**：如果 token 过期/无效，第一次 `fetchNow()` 返回 401，触发 `handleAuthError`（显示登录面板），但 catch 后不会阻止 `syncService.start()`。这意味着登录面板已经弹出来了，但后台每 2 秒仍在发 401 请求，持续刷日志。

**修复**：auth 错误时停止启动心跳：
```javascript
async startHeartbeat() {
    this.gameAPI.setToken(this.token);
    try {
        await this.syncService.fetchNow();
        this.syncService.start();
    } catch (error) {
        if (error.payload?.error && this.handleAuthError) {
            this.handleAuthError(error.payload);
        }
        // 不 start()
    }
}
```

### BUG-4：`BuildingUIRenderer.getDescription()` 中 `effect.level` 永远为 `undefined`

**位置**：`frontend/js/ui/BuildingUIRenderer.js`

**代码**：
```javascript
getDescription(id, buildingEffects) {
    const effect = buildingEffects?.byBuilding?.[id] || {};
    if (effect.level > 0) return '当前效果已按后端最新建筑等级计算';  // ← 永远 false
    return '建造后由后端实时计算效果与升级成本';
}
```

**问题**：`buildingEffects.byBuilding[id]` 的结构是 `{ foodOutputMultiplier, populationCapBonus, happinessBonus }`，没有 `level` 字段。所以 `effect.level` 永远是 `undefined`，永远不会走上面的分支。

**修复**：改用 `FrontendBuildingState.getLevel(state.buildings, id)` 判断：
```javascript
getDescription(id, buildingEffects, buildings) {
    const level = global.FrontendBuildingState.getLevel(buildings, id);
    if (level > 0) return '当前效果已按后端最新建筑等级计算';
    return '建造后由后端实时计算效果与升级成本';
}
```

### BUG-5：注册后前端没有收到 `gameState`，比登录少一步

**位置**：`frontend/auth.js` + `backend/routes/playerRoutes.js`

**问题**：
- `login` API 返回：`{ playerId, token, gameState, tutorial, offlineIncome }`
- `register` API 返回：`{ playerId, deviceId, token }`（没有 gameState）

注册成功后前端直接 `startHeartbeat()`，等 2 秒心跳才看到状态。而登录后前端可以立即用 `gameState` 渲染。两者不一致。

**修复**（二选一）：
- **方案A**：注册 API 也返回 `gameState` 和 `tutorial`
- **方案B**：前端注册成功后也调一次 `/api/game/state` 立即同步

### BUG-6：`TutorialController.setState()` 中 `autoStarted` 逻辑可能导致重复请求

**位置**：`frontend/js/controllers/TutorialController.js`

**代码**：
```javascript
setState(tutorial) {
    this.state = tutorial || { completed: false, currentStep: 0 };
    localStorage.setItem('tutorialStep', String(this.state.currentStep));
    if (!this.state.completed && this.state.currentStep === 0 && !this.autoStarted) {
        this.autoStarted = true;
        setTimeout(() => this.advanceTo(1).catch(() => {}), global.GameConfig.TUTORIAL_START_DELAY_MS);
    }
}
```

**问题**：`autoStarted` 是内存变量（不是 localStorage 持久化的）。如果玩家 step 0 时刷新页面：
1. 页面加载 → `autoStarted = false`（新实例）
2. `setState({ completed: false, currentStep: 0 })` → 触发 autoStart
3. 但 localStorage 里可能已经是 step 1（如果上次 autoStart 成功但页面刷新了）

这会导致 API 重复收到 `advanceTutorial(1)` 请求。虽然后端 `manualAdvance` 有 `if (step <= currentStep) return` 防护，但会多一次无意义的 API 调用。

**修复**：从 localStorage 恢复 `autoStarted`：
```javascript
this.autoStarted = localStorage.getItem('tutorialAutoStarted') === 'true';
// 触发后：
localStorage.setItem('tutorialAutoStarted', 'true');
```

### BUG-7：`auth.js` 的 `loginOrRegister` 错误处理不完整

**位置**：`frontend/auth.js`

**代码**：
```javascript
if (data.token) { ... } else {
    const el = document.getElementById('loginMessage');
    if (el) el.textContent = data.message || '登录失败';
}
```

**问题**：如果服务器返回非 JSON（比如 502 错误返回 HTML），`await resp.json()` 会解析失败，`data` 变成 `{}`，进入 else 分支显示 "登录失败"。但 catch 分支没有处理 HTTP 错误码（比如 500、502）。

**修复**：检查 `resp.ok`：
```javascript
if (!resp.ok) {
    const el = document.getElementById('loginMessage');
    if (el) el.textContent = data.message || `服务器错误 (${resp.status})`;
    return;
}
```

### BUG-8：`GameStateRepository.findAll()` 可能包含过期的玩家数据

**位置**：`backend/repositories/GameStateRepository.js`

**代码**：
```javascript
findAll() {
    const rows = this.db.prepare('SELECT playerId FROM game_states').all();
    return rows.map((row) => this.findByPlayerId(row.playerId)).filter(Boolean);
}
```

**问题**：`SELECT playerId FROM game_states` 选出所有 game_states 的 playerId，但如果某个玩家在 `players` 表里被删除了（比如 reset），他的 game_state 可能还留在数据库里（因为 `resetPlayer` 只 delete players 和 game_states，但如果手动操作数据库可能不一致）。

**当前代码**：`resetPlayer` 确实 `DELETE FROM game_states WHERE playerId = ?` 和 `DELETE FROM players WHERE playerId = ?`，所以数据是一致的。这不是严重问题。

### BUG-9：`ResourceTickCalculator.applyPopulationGrowth` 的 `while` 循环在极端情况下可能耗时

**位置**：`backend/calculators/ResourceTickCalculator.js`

**代码**：
```javascript
while (population.growthProgress >= interval) {
    population.growthProgress -= interval;
    // ...
}
```

**问题**：如果服务器宕机很久后重启，tick 中遍历所有玩家，`growthProgress` 可能累积了几百万秒。while 循环会执行很多次。虽然 `growthProgress` 在 save 时会被重置，但如果数据库里存了一个很大的值...

实际上，`growthProgress` 在 tick 中是累加的，每次 +1 秒。即使服务器停了 1 小时，重启后第一次 tick 所有玩家的 `growthProgress` 都 +1，不会一次暴增。所以这个不是问题。

### BUG-10：`BuildingConfig` 的 `raw` 导出是可变引用

**位置**：`backend/config/BuildingConfig.js`

**代码**：
```javascript
module.exports = { raw: config, ... };
```

**问题**：`raw` 直接暴露 `config` 对象引用。如果任何调用者修改 `raw.buildings.farm.maxLevel = 99`，会永久影响所有后续调用。目前没有代码修改 `raw`，但这是一个潜在的 footgun。

**修复**：深拷贝暴露：
```javascript
module.exports = { raw: () => JSON.parse(JSON.stringify(config)), ... };
```

### BUG-12：`TutorialService` 的 step 编号有断层（3 和 6 不存在）

**位置**：`backend/services/TutorialService.js`

**代码**：
```javascript
const map = {
    tutorialStarted: 1,
    civilizationTabOpened: 2,
    eraAdvanced: 4,        // ← 跳过了 3
    buildingsTabOpened: 5,
    farmBuilt: 7,          // ← 跳过了 6
};
```

**问题**：step 3 和 step 6 不存在。`canAccessTab` 中：
- `step <= 3` → 只有 civilization 标签
- `step === 4` → civilization + buildings
- `step <= 6` → 只有 buildings

如果某个操作意外把 tutorial 推进到 step 3 或 6（比如手动调用 `manualAdvance(3)`），`canAccessTab` 的行为会是：
- step 3：`step <= 3` → 只有 civilization（和 step 2 一样）
- step 6：`step <= 6` → 只有 buildings（和 step 5 一样）

这不是 BUG，只是编号不连续。但如果以后需要在这些步骤做特殊处理，会 confusion。

**建议**：补全编号或加注释说明断层是故意的。

### BUG-13：`deploy.sh` 的健康检查 URL 用 `127.0.0.1`

**位置**：`deploy.sh`

**代码**：
```bash
curl -fsS "http://127.0.0.1:${API_PORT}/api/health"
```

**问题**：如果 PM2 配置了绑定到 `0.0.0.0`，`127.0.0.1` 可以访问。但如果服务器有防火墙或特殊网络配置，`127.0.0.1` 可能不通。更安全的做法是直接用 `localhost` 或检查 `pm2 status`。

这不是严重问题，只是部署健壮性。

### BUG-14：前端 `TutorialController` 的 `getTargetKey()` 和 `getTarget()` 在 step 7 时返回 null

**位置**：`frontend/js/controllers/TutorialController.js`

**代码**：
```javascript
getTargetKey() {
    const step = this.state.currentStep;
    if (step === 1) return 'tab-civilization';
    if (step === 2) return 'btn-advance-era';
    if (step === 4) return 'tab-buildings';
    if (step === 5) return 'card-farm';
    return null;
}
```

**问题**：step 7（完成）时返回 null，`render()` 中 `target = null`，`this.renderer.show(target, ...)` 会传 null。需要确认 `TutorialUIRenderer.show()` 是否能处理 null target。

如果 `show(null, message)` 会报错或显示在错误位置，这是个 BUG。但如果 `renderer.hide()` 在 completed 时已经被调用，不会走到 `show`。看 `render()`：
```javascript
if (this.state.completed || this.state.currentStep === 0) {
    this.renderer.hide();
    return;
}
```

step 7 时 `completed = true`，所以不会走到 `show`。这不是 BUG。
