# 要员制、宜居度与攻略入口设计 v0.1

时间戳：2026-05-23 02:09:18 +08:00

## 设计目标

本阶段把玩家直接操作的“人口”调整为“要员”。要员代表可被分配到农业、研究、工匠等关键岗位的一组核心人员；普通人口作为城市规模展示。

建议规则：

- 1 个要员 = 100 人口。
- 后端现有 `population.total / farmers / scholars / craftsmen / unassigned` 暂时继续作为“要员数量”使用。
- UI 派生展示人口规模：`显示人口 = 要员数量 * 100`。
- 不把后端人口字段整体乘以 100，避免产出、消耗、时代门槛全面失衡。

## 概念拆分

### 要员

要员是玩家分配的核心劳动力单位。现有农民、学者、工匠本质上都应理解为要员岗位。

推荐文案调整：

- `人口管理` 改为 `要员分配`。
- `待分配人口` 改为 `待分配要员`。
- `总人口` 在分配面板中改为 `要员` 或 `要员总数`。
- 左上角时代名下方新增小字：`人口：300`。

### 人口

人口是城市规模展示值，不直接参与玩家点击分配。

人口规模用于让文明发展更有真实感，例如：

- 3 要员显示为人口 300。
- 6 要员显示为人口 600。
- 12 要员显示为人口 1200。

### 宜居度

宜居度表达城市规划质量。它由居住、农业、生产、军事等建筑搭配决定，主要影响人口增长速度。

建议初始值为 0，0 表示中性且正常，不代表差。

### 幸福度

幸福度表达娱乐、精神、仪式、文化等生活满意度。当前阶段建议固定为 100，不受现有民居、神庙、事件 buff 影响。后续等娱乐设施或精神设施设计完成后，再统一接入。

这样可以形成清晰分工：

- 宜居度：城市规划。
- 幸福度：娱乐规划。

## 人口上限拆分

建议将人口上限拆成两个来源：

1. 时代上限：当前时代允许城市达到的最大规模。
2. 民居上限：当前民居规模能容纳的最大规模。

实际上限：

```text
实际要员上限 = min(时代要员上限, 民居要员上限)
显示人口上限 = 实际要员上限 * 100
```

推荐第一版时代上限：

| 时代 | 要员上限 | 显示人口上限 |
| --- | ---: | ---: |
| 原始时代 | 3 | 300 |
| 农耕时代 | 6 | 600 |
| 聚落时代 | 9 | 900 |
| 城邦时代 | 12 | 1200 |
| 边境时代 | 16 | 1600 |
| 古典时代 | 20 | 2000 |

当前民居配置是每级提供 3 个要员上限，基础上限是 3。因此：

```text
民居要员上限 = 3 + 民居等级 * 3
```

这与当前早期流程兼容：

- 初始上限 3。
- 建造 1 级民居后上限 6。
- 升到 3 级民居后上限 12。

## 宜居度内部规则草案

玩家攻略里不展示精确比例，只提示“需要搭配”。开发实现时可以使用隐藏评分。

建议以建筑等级作为“规模单位”，因为当前系统不是多栋建筑实例，而是单个建筑有等级。

内部方向：

- 民居与农田：民居需要足够农田配套，农田不足会降低宜居度，接近理想搭配会提高宜居度。
- 民居与伐木场：伐木场提供生产能力，但会挤压环境，过密会降低宜居度，适度配套更好。
- 民居与兵营：兵营提供安全与士兵训练，但过早或过度军事化会降低宜居度，需要足够民居规模承接。

第一版建议：

```text
宜居度范围：-50 到 +50
人口增长系数 = clamp(1 + 宜居度 / 100, 0.5, 1.5)
```

暂时不让宜居度影响资源产出，只影响人口增长速度。

## 攻略入口设计

建议新增一个轻量的“攻略”入口，位置可以放在右下角任务图标上方，或放进设置/日志一侧的轻量入口。入口不应该打断游戏流程，玩家需要时再打开。

攻略面板结构：

| 分类 | 内容目标 |
| --- | --- |
| 入门 | 解释时代、资源、要员、任务目标。 |
| 要员分配 | 说明农民、学者、工匠分别影响食物、知识、木材。 |
| 城市规划 | 说明宜居度来自建筑搭配，会影响人口增长。 |
| 资源生产 | 说明农田、伐木场、学院、工坊等会改变对应产出。 |
| 时代进阶 | 说明进阶需要资源、建设和安全条件。 |
| 事件与安全 | 说明事件会带来奖励、消耗或威胁，军事设施和士兵影响应对能力。 |
| 探索与领土 | 说明侦察、占领和分城会影响城市发展。 |

玩家侧文案原则：

- 只写方向，不写公式。
- 不写具体比例和隐藏阈值。
- 用“搭配”“过密”“配套”“城市规模”等词引导玩家理解。

示例文案：

```text
农田会影响粮食供应，也会改变城市的生活稳定性。民居附近保持足够农业配套，有利于人口增长。
```

```text
伐木场能带来木材，但生产设施过密会挤压居住环境。随着城市扩大，生产区也需要更合理地安排。
```

```text
兵营能训练士兵并提升安全感，但城市过早过度军事化，也会让生活环境变得紧张。
```

```text
宜居度反映城市规划，主要影响人口增长。幸福度反映娱乐与精神生活，后续会由娱乐设施和文化设施影响。
```

## 当前产出公式复盘

代码来源：

- `backend/calculators/ResourceTickCalculator.js`
- `backend/calculators/BuildingEffectCalculator.js`
- `backend/config/GameConfig.js`
- `shared/buildingConfig.json`

当前基础配置：

| 项目 | 当前值 |
| --- | ---: |
| 每农民食物产出 | 1.0 / 秒 |
| 每人口食物消耗 | 0.2 / 秒 |
| 每人口基础知识 | 0.05 / 秒 |
| 每学者额外知识 | 0.15 / 秒 |
| 每工匠基础木材系数 | 1.0 |
| 伐木场每级基础木材 | 2 |
| 人口自然增长间隔 | 120 秒 |

食物：

```text
食物产出 = 农民数
  * 1.0
  * 农田产出系数
  * 领土食物加成
  * 幸福度系数
  * 全局产出系数
  * 食物事件 buff

食物消耗 = 总人口数 * 0.2

食物净增长 = 食物产出 - 食物消耗
```

当前农田产出系数：

```text
农田产出系数 = 1 + 农田等级 * 0.5
```

知识：

```text
基础知识 = 总人口数 * 0.05
学者知识 = 学者数 * 0.15 * 学院加成 * 领土知识加成
知识总产出 = (基础知识 + 学者知识) * 全局产出系数 * 知识事件 buff
```

木材：

```text
木材产出 = 工匠数
  * 1.0
  * 伐木场基础木材
  * 工匠产出加成
  * 领土木材加成
  * 全局产出系数
  * 木材事件 buff
```

当前伐木场基础木材：

```text
伐木场基础木材 = 伐木场等级 * 2
```

人口增长：

```text
每 120 秒尝试增长 1 个单位。
如果食物 > 0 且未达到人口上限，则总人口 +1，待分配 +1。
```

当前人口增长尚未受幸福度或宜居度影响。

## 要员制影响计算

### 推荐做法：仅 UI 展示乘以 100

如果后端仍以“要员”为计算单位，只在 UI 显示人口时乘以 100：

```text
后端 population.total = 3
UI 显示人口 = 300
```

则当前所有产出、消耗、时代门槛、任务奖励都不需要跟着改。

原因：

- 农民、学者、工匠本来就是玩家分配单位，改名为要员即可。
- 食物消耗仍按要员计算，不会突然增加 100 倍。
- 知识与木材仍按要员岗位计算，不会突然膨胀。
- 人口增长从 `+1 人口单位` 解释为 `+1 要员`，UI 显示为人口规模增加 100。

结论：要员制本身不需要微调产出公式。

### 不推荐做法：后端人口整体乘以 100

如果把后端 `population.total/farmers/scholars/craftsmen` 全部乘以 100，会产生严重连锁问题：

- 食物产出约 100 倍。
- 食物消耗约 100 倍。
- 基础知识约 100 倍。
- 学者知识约 100 倍。
- 工匠木材约 100 倍。
- 当前时代资源门槛会被快速冲穿。
- 人口增长每 120 秒 +1 真实人口会显得过慢，但如果 +100 又会和现有分配逻辑冲突。

结论：不要把后端人口改成真实人口。真实人口只作为展示派生值。

## 是否需要微调

### 只做要员制

不需要调。

只要保持后端单位不变，所有产出仍然按当前节奏运行。UI 文案变得更合理，但数值不会变化。

### 固定幸福度为 100

如果同时移除民居和神庙对幸福度的影响，当前早期食物产出会略降。

以 1 级农田、1 级民居、3 名农民为例：

当前：

```text
食物产出 = 3 * 1.0 * 1.5 * 1.05 = 4.725 / 秒
食物消耗 = 3 * 0.2 = 0.6 / 秒
净增长 = 4.125 / 秒
```

固定幸福度 100 后：

```text
食物产出 = 3 * 1.0 * 1.5 = 4.5 / 秒
食物消耗 = 0.6 / 秒
净增长 = 3.9 / 秒
```

差异：

```text
净增长降低 0.225 / 秒，约 5.5%
```

以 1 级农田、1 级民居、4 要员，其中 3 农民 1 学者为例：

当前：

```text
食物产出 = 3 * 1.0 * 1.5 * 1.05 = 4.725 / 秒
食物消耗 = 4 * 0.2 = 0.8 / 秒
净增长 = 3.925 / 秒
知识 = 4 * 0.05 + 1 * 0.15 = 0.35 / 秒
```

固定幸福度 100 后：

```text
食物产出 = 3 * 1.0 * 1.5 = 4.5 / 秒
食物消耗 = 0.8 / 秒
净增长 = 3.7 / 秒
知识 = 0.35 / 秒
```

差异：

```text
食物净增长降低 0.225 / 秒，约 5.7%
知识不受影响。
```

判断：

- 早期影响很小，不需要立刻调资源门槛。
- 如果后续移除所有幸福度产出加成，主要影响是食物略慢。
- 可以先观察测试流程。如果玩家到聚落或城邦慢了，再把主线任务奖励或早期农田系数小幅补回来。

### 引入宜居度影响人口增长

宜居度只影响人口增长，不影响资源产出。因此它不会直接改变食物、木材、知识产量。

默认宜居度为 0 时：

```text
增长间隔仍然是 120 秒。
```

如果宜居度为 +20：

```text
增长速度约 +20%，等效增长间隔约 100 秒。
```

如果宜居度为 -20：

```text
增长速度约 -20%，等效增长间隔约 150 秒。
```

判断：

- 初期默认 0 不影响现有流程。
- 宜居度作为规划奖励，适合调人口节奏，不适合直接调资源产出。
- 第一版不建议让宜居度过早大幅为负，避免新手卡住要员增长。

## 结论

1. 要员制只改 UI 解释和显示，不需要改产出公式。
2. 后端人口不要乘以 100，否则产出与门槛会全面失衡。
3. 幸福度固定为 100 后，早期食物净增长大约降低 5% 到 6%，暂时可以不调。
4. 宜居度建议先只影响人口增长，默认 0 不改变现有节奏。
5. 攻略入口可以加入，但攻略内容只提示影响方向，不暴露隐藏比例。
6. 人口上限建议拆为时代上限和民居上限，实际取较小值。

## 分步实施计划

记录时间：2026-05-23 02:21:10 +08:00

### 第 1 步：要员制 UI 展示

版本目标：`0.1.71`

实施范围：

- 将玩家分配面板文案从“人口管理”调整为“要员分配”。
- 将分配面板中的“总人口 / 待分配”调整为“要员 / 待分配要员”。
- 左上角时代名下方增加小字人口规模，例如初始 3 要员显示为 `人口：300`。
- 城市切换器与文明概览中的人口显示改为展示人口规模，即 `要员 * 100`。
- 后端 `population.total / farmers / scholars / craftsmen / unassigned` 不做乘以 100 的数据迁移。
- 产出、消耗、任务、时代进阶、人口增长公式均不在本步调整。

测试方法：

- 登录游戏后观察左上角时代名下方是否出现 `人口：300`。
- 进入资源页，检查人口面板标题是否为“要员分配”。
- 检查面板内是否显示“要员 3/3”与“待分配要员 0”一类文案。
- 切换要员岗位，确认加减按钮仍能正常分配农民、学者、工匠。
- 打开文明页，检查概览人口是否显示为 300、600 这类人口规模，而不是 3、6。

预期结果：

- 视觉和文案变成要员制，但游戏节奏不变。
- 食物、木材、知识产出不发生百倍变化。
- 引导、任务、时代进阶条件不因本步改动而卡住。

### 第 2 步：幸福度冻结为 100

版本目标：待第 1 步通过测试后推进。

实施范围：

- 移除当前建筑或事件对幸福度生产系数的直接影响。
- 幸福度 UI 暂时固定显示 100%。
- 保留字段结构，供后续娱乐设施、文化设施接入。

测试重点：

- 早期食物净增长只允许出现约 5% 到 6% 的下降。
- 主线任务奖励和时代进阶节奏不能因此变慢到卡流程。

### 第 3 步：宜居度展示与默认值

版本目标：待第 2 步通过测试后推进。

实施范围：

- 新增宜居度字段，默认 0。
- UI 增加宜居度展示，但第一版不让它影响资源产出。
- 攻略文案提示宜居度来自城市规划，不展示具体公式。

测试重点：

- 默认存档和新建存档都显示宜居度 0。
- 未建造相关设施时，不改变当前资源生产。

### 第 4 步：人口上限拆分

版本目标：待第 3 步通过测试后推进。

实施范围：

- 将人口上限拆为时代要员上限与民居要员上限。
- 实际要员上限取两者较小值。
- UI 同时解释“人口规模”和“要员上限”。

测试重点：

- 初始时代不会突破 3 要员。
- 建造民居后仍能顺畅承接新手流程。
- 时代进阶后人口成长不会被旧的单一上限卡住。

### 第 5 步：建筑规模与维护成本

版本目标：待第 4 步通过测试后推进。

实施范围：

- 取消硬性的建筑等级上限，改为“规模”概念。
- 为建筑规模增加维护成本。
- 调整建筑产出曲线，避免线性堆等级导致数值爆炸。
- 攻略中只提示“规模越大，维护压力越高”，不展示精确公式。

测试重点：

- 建筑升级不再被旧等级上限挡住。
- 维护成本不会让新手阶段资源直接变负。
- 农田、民居、伐木场、兵营的搭配能影响后续宜居度设计。
