# 要员制、宜居度与攻略入口设计 v0.1

时间戳：2026-05-23 02:09:18 +08:00

## 设计目标

本阶段把玩家直接操作的“人口”调整为“要员”。要员代表可被分配到农业、研究、工匠等关键岗位的一组核心人员；普通人口作为城市规模展示。

建议规则：

- 1 个要员 = 100 人口。
- 后端现有 `population.total / farmers / scholars / craftsmen / unassigned` 暂时继续作为“要员数量”使用。
- UI 派生展示人口规模：`显示人口 = 要员数量 * 100`。
- 不把后端人口字段整体乘以 100，避免产出、消耗、时代门槛全面失衡。

## 概念拆分

### 要员

要员是玩家分配的核心劳动力单位。现有农民、学者、工匠本质上都应理解为要员岗位。

推荐文案调整：

- `人口管理` 改为 `要员分配`。
- `待分配人口` 改为 `待分配要员`。
- `总人口` 在分配面板中改为 `要员` 或 `要员总数`。
- 左上角时代名下方新增小字：`人口：300`。

### 人口

人口是城市规模展示值，不直接参与玩家点击分配。

人口规模用于让文明发展更有真实感，例如：

- 3 要员显示为人口 300。
- 6 要员显示为人口 600。
- 12 要员显示为人口 1200。

### 宜居度

宜居度表达城市规划质量。它由居住、农业、生产、军事等建筑搭配决定，主要影响人口增长速度。

建议初始值为 0，0 表示中性且正常，不代表差。

### 幸福度

幸福度表达娱乐、精神、仪式、文化等生活满意度。当前阶段建议固定为 100，不受现有民居、神庙、事件 buff 影响。后续等娱乐设施或精神设施设计完成后，再统一接入。

这样可以形成清晰分工：

- 宜居度：城市规划。
- 幸福度：娱乐规划。

## 人口上限拆分

建议将人口上限拆成两个来源：

1. 时代上限：当前时代允许城市达到的最大规模。
2. 民居上限：当前民居规模能容纳的最大规模。

实际上限：

```text
实际要员上限 = min(时代要员上限, 民居要员上限)
显示人口上限 = 实际要员上限 * 100
```

推荐第一版时代上限：

| 时代 | 要员上限 | 显示人口上限 |
| --- | ---: | ---: |
| 原始时代 | 3 | 300 |
| 农耕时代 | 6 | 600 |
| 聚落时代 | 9 | 900 |
| 城邦时代 | 12 | 1200 |
| 边境时代 | 16 | 1600 |
| 古典时代 | 20 | 2000 |

当前民居配置是每级提供 3 个要员上限，基础上限是 3。因此：

```text
民居要员上限 = 3 + 民居等级 * 3
```

这与当前早期流程兼容：

- 初始上限 3。
- 建造 1 级民居后上限 6。
- 升到 3 级民居后上限 12。

## 宜居度内部规则草案

玩家攻略里不展示精确比例，只提示“需要搭配”。开发实现时可以使用隐藏评分。

建议以建筑等级作为“规模单位”，因为当前系统不是多栋建筑实例，而是单个建筑有等级。

内部方向：

- 民居与农田：民居需要足够农田配套，农田不足会降低宜居度，接近理想搭配会提高宜居度。
- 民居与伐木场：伐木场提供生产能力，但会挤压环境，过密会降低宜居度，适度配套更好。
- 民居与兵营：兵营提供安全与士兵训练，但过早或过度军事化会降低宜居度，需要足够民居规模承接。

第一版建议：

```text
宜居度范围：-50 到 +50
人口增长系数 = clamp(1 + 宜居度 / 100, 0.5, 1.5)
```

暂时不让宜居度影响资源产出，只影响人口增长速度。

## 攻略入口设计

建议新增一个轻量的“攻略”入口，位置可以放在右下角任务图标上方，或放进设置/日志一侧的轻量入口。入口不应该打断游戏流程，玩家需要时再打开。

攻略面板结构：

| 分类 | 内容目标 |
| --- | --- |
| 入门 | 解释时代、资源、要员、任务目标。 |
| 要员分配 | 说明农民、学者、工匠分别影响食物、知识、木材。 |
| 城市规划 | 说明宜居度来自建筑搭配，会影响人口增长。 |
| 资源生产 | 说明农田、伐木场、学院、工坊等会改变对应产出。 |
| 时代进阶 | 说明进阶需要资源、建设和安全条件。 |
| 事件与安全 | 说明事件会带来奖励、消耗或威胁，军事设施和士兵影响应对能力。 |
| 探索与领土 | 说明侦察、占领和分城会影响城市发展。 |

玩家侧文案原则：

- 只写方向，不写公式。
- 不写具体比例和隐藏阈值。
- 用“搭配”“过密”“配套”“城市规模”等词引导玩家理解。

示例文案：

```text
农田会影响粮食供应，也会改变城市的生活稳定性。民居附近保持足够农业配套，有利于人口增长。
```

```text
伐木场能带来木材，但生产设施过密会挤压居住环境。随着城市扩大，生产区也需要更合理地安排。
```

```text
兵营能训练士兵并提升安全感，但城市过早过度军事化，也会让生活环境变得紧张。
```

```text
宜居度反映城市规划，主要影响人口增长。幸福度反映娱乐与精神生活，后续会由娱乐设施和文化设施影响。
```

## 当前产出公式复盘

代码来源：

- `backend/calculators/ResourceTickCalculator.js`
- `backend/calculators/BuildingEffectCalculator.js`
- `backend/config/GameConfig.js`
- `shared/buildingConfig.json`

当前基础配置：

| 项目 | 当前值 |
| --- | ---: |
| 每农民食物产出 | 1.0 / 秒 |
| 每人口食物消耗 | 0.2 / 秒 |
| 每人口基础知识 | 0.05 / 秒 |
| 每学者额外知识 | 0.15 / 秒 |
| 每工匠基础木材系数 | 1.0 |
| 伐木场每级基础木材 | 2 |
| 人口自然增长间隔 | 120 秒 |

食物：

```text
食物产出 = 农民数
  * 1.0
  * 农田产出系数
  * 领土食物加成
  * 幸福度系数
  * 全局产出系数
  * 食物事件 buff

食物消耗 = 总人口数 * 0.2

食物净增长 = 食物产出 - 食物消耗
```

当前农田产出系数：

```text
农田产出系数 = 1 + 农田等级 * 0.5
```

知识：

```text
基础知识 = 总人口数 * 0.05
学者知识 = 学者数 * 0.15 * 学院加成 * 领土知识加成
知识总产出 = (基础知识 + 学者知识) * 全局产出系数 * 知识事件 buff
```

木材：

```text
木材产出 = 工匠数
  * 1.0
  * 伐木场基础木材
  * 工匠产出加成
  * 领土木材加成
  * 全局产出系数
  * 木材事件 buff
```

当前伐木场基础木材：

```text
伐木场基础木材 = 伐木场等级 * 2
```

人口增长：

```text
每 120 秒尝试增长 1 个单位。
如果食物 > 0 且未达到人口上限，则总人口 +1，待分配 +1。
```

当前人口增长尚未受幸福度或宜居度影响。

## 要员制影响计算

### 推荐做法：仅 UI 展示乘以 100

如果后端仍以“要员”为计算单位，只在 UI 显示人口时乘以 100：

```text
后端 population.total = 3
UI 显示人口 = 300
```

则当前所有产出、消耗、时代门槛、任务奖励都不需要跟着改。

原因：

- 农民、学者、工匠本来就是玩家分配单位，改名为要员即可。
- 食物消耗仍按要员计算，不会突然增加 100 倍。
- 知识与木材仍按要员岗位计算，不会突然膨胀。
- 人口增长从 `+1 人口单位` 解释为 `+1 要员`，UI 显示为人口规模增加 100。

结论：要员制本身不需要微调产出公式。

### 不推荐做法：后端人口整体乘以 100

如果把后端 `population.total/farmers/scholars/craftsmen` 全部乘以 100，会产生严重连锁问题：

- 食物产出约 100 倍。
- 食物消耗约 100 倍。
- 基础知识约 100 倍。
- 学者知识约 100 倍。
- 工匠木材约 100 倍。
- 当前时代资源门槛会被快速冲穿。
- 人口增长每 120 秒 +1 真实人口会显得过慢，但如果 +100 又会和现有分配逻辑冲突。

结论：不要把后端人口改成真实人口。真实人口只作为展示派生值。

## 是否需要微调

### 只做要员制

不需要调。

只要保持后端单位不变，所有产出仍然按当前节奏运行。UI 文案变得更合理，但数值不会变化。

### 固定幸福度为 100

如果同时移除民居和神庙对幸福度的影响，当前早期食物产出会略降。

以 1 级农田、1 级民居、3 名农民为例：

当前：

```text
食物产出 = 3 * 1.0 * 1.5 * 1.05 = 4.725 / 秒
食物消耗 = 3 * 0.2 = 0.6 / 秒
净增长 = 4.125 / 秒
```

固定幸福度 100 后：

```text
食物产出 = 3 * 1.0 * 1.5 = 4.5 / 秒
食物消耗 = 0.6 / 秒
净增长 = 3.9 / 秒
```

差异：

```text
净增长降低 0.225 / 秒，约 5.5%
```

以 1 级农田、1 级民居、4 要员，其中 3 农民 1 学者为例：

当前：

```text
食物产出 = 3 * 1.0 * 1.5 * 1.05 = 4.725 / 秒
食物消耗 = 4 * 0.2 = 0.8 / 秒
净增长 = 3.925 / 秒
知识 = 4 * 0.05 + 1 * 0.15 = 0.35 / 秒
```

固定幸福度 100 后：

```text
食物产出 = 3 * 1.0 * 1.5 = 4.5 / 秒
食物消耗 = 0.8 / 秒
净增长 = 3.7 / 秒
知识 = 0.35 / 秒
```

差异：

```text
食物净增长降低 0.225 / 秒，约 5.7%
知识不受影响。
```

判断：

- 早期影响很小，不需要立刻调资源门槛。
- 如果后续移除所有幸福度产出加成，主要影响是食物略慢。
- 可以先观察测试流程。如果玩家到聚落或城邦慢了，再把主线任务奖励或早期农田系数小幅补回来。

### 引入宜居度影响人口增长

宜居度只影响人口增长，不影响资源产出。因此它不会直接改变食物、木材、知识产量。

默认宜居度为 0 时：

```text
增长间隔仍然是 120 秒。
```

如果宜居度为 +20：

```text
增长速度约 +20%，等效增长间隔约 100 秒。
```

如果宜居度为 -20：

```text
增长速度约 -20%，等效增长间隔约 150 秒。
```

判断：

- 初期默认 0 不影响现有流程。
- 宜居度作为规划奖励，适合调人口节奏，不适合直接调资源产出。
- 第一版不建议让宜居度过早大幅为负，避免新手卡住要员增长。

## 结论

1. 要员制只改 UI 解释和显示，不需要改产出公式。
2. 后端人口不要乘以 100，否则产出与门槛会全面失衡。
3. 幸福度固定为 100 后，早期食物净增长大约降低 5% 到 6%，暂时可以不调。
4. 宜居度建议先只影响人口增长，默认 0 不改变现有节奏。
5. 攻略入口可以加入，但攻略内容只提示影响方向，不暴露隐藏比例。
6. 人口上限建议拆为时代上限和民居上限，实际取较小值。

## 术语更新：人才分配

玩家侧文案从“要员”进一步调整为“人才”。“人才”比“要员”更适合长期扩展，因为后续职业不一定都是行政或管理岗位，也可能包括士兵、祭司、商人、工程师、医者等更具体的人群。

建议规则：

- 仍保持 `1 人才 = 100 人口` 的展示换算。
- 后端现有 `population.total / farmers / scholars / craftsmen / unassigned` 仍暂时作为“可分配人才单位”使用，不做数据迁移。
- 代码内部可以继续沿用既有 population 字段，避免本阶段引入大规模重命名风险。

## 宜居度、地理要素与城市方针设计

更新时间：2026-05-23 02:45:34 +08:00

### 设计目标

宜居度不写死单一配比，而是用“城市规划方向是否合理”来评价。不同地理环境下，合理规划的方向不同，玩家不应该背固定公式，而应该根据城市环境、发展目标和当前时代做判断。

### 城市地理要素

每座城市建议拥有一个主地理要素，后续可以扩展副要素或复合地貌。

第一版主地理要素建议：

| 地理要素 | 规划倾向 | 宜居度方向 |
| --- | --- | --- |
| 平原 | 适合农业、人口扩张和均衡建设 | 农田与民居配套更容易形成正向宜居 |
| 山地 | 适合矿产、石料、防御和较慢扩张 | 工业与防御设施更容易合理，但过度居住扩张压力更高 |
| 沿海 | 适合贸易、粮食补充、探索和人口流动 | 均衡发展与探索设施更容易带来正向宜居 |
| 森林 | 适合木材、狩猎和早期生产 | 伐木与居住需要保持距离或规模平衡 |
| 河谷 | 适合农业、人口增长和早期文明 | 农田、民居、知识设施组合更容易稳定增长 |

玩家攻略只展示大方向，例如：

```text
平原城市更适合承载农业和人口扩张，保持粮食供应与居住规模的配套，有利于城市稳定发展。
```

```text
山地城市更适合发展矿产、防御和工程建设，但居住扩张会更考验规划，需要避免生产设施过度挤压生活空间。
```

### 宜居度评估原则

宜居度不采用固定公开比例。内部实现可以使用“方向评分”：

- 城市地理要素提供规划倾向。
- 建筑结构提供实际城市形态。
- 人才方针提供当前发展意图。
- 当前时代决定哪些倾向更重要。

第一版不要把某个固定比例写死为“正确答案”。例如平原不是永远必须 `1 民居 : 2 农田`，而是“农业与居住配套越充分，越容易稳定”。山地不是必须少建民居，而是“生产、防御、居住之间需要更谨慎的平衡”。

### 人才方针入口

在 `人才分配` 标题右侧增加一个轻量按钮，建议命名为 `方针`。

入口目标：

- 让玩家一键切换城市发展方向。
- 根据方针自动调整人才岗位分配。
- 支持系统预设和玩家自定义。
- 后续新增职业时，不需要重做旧方针结构。

### 系统预设方针

第一版建议提供：

| 方针 | 目标 | 适用场景 |
| --- | --- | --- |
| 均衡发展 | 尽量保持粮食、知识、木材都有人才投入 | 默认状态，新玩家不容易卡资源 |
| 农业优先 | 优先保障粮食生产和人口承载 | 粮食不足、准备扩张人口 |
| 知识优先 | 优先投入学者，加快科技和时代进阶 | 资源稳定但知识不足 |
| 工业优先 | 优先投入工匠，提高木材和后续工业资源 | 建筑升级、军事建设前 |
| 扩张准备 | 在粮食与工业之间保持偏重，为建筑和人口增长做准备 | 进入新阶段前 |
| 军备准备 | 后续士兵或军事职业接入后使用 | 侦察、占领、防御威胁前 |

### 自定义方针微调

更新时间：2026-05-23 02:56:28 +08:00

自定义方针不要做成自由输入表单，也不要做精准百分比或数值拉条。它应该是“在系统默认方针基础上的轻量微调”。

推荐交互：

1. 玩家先选择一个系统方针作为底稿，例如 `均衡发展`、`农业优先`、`知识优先`、`工业优先`。
2. 面板显示几个发展倾向，每个倾向只有 3 档。
3. 玩家点击档位按钮微调，例如 `农业 3 档`、`工业 1 档`、`知识 2 档`。
4. 系统自动生成方针名称，例如 `农业优先·工业辅助`、`均衡发展·偏知识`。
5. 玩家保存后进入自定义方针列表。

第一版不提供自由命名输入。这样能减少 H5 和小游戏输入框差异，也能避免玩家被“配数值”的压力打断。

推荐倾向档位：

| 倾向 | 1 档 | 2 档 | 3 档 |
| --- | --- | --- | --- |
| 农业 | 少量关注 | 正常投入 | 重点保障 |
| 知识 | 少量关注 | 正常投入 | 重点研究 |
| 工业 | 少量关注 | 正常投入 | 重点建设 |
| 军备 | 少量准备 | 正常训练 | 重点备战 |
| 探索 | 少量探索 | 正常探索 | 重点开拓 |

第一版只展示当前已解锁或即将有意义的倾向：

- 原始、农耕、聚落前期：农业、知识、工业。
- 解锁兵营后：增加军备。
- 解锁侦察和领土后：增加探索。

数据结构建议：

```json
{
  "id": "custom_balanced_agriculture_3_industry_1",
  "type": "custom",
  "basePolicyId": "balanced",
  "displayName": "均衡发展·偏农业",
  "tiers": {
    "agriculture": 3,
    "knowledge": 2,
    "industry": 1
  }
}
```

这里的 `tiers` 不是公开精准公式，只是玩家表达发展倾向的低精度档位。内部可以把档位转成职业权重，但 UI 不展示权重。

内部换算原则：

```text
最终职业权重 = 系统方针基础权重 + 档位倾向修正
```

例如：

- `农业 3 档` 会提高农民目标占比。
- `知识 3 档` 会提高学者目标占比。
- `工业 3 档` 会提高工匠目标占比。
- 如果某倾向对应职业尚未解锁，该倾向暂不参与计算，但可以保留在配置中。

这种方案的好处：

- 玩家不需要输入文字。
- 玩家不需要凑百分比。
- 玩家不需要拖精准数值条。
- 系统默认方针仍是主框架，微调不会过度跑偏。
- 后续新增职业时，只需要把职业挂到倾向标签上，旧自定义方针仍然能工作。

### 支持后续职业增加

方针配置不应写死农民、学者、工匠三个位置，而应以职业 id 为 key。

建议职业定义提供：

```json
{
  "id": "farmer",
  "name": "农民",
  "resourceFocus": ["food"],
  "unlockEra": 0,
  "policyTags": ["agriculture", "growth"]
}
```

方针应用时：

- 只处理当前已解锁职业。
- 方针中不存在的新职业可由系统方针基础权重补齐。
- 方针中存在但当前未解锁的职业不参与分配。
- 保留最少 1 人才的职业保护可以作为后续选项，但第一版不强制。

### 自动分配行为

点击某个方针后，系统计算目标分配，并一次性提交分配结果。

建议流程：

1. 玩家打开 `方针` 面板。
2. 选择系统预设，或选择一个由系统预设微调保存的自定义方针。
3. 面板显示预计分配，例如 `农民 3、学者 1、工匠 2`。
4. 玩家点击 `应用`。
5. 前端提交一次 `applyTalentPolicy` 动作。
6. 后端校验当前城市、已解锁职业、总人才数量后写入新分配。

第一版可以不做持续自动托管，只做“一键应用”。持续托管会和玩家手动调整互相打架，建议等基础体验稳定后再决定是否加入。

### 方针与地理要素关系

地理要素不直接强制改变方针比例，而是影响推荐与宜居度方向：

- 平原城市默认推荐 `农业优先` 或 `均衡发展`。
- 山地城市默认推荐 `工业优先` 或 `军备准备`。
- 沿海城市默认推荐 `均衡发展` 或后续贸易/探索方针。
- 森林城市默认推荐 `工业优先`，但攻略提示注意居住与生产平衡。
- 河谷城市默认推荐 `农业优先` 或 `知识优先`。

这样玩家可以反着玩，但会承担城市规划压力；系统只给方向，不把玩家锁死。
