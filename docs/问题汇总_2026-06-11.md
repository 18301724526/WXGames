## 1. 总体结论

项目已经具备许多成熟游戏工程方向的候选模块和文档，例如 Canvas 层契约、后端权威命令、runtime/presenter/renderer 分层、稳定积木门禁、部署版本识别等。但当前实现、测试和文档事实还没有完全对齐。

最严重的问题不是单个地图单位 bug，而是多处职责边界仍处在过渡态：

- 世界地图动态对象仍被多个渲染层同时拥有，导致行军单位重复绘制、拖动时消失或错位。
- 后端读接口会推进并保存游戏状态，后台 tick 也会保存同一份状态，容易产生状态竞态、504 和教程异常。
- 架构测试已经能暴露部分问题，但门禁和文档仍存在缺口，容易出现“文档说成熟、代码还未成熟”的误判。
- 线上加载、轮询、超时和错误提示仍偏开发期实现，不足以支撑稳定线上体验。

## 2. P0 问题

### P0-001 世界地图 actor 分层没有真正落地

成熟方案:

- `worldMap`: 地形、格子、建筑底图、静态缓存。
- `worldFog`: 战争迷雾、遮罩，视觉层，不拥有玩法输入。
- `worldActor`: 单位、行军动画、行军箭头、选中圈，全部是世界坐标对象。
- `mainHud`: 按钮、面板、提示、屏幕坐标 UI，不画单位本体。

当前证据:

- 已修复：`frontend/js/platform/renderers/WorldMapTileMapRenderer.js` 的 `renderWorldTileMapHitTargets()` 不再调用 `addWorldActorHitTargets()`。
- 已修复：`frontend/js/platform/renderers/WorldMapTileMapRenderer.js` 的 `renderWorldTileMapLayers()` 不再调用 `renderWorldActors()` 或 `addWorldActorHitTargets()`。
- 已修复：`frontend/js/platform/renderers/WorldMapLayerCanvasRenderer.js` 的 `renderWorldMapActorLayer()` 调用 `renderWorldActors()` 后只注册 actor hit targets，不再调用 `renderWorldMarchHud()`。
- 已修复：`frontend/js/platform/renderers/CanvasFrameRenderer.js` 与 `frontend/js/platform/renderers/HudOverlayCanvasRenderer.js` 在 `mainHud` pass 中调用 `renderMapHomeWorldMarchHud()`。
- `frontend/js/platform/CanvasGameShellWorldMapDragRuntime.js` 已把 `worldMap`、`worldFog`、`worldActor` 三层做相同拖动 transform，这说明项目方向已经承认 `worldActor` 是独立世界层，但 map 层仍在重复绘制 actor。

会发生什么:

- 行军时出现两个小人：一套由 map 层画，一套由 actor 层画。
- 拖动地图时，一套跟着世界 transform 走，另一套可能停在屏幕坐标或被缓存/清屏影响。
- actor hit target 可能重复、优先级错乱或与视觉位置不一致。
- 只测单个渲染器方法容易误判通过，真实运行时会在多层组合后暴露问题。

建议处理:

- map 层只发布 `lastWorldTileMapContext` 和静态 hit targets，不画 actor，不注册 actor target。
- actor 层成为唯一世界坐标 actor 绘制与 actor hit target owner。
- march command HUD 迁回 `mainHud` pass，由 HUD 层消费 actor context。
- 把上述规则写成架构契约测试，并纳入 `npm run test:architecture`。

### P0-002 后端读接口推进并保存状态

成熟方案:

- Query/projection 路径只读，不推进世界时间线，不保存状态。
- Runtime advance 只能在明确的 command、tick 或同步事务中发生。
- 状态推进入口必须有幂等、锁/事务、版本或乐观并发控制。

当前证据:

- 已修复：`backend/routes/gameRoutes.js` 的 `GET /api/game/state` 改为 `loadReadOnlyGameState()`，不再调用 `applyOnlineProgress()`。
- 已修复：同一路径只做 tutorial normalize 和 DTO projection，不再同步教程推进、生成事件、`touchPlayerActiveAt()` 或 `repository.save(gameState)`。
- 已修复：`GET /api/game/tasks` 同样改为只读 projection，不再生成事件、touch 或保存。
- `backend/server.js` 的后台 tick 每 5 秒查找最近活跃玩家，调用 `advanceRuntimeState()`、城市推进、领土/事件处理并保存。

会发生什么:

- 玩家只是刷新、进入游戏或拉状态，也可能改变教程、事件、探索、城市产出和地图状态。
- 多个请求、后台 tick 和 reset/action 交错时，最后写入者可能覆盖前一个写入者。
- `/api/game/state` 变重后容易拖慢首屏，导致加载 100% 后等待，最终报 504。
- 教程“明明已完成却又触发”的问题可能由读路径推进、状态归一化、reset 失败或旧状态覆盖共同造成。

建议处理:

- 明确拆分 `queryGameStateProjection()` 和 `advanceAndCommitRuntimeState()`。
- `/api/game/state` 默认只读；如确需同步推进，应改成命令式 endpoint 或 server tick，并带事务和版本号。
- 背景 tick、action、reset、state sync 统一经过 repository transaction 或 unit-of-work。

### P0-003 持久化缺少事务和版本边界

成熟方案:

- 对同一玩家状态的修改必须有事务边界。
- 主状态表与共享世界表要同事务提交。
- 大 JSON 存档要有 schema version、migration history 和乐观并发版本。

当前证据:

- 已修复：`backend/repositories/GameStateRepository.js` 不再使用 `INSERT OR REPLACE`，改为 `INSERT ... ON CONFLICT DO UPDATE` 并维护 `revision`。
- 已修复：`save()` 委托 `saveAtomic()`，主状态与 `saveSharedWorldTerritories(gameState)` 在同一 SQLite transaction 中提交。
- 已修复：`resetPlayerState(playerId, gameState)` 同事务删除主状态和 `shared_world_territories` 中该玩家派生占有记录，再写入新状态。
- `findByPlayerId()` 会读取玩家私有 territories 后再 merge shared world territories。

会发生什么:

- 保存过程中某一步失败会留下半提交状态。
- reset 与后台 tick/action 交错时，共享世界表可能残留旧占领信息。
- 多请求同时读同一旧状态并保存时，会出现丢更新。

建议处理:

- repository 提供 `saveAtomic(gameState)`，主状态和共享世界表同事务。
- 增加 `revision` 或 `updatedAt` 乐观并发检查。
- reset 要同时清理该玩家在共享世界中的派生占有记录。

## 3. P1 问题

### P1-001 启动加载链路把资源加载和首包状态混在一起

成熟方案:

- 启动阶段明确拆分 boot、auth、manifest、asset preload、first state sync、scene ready。
- 每个阶段都输出耗时、进度、失败原因。
- 首包 API 必须有 timeout、retry/backoff 和用户可理解的错误状态。

当前证据:

- `frontend/auth.js` 中 `startAuthenticatedSession()` 先 `waitForAuthenticatedAssets()`，最后才 `game.startHeartbeat()`。
- `frontend/js/platform/CanvasGameAppStateSync.js` 中 `startHeartbeat()` 先 `syncOnce()` 拉 `/game/state`，成功后才启动 heartbeat。
- 已修复当前范围：`frontend/js/api/GameAPI.js` 使用 `AbortController` timeout，默认 10s；每个请求带 `X-Client-Request-ID`，错误包含 method、path、requestId、attempts、duration、status、code 和 retryable 信息。
- 已修复当前范围：`GameAPI` 只对 GET/HEAD 的 408/425/429/5xx、timeout 和网络错误做短重试；POST action 不自动重试，避免重复提交命令。

会发生什么:

- 资源加载到 100% 后，用户仍会等待首包状态；如果后端慢，就表现为“100% 卡住”。
- Gateway Timeout 仍会作为失败暴露，但前端现在能拿到 requestId、耗时、尝试次数和结构化 code，便于定位首包状态/API。
- 排查时容易误以为是资源问题，实际瓶颈在首包状态/API。

建议处理:

- 已完成：H5LoadTrace 保留 `assets:preload`、`state:first-sync`、`state:syncOnce` 和 API span。
- 已完成：GameAPI 增加 timeout、request id、耗时、重试策略和结构化错误。
- 已完成当前范围：H5LoadTrace 可通过 `setReporter()` 将 phase/API/资源加载失败上报给 `GameAPI.reportClientEvent()`，后端 `POST /api/client-events` 归一化后写入观测窗口。
- 后续预期：首屏 loading 文案进一步显示当前阶段，并把结构化错误映射为用户可理解的提示。

### P1-002 轮询频率高且缺少失败退避

成熟方案:

- heartbeat、state refresh、version check 分频管理。
- 网络失败时指数退避，避免把后端故障放大。
- `/version` 可使用部署 manifest 或 ETag，不必 5 秒无条件打一次。

当前证据:

- `frontend/js/config/GameConfig.js` 中 `HEARTBEAT_INTERVAL_MS = 1000`，`UPDATE_CHECK_INTERVAL_MS = 5000`。
- 已修复当前范围：`frontend/js/services/UpdateChecker.js` 失败后设置指数退避窗口，窗口到期前不再调用 `getVersion()`。
- 已修复当前范围：`frontend/js/services/GameStateSync.js` heartbeat 失败后设置指数退避窗口，窗口到期前不再调用 heartbeat；成功后清空失败计数和退避窗口。
- 已修复当前范围：`/api/version` 现在返回 ETag，命中 `If-None-Match` 时返回 304；`GameAPI.getVersion()` 移除 cache-busting query，带上上次 ETag 并在 304 时复用本地版本快照。
- 已修复当前范围：`VersionService` 可读取部署 manifest，响应中包含 `deployedCommit`、`deployedAt`、`branch` 和稳定 `etag`。
- `GameStateSync` heartbeat 后仍可能在世界探索节点临近时触发 `getState()`，但该触发已有最小间隔限制，普通 query 路径也已改为只读 projection。

会发生什么:

- 后端慢或 504 时，heartbeat 和 version check 不再固定频率连续打；结构化错误和退避窗口会降低故障放大。
- 玩家控制台仍会记录失败，但可以按 requestId、path、attempts 和 next backoff window 定位根因。
- 小规模开发服也可能被轮询拖慢。

建议处理:

- 已完成：网络失败后 version check 和 heartbeat 使用退避。
- 已完成：首包状态失败时 heartbeat 不会启动；已启动的 heartbeat/version check 失败后会退避。
- 已完成当前范围：给 `/version` 增加 ETag/304、轻量缓存和 deploy manifest 元数据。
