# 问题汇总 - 2026-06-11

日期 / Date: 2026-06-11

状态 / Status: audit summary

用途 / Purpose:

记录 2026-06-11 针对整个项目的架构、运行时、后端状态、测试门禁、部署和文档一致性审计结论。本文不是修复记录，不代表下列问题已经解决；后续修复应以本文作为问题索引，并同步更新权威架构文档和责任索引。

## 1. 总体结论

项目已经具备许多成熟游戏工程方向的候选模块和文档，例如 Canvas 层契约、后端权威命令、runtime/presenter/renderer 分层、稳定积木门禁、部署版本识别等。但当前实现、测试和文档事实还没有完全对齐。

最严重的问题不是单个地图单位 bug，而是多处职责边界仍处在过渡态：

- 世界地图动态对象仍被多个渲染层同时拥有，导致行军单位重复绘制、拖动时消失或错位。
- 后端读接口会推进并保存游戏状态，后台 tick 也会保存同一份状态，容易产生状态竞态、504 和教程异常。
- 架构测试已经能暴露部分问题，但门禁和文档仍存在缺口，容易出现“文档说成熟、代码还未成熟”的误判。
- 线上加载、轮询、超时和错误提示仍偏开发期实现，不足以支撑稳定线上体验。

## 2. P0 问题

### P0-001 世界地图 actor 分层没有真正落地

成熟方案:

- `worldMap`: 地形、格子、建筑底图、静态缓存。
- `worldFog`: 战争迷雾、遮罩，视觉层，不拥有玩法输入。
- `worldActor`: 单位、行军动画、行军箭头、选中圈，全部是世界坐标对象。
- `mainHud`: 按钮、面板、提示、屏幕坐标 UI，不画单位本体。

当前证据:

- `frontend/js/platform/renderers/WorldMapTileMapRenderer.js` 的 `renderWorldTileMapHitTargets()` 仍调用 `addWorldActorHitTargets()`。
- `frontend/js/platform/renderers/WorldMapTileMapRenderer.js` 的 `renderWorldTileMapLayers()` 仍调用 `renderWorldActors()` 和 `addWorldActorHitTargets()`。
- `frontend/js/platform/renderers/WorldMapLayerCanvasRenderer.js` 的 `renderWorldMapActorLayer()` 会调用 `renderWorldActors()`，之后又调用 `renderWorldMarchHud()`。
- `frontend/js/platform/CanvasGameShellWorldMapDragRuntime.js` 已把 `worldMap`、`worldFog`、`worldActor` 三层做相同拖动 transform，这说明项目方向已经承认 `worldActor` 是独立世界层，但 map 层仍在重复绘制 actor。

会发生什么:

- 行军时出现两个小人：一套由 map 层画，一套由 actor 层画。
- 拖动地图时，一套跟着世界 transform 走，另一套可能停在屏幕坐标或被缓存/清屏影响。
- actor hit target 可能重复、优先级错乱或与视觉位置不一致。
- 只测单个渲染器方法容易误判通过，真实运行时会在多层组合后暴露问题。

建议处理:

- map 层只发布 `lastWorldTileMapContext` 和静态 hit targets，不画 actor，不注册 actor target。
- actor 层成为唯一世界坐标 actor 绘制与 actor hit target owner。
- march command HUD 迁回 `mainHud` pass，由 HUD 层消费 actor context。
- 把上述规则写成架构契约测试，并纳入 `npm run test:architecture`。

### P0-002 后端读接口推进并保存状态

成熟方案:

- Query/projection 路径只读，不推进世界时间线，不保存状态。
- Runtime advance 只能在明确的 command、tick 或同步事务中发生。
- 状态推进入口必须有幂等、锁/事务、版本或乐观并发控制。

当前证据:

- `backend/routes/gameRoutes.js` 的 `GET /api/game/state` 会调用 `applyOnlineProgress()`。
- 同一路径随后会同步教程、生成事件、`touchPlayerActiveAt()` 并 `repository.save(gameState)`。
- `backend/server.js` 的后台 tick 每 5 秒查找最近活跃玩家，调用 `advanceRuntimeState()`、城市推进、领土/事件处理并保存。

会发生什么:

- 玩家只是刷新、进入游戏或拉状态，也可能改变教程、事件、探索、城市产出和地图状态。
- 多个请求、后台 tick 和 reset/action 交错时，最后写入者可能覆盖前一个写入者。
- `/api/game/state` 变重后容易拖慢首屏，导致加载 100% 后等待，最终报 504。
- 教程“明明已完成却又触发”的问题可能由读路径推进、状态归一化、reset 失败或旧状态覆盖共同造成。

建议处理:

- 明确拆分 `queryGameStateProjection()` 和 `advanceAndCommitRuntimeState()`。
- `/api/game/state` 默认只读；如确需同步推进，应改成命令式 endpoint 或 server tick，并带事务和版本号。
- 背景 tick、action、reset、state sync 统一经过 repository transaction 或 unit-of-work。

### P0-003 持久化缺少事务和版本边界

成熟方案:

- 对同一玩家状态的修改必须有事务边界。
- 主状态表与共享世界表要同事务提交。
- 大 JSON 存档要有 schema version、migration history 和乐观并发版本。

当前证据:

- `backend/repositories/GameStateRepository.js` 使用 `INSERT OR REPLACE INTO game_states` 保存整份 JSON 状态。
- `save()` 后再调用 `saveSharedWorldTerritories(gameState)` 写共享世界领土。
- 未看到 `BEGIN` / `COMMIT` / `transaction()` 包裹主状态和共享世界写入。
- `findByPlayerId()` 会读取玩家私有 territories 后再 merge shared world territories。

会发生什么:

- 保存过程中某一步失败会留下半提交状态。
- reset 与后台 tick/action 交错时，共享世界表可能残留旧占领信息。
- 多请求同时读同一旧状态并保存时，会出现丢更新。

建议处理:

- repository 提供 `saveAtomic(gameState)`，主状态和共享世界表同事务。
- 增加 `revision` 或 `updatedAt` 乐观并发检查。
- reset 要同时清理该玩家在共享世界中的派生占有记录。

## 3. P1 问题

### P1-001 启动加载链路把资源加载和首包状态混在一起

成熟方案:

- 启动阶段明确拆分 boot、auth、manifest、asset preload、first state sync、scene ready。
- 每个阶段都输出耗时、进度、失败原因。
- 首包 API 必须有 timeout、retry/backoff 和用户可理解的错误状态。

当前证据:

- `frontend/auth.js` 中 `startAuthenticatedSession()` 先 `waitForAuthenticatedAssets()`，最后才 `game.startHeartbeat()`。
- `frontend/js/platform/CanvasGameAppStateSync.js` 中 `startHeartbeat()` 先 `syncOnce()` 拉 `/game/state`，成功后才启动 heartbeat。
- `frontend/js/api/GameAPI.js` 使用 `fetch()`，没有显式 `AbortController` timeout、重试或退避。

会发生什么:

- 资源加载到 100% 后，用户仍会等待首包状态；如果后端慢，就表现为“100% 卡住”。
- Gateway Timeout 会直接暴露为 504，前端 UI 往往只显示“请求失败”。
- 排查时容易误以为是资源问题，实际瓶颈在首包状态/API。

建议处理:

- H5LoadTrace 继续保留，但阶段命名要区分 `assets:preload` 和 `state:first-sync`。
- GameAPI 增加超时、请求 id、耗时、重试策略和结构化错误。
- 首屏 loading 文案和控制台日志都要显示当前阶段。

### P1-002 轮询频率高且缺少失败退避

成熟方案:

- heartbeat、state refresh、version check 分频管理。
- 网络失败时指数退避，避免把后端故障放大。
- `/version` 可使用部署 manifest 或 ETag，不必 5 秒无条件打一次。

当前证据:

- `frontend/js/config/GameConfig.js` 中 `HEARTBEAT_INTERVAL_MS = 1000`，`UPDATE_CHECK_INTERVAL_MS = 5000`。
- `frontend/js/services/UpdateChecker.js` 启动后定时调用 `getVersion()`。
- `frontend/js/services/GameStateSync.js` heartbeat 后还可能在世界探索节点临近时触发 `getState()`。

会发生什么:

- 后端慢或 504 时，客户端继续打 heartbeat、state、version，故障会被放大。
- 玩家控制台会连续出现 504，很难定位根因。
- 小规模开发服也可能被轮询拖慢。

建议处理:

- 网络失败后 version check 和 heartbeat 使用退避。
- 首包状态失败时暂停非关键轮询。
- 给 `/version` 增加轻量缓存或 deploy manifest 直读策略。

### P1-003 管理接口缺少角色边界

成熟方案:

- 登录认证和管理员授权分开。
- 配置导入、回滚、模板下载等 admin 能力必须校验角色、审计来源，并限制请求体大小和频率。

当前证据:

- `backend/routes/adminRoutes.js` 所有 admin endpoint 只使用 `authMiddleware`。
- 白名单普通测试账号登录后理论上即可访问任务定义预览、导入、回滚接口。

会发生什么:

- 开发服测试账号可修改配置，污染后续测试结论。
- 如果部署到更开放环境，会有配置篡改风险。

建议处理:

- 增加 `adminMiddleware` 或账号角色表。
- 管理操作记录 operator、source、diff、rollback id。
- 生产环境默认关闭或限制 admin import。

### P1-004 部署和 CI 没有形成完整闭环

成熟方案:

- GitHub 或服务器侧必须在部署前跑同一套架构门禁。
- 推送、部署、版本提示、PM2 重启、健康检查、回滚都有明确证据。

当前证据:

- 本地未发现 `.github` workflow。
- `deploy.sh` 已有部署保护、健康检查和版本写入，是好的基础。
- `scripts/post-receive.sample` 说明 private bare repo push 可以触发部署，但本地不能证明服务器 hook 当前一定安装。
- `scripts/run-architecture-smoke.js` 是手工白名单，新增架构测试需要人工加入。

会发生什么:

- GitHub push 与服务器部署关系容易被误判。
- 新测试或新契约如果没有进入白名单，门禁可能误绿。
- 线上版本更新提示能说明部署版本变化，但不能证明完整 CI 已通过。

建议处理:

- 增加 CI workflow 或服务器 pre-deploy gate。
- `test:architecture` 至少自动发现 `*.contract.test.js` 或集中 contract tests。
- 部署日志和当前 deployed commit 写入固定位置，并在文档中记录查询命令。

## 4. P2 问题

### P2-001 文档与代码事实不一致

成熟方案:

- 权威架构文档、责任索引、代码实现、测试门禁必须同口径。
- 发现代码进入新架构层时，应同步更新权威文档，而不是只更新进度记录。

当前证据:

- `docs/current_technical_architecture_2026-06-09.md` 的物理画布栈只列 `worldMap`、`worldFog`、`mainHud`。
- `frontend/js/platform/CanvasLayerRegistry.js` 已明确存在 `worldActor`，且物理层顺序为 `worldMap`、`worldFog`、`worldActor`、`mainHud`。
- `docs/architecture_module_responsibility_index_2026-06-08.md` 的更新记录声称 map-home march HUD 已回到 `mainHud` pass，但当前代码仍存在 actor/world layer 调用 HUD 的路径。

会发生什么:

- 后续修复者不知道以哪个文档为准。
- 测试可能按旧文档写，代码按新结构走，导致门禁不稳定。

建议处理:

- 更新 `current_technical_architecture_2026-06-09.md`，把 `worldActor` 纳入物理画布栈。
- 同步修正责任索引中过时的“已完成”描述。
- 文档中明确当前未完成项，不把 candidate 写成 stable。

### P2-002 前端入口仍是手写 script 顺序和手写 cache busting

成熟方案:

- 使用 bundler/manifest 管理依赖顺序、hash、chunk 和缓存。
- 如果暂不引入 bundler，也应生成 script manifest 并测试依赖拓扑。

当前证据:

- `frontend/index.html` 手写约 190 个 script 标签和 `?v=` 参数。
- 多个版本参数来自历史功能名，例如 `world-march-continuous-v3`、`canvas-reset-confirm-v1` 等。

会发生什么:

- 顺序依赖靠人工维护，新增模块容易漏加载或加载晚。
- 文件改了但 query version 未改时，线上缓存可能仍旧。
- 版本提示和真实资源一致性难以严格证明。

建议处理:

- 短期增加 script manifest 校验，检测文件存在、顺序依赖和版本参数。
- 中期迁移到 Vite/Rollup 等构建产物 manifest。

### P2-003 安全和账号存储仍是开发期方案

成熟方案:

- JWT secret 必须来自环境变量，缺失时生产启动失败。
- CORS 按域名限制。
- 前端不应明文保存密码。

当前证据:

- `backend/server.js` 存在默认 JWT secret。
- `backend/server.js` CORS 使用 `origin: '*'`。
- `frontend/js/ui/H5AuthStorageAdapter.js` 在记住密码时保存 `cf_remember_password`。

会发生什么:

- 开发服风险可控，但不能作为线上安全基线。
- 测试账号泄漏或浏览器本地存储被读取时，账号可被复用。

建议处理:

- 生产环境强制 `JWT_SECRET`。
- CORS 配置化，默认只允许正式域名。
- 移除记住密码，只保留记住用户名或使用安全 token。

### P2-004 仓库卫生仍有历史备份文件

成熟方案:

- 备份、数据库、运行时日志不能进入 Git。
- 已进入 Git 的历史备份文件应移出版本控制。

当前证据:

- `git ls-files` 显示 `backend/server.js.bak2` 仍被跟踪。
- `.gitignore` 已经忽略 `backend/*.bak*`，说明当前规则和历史跟踪状态不一致。

会发生什么:

- 审计和搜索时混入旧实现。
- 误读旧备份文件可能导致错误判断。

建议处理:

- 后续单独提交移除已跟踪备份文件。
- 增加仓库卫生检查，阻止 `.bak`、数据库、密钥文件进入 Git。

## 6. 推荐修复顺序

1. 先统一文档和契约：把 `worldActor` 明确写进权威 Canvas 物理层，标明当前 actor/HUD ownership 未完成。
2. 修复 world map 分层：map 层只发布 context，actor 层独占 actor 绘制和 actor hit target，HUD 层独占 command HUD。
3. 把 ownership contract 加入架构门禁，防止再出现两个小人。
4. 拆后端 query/command/tick 边界，禁止普通读接口保存状态。
5. 给 repository save/reset/shared-world 写入加事务和版本边界。
6. 改前端启动链路：资源加载、首包状态、版本检查分阶段可观测，并补 API timeout/backoff。
7. 补 CI/deploy gate，让 GitHub 和服务器部署都必须通过同一套门禁。
8. 更新过时文档和责任索引，避免“文档已完成、代码未完成”的误导。
