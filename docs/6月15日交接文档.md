# 6月15日交接文档

## 目标

把大地图/世界探索从“旧字段、旧入口、旧诊断反复回流”的混沌状态，推进到可过头部大厂技术评审的结构：坐标身份统一、服务端权威、多人同步事实明确、前端日志可复现、旧 UI/旧逻辑不能隐藏回归。

当前最核心的技术目标是：所有 world-map / world-march / scout / reveal / AOI / timeline / diagnostic 链路里，只要记录带 `q/r` 或 `x/y` 坐标，就必须由坐标生成 canonical tile identity；旧 `tileId` / `id` 只能在没有坐标的历史摘要里作为兼容事实，不能覆盖坐标事实。

## 下一步建议

继续用“一刀一红测”的节奏清理剩余旧身份入口，不要批量推断：

1. 搜索活动源码中仍存在的 `tileId ||`、`tile.id ||`、`source.id || tile_`、`revealedTileIds` 直接透传点。
2. 每次只选一个边界明确的模块，先写红测证明旧字段能污染事实，再修复，再更新文档。
3. 优先级建议：
   - 继续优先看 `revealedTileIds` 直接透传点，尤其是 presenter fallback、diagnostic summary、legacy scout report / trail 写入。
   - 多人同步链路：timeline / AOI / worker / presence 相关输出必须只消费服务端归一化事实。
   - 前端 picking/render 链路：不能让 renderer hitTarget 或旧 action payload 重新成为玩法输入权威。
   - 后端 legacy scout 链路：所有 scout report / revealArea / trail 写入必须坐标派生。

## 不要踩的坑

- 不要“隐藏旧入口”当作删除。旧页面、旧 renderer、旧 action 如果退役，必须删除或由 `check-retired-legacy-code.js` 锁死。
- 不要因为当前项目里有旧实现，就迁就旧实现当正确方案。目标是服务端权威、多端可同步、日志可复现、渲染不持有玩法事实。
- 不要批量删引用链。先查调用者和当前游戏行为；不明确是否仍有用时，先问，不要自己猜。
- 不要用 PowerShell 乱码文本当 patch 上下文。中文文档用 UTF-8 读取，patch 尽量用 ASCII 锚点。
- 不要把 renderer/native event/Promise/runtime object 写进 `ClientOperationLog` 或 `WorldMarchTrace`。日志只能是 compact serializable evidence。
- 不要改完不提交不推送。用户需要前端看到更新，必须保持本地、GitHub、服务器三端一致。

## 交接判断标准

这条线不是“功能看起来能跑”就结束。完成标准是：

- 有红测证明旧字段曾经能污染事实。
- 修复后红测转绿。
- 相关模块进入 architecture smoke。
- 文档写清楚模块责任、扩展方式、回归命令。
- 本地、GitHub、服务器部署 commit 一致。
- `/api/health` 能确认服务器已部署到同一 commit。
