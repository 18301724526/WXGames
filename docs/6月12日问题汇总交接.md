## 当前只读定位事实

### 行军相关绘制入口

目前行军相关至少有 4 类入口，不是单一绘制点：

1. 行军小人 sprite 与行军箭头

   - `frontend/js/platform/renderers/WorldActorCanvasRenderer.js`
   - `renderActors()` 对每个 actor 调用：
     - `getActorScreenPoint()`
     - `getActorTargetScreenPoint()`
     - `drawMarchArrow()`
     - `drawActorUnit()`
     - `addActorHitTarget()`
   - `drawActorUnit()` 通过 `TutorialIntroUnitRenderer.renderUnit()` 或 `ctx.drawImage()` 画单位。

2. actor/HUD 组合代理

   - `frontend/js/platform/renderers/WorldMapActorHudRenderer.js`
   - `renderWorldActors()` 只是转发到 `worldActorRenderer.renderActors()`。
   - `renderWorldMarchHud()` 只是转发到 `WorldMarchHudCanvasRenderer`。

3. 选中行军单位后的 HUD

   - `frontend/js/platform/renderers/WorldMarchHudCanvasRenderer.js`
   - `renderActorHud()` 使用 `WorldMarchSystem.getTileScreenCenter(actor.current || actor.origin, viewport, geometry)` 重新算位置，再绘制按钮和 hit target。

4. 地图/actor layer 入口

   - `frontend/js/platform/renderers/WorldMapLayerCanvasRenderer.js`
   - `renderWorldMapActorLayer()` 会：
     - `beginFrame()`
     - `setHitTargets([])`
     - `clearAll()`
     - clip 到 `frame`
     - 调用 `renderWorldActors(actors, viewport, geometry)`
     - 调用 `addWorldActorHitTargets(actors, viewport, geometry)`
     - 发布 `lastMapHomeWorldHudContext`

### canvas/layer 结构

物理图层顺序来自 `frontend/js/platform/CanvasLayerRegistry.js`：

- `worldMap`
- `worldFog`
- `worldActor`
- `mainHud`

`H5CanvasRuntime.setLayerTranslate()` 对具体 canvas 设置 CSS transform：

- `canvas.style.transform = translate3d(x, y, 0)`

拖动期间 `CanvasGameShellWorldMapDragRuntime.updateWorldMapDragCompositor()` 会尝试对三层一起平移：

- `worldMap`
- `worldFog`
- `worldActor`

`mainHud` 不在这个 CSS transform 列表中。也就是说 HUD 如果正常移动，很可能不是因为 DOM/canvas 图层被平移，而是 HUD 每帧根据当前 `viewport + geometry + actor.current` 重新算出来。

### hit target 为什么可能正常

`WorldMapRuntime.syncHitTargetsFromRenderer()` 会合并：

- `renderer.hitTargets`
- `renderer.worldActorLayerRenderer.hitTargets`

并把它们保存为 `baseHitTargets`。

拖动期间 `WorldMapRuntime.setDragLayerOffset()` 会把 `dragLayerOffset` 应用到 hit targets：

- `this.hitTargets = this.getOffsetHitTargets()`
- `getOffsetHitTargets()` 内部调用 `WorldMapRuntimeCameraPolicy.applyOffsetToHitTargets()`

因此，点击目标可以跟随拖动，不必证明 sprite 像素也跟随。点击目标链路有自己的 offset 机制。

### HUD 为什么可能正常

`WorldMarchHudCanvasRenderer.renderActorHud()` 用 actor 的世界/瓦片坐标重新计算屏幕点：

- `getTileScreenCenter(actor.current || actor.origin || {}, viewport, geometry)`

如果它拿到的是当前 runtime/context 的 viewport，HUD 可以跟随。这个路径和 `WorldActorCanvasRenderer.drawActorUnit()` 不是同一个最终绘制 surface，也不是同一套 transform 保障。

## 架构判断

这不是单纯“某个 drawImage 少加 offset”的渲染小错误。更像是地图拖动路径里存在两套并行坐标治理：

1. 地图/actor 可视层：依赖 snapshot、baked camera、CSS layer transform、actor layer redraw。
2. 交互/HUD 层：依赖 runtime hit target offset 和 HUD 按 viewport 重算。

用户现场看到“sprite 不动，但点击/HUD 正常”，正好说明这两套链路没有被一个统一的 camera/screen-space contract 约束。

更严重的可能性有两类：

1. 真正可见的行军小人并不是上一轮测试覆盖的 `renderWorldMapActorLayer -> renderWorldActors -> drawActorUnit` 那条路径，而是另一个缓存层、旧图层或旧 context 留下来的像素。
2. `worldActor` 确实被绘制了，但拖动时 `refreshWorldMapLayerFromSnapshot()` 频繁清 transform、重画 snapshot/actor layer，导致 actor layer 被重绘到旧的 screen-space，而 hit target/HUD 仍按 runtime offset 更新。

当前证据不足以在不跑真实浏览器像素测试的情况下二选一，因此不要继续直接补丁。

## 为什么上一轮测试会绿但线上仍失败

上一轮测试主要验证：

- actor layer render 被调用；
- `worldMapRuntimeContext` 能从 snapshot/full render 传到 actor layer；
- hit targets 能从 renderer 同步到 runtime；
- 架构文档/模块边界检查通过。

它没有验证：

- 真实浏览器里 `worldActor` canvas 的像素是否跟 `worldMap` canvas 同步移动；
- `drawActorUnit()` 最终落在哪个可见 canvas 上；
- 拖动中 CSS transform、clearTransform、snapshot redraw 的时序是否互相抵消；
- 同一 actor 的 sprite 像素、hit target、HUD 三者在拖动前后是否保持同一 screen-space 关系。

所以测试通过不等于用户看到的小人已经移动。这个结论必须交给下一轮 TDD 修正。

## 下一步必须先写的红测

不要先改代码。建议先补一个能失败的浏览器/渲染层测试，目标是证明真实视觉路径失效。

最小测试思路：

1. 启动真实游戏页面，进入有行军 actor 的地图。
2. 在 actor 当前屏幕位置附近放一个可检测视觉锚点，或者读取 `worldActor` canvas 上 actor 像素 bounding box。
3. 记录拖动前：
   - 地图 tile 锚点 screen 坐标；
   - actor sprite screen 坐标；
   - actor hit target 坐标；
   - actor HUD 坐标。
4. 模拟拖动地图，例如水平拖动 80px。
5. 断言：
   - 地图 tile 锚点发生同方向 screen 位移；
   - actor sprite 与地图 tile 锚点保持相同位移；
   - actor hit target 与 actor sprite 保持同一 screen-space；
   - actor HUD 与 actor sprite 保持同一 screen-space。
6. 如果“hit target/HUD 变了但 sprite 像素没变”，测试必须红。

用户提出的简化版也可作为第一步：

- 游戏运行时在中心点放一个锚点；
- 移动地图；
- 移动后判断锚点与行军小人的屏幕相对坐标是否一致；
- 不一致就红。

但要注意：只测 JS 返回坐标不够，必须至少验证一次真实 canvas 像素或可见 layer transform，否则仍可能重复上一轮“逻辑绿、视觉失败”的问题。

## 建议的成熟治理方案

下一轮修复应先建立统一 contract，再改实现：

1. 定义唯一 camera source of truth。

   - `WorldMapRuntime.camera` 或等价 camera state 应是唯一世界到屏幕映射输入。
   - snapshot/baked camera 只能是性能缓存，不应该成为另一套事实来源。

2. 定义坐标空间类型。

   - world/tile coordinate
   - map layer logical coordinate
   - screen/HUD coordinate
   - CSS transformed physical canvas coordinate

3. 禁止 renderer 自己猜 offset。

   - actor sprite、tile、fog、hit target、HUD 必须从同一个 projection 函数或同一个 render context 派生。
   - `applyOffsetToHitTargets()` 这种补偿逻辑需要被纳入 contract，不能只给 hit target 单独享有。

4. 建立 visual parity test。

   - 每次拖动相关改动都必须跑 sprite/hit/HUD 三链路一致性测试。
   - 只验证调用参数和 context 传递不够。

5. 明确 layer transform 策略。

   - 要么拖动期间所有地图可视层使用 CSS transform，并且 actor sprite 在同一个 transformed layer。
   - 要么所有内容都按 camera 重绘，不依赖 CSS transform。
   - 不应长期混用“地图 CSS transform + HUD 重算 + hit target offset + actor snapshot redraw”而没有边界说明。

## 下一台 Codex 接手建议

1. 先不要改渲染代码。
2. 先写红测，证明用户看到的“sprite 不动但 hit/HUD 正常”。
3. 用测试定位真实可见 sprite 来自哪个 canvas：
   - `worldActor`
   - `mainHud`
   - 旧 backbuffer/snapshot
   - 或其他 legacy path
4. 用浏览器 devtools/Playwright 检查拖动过程中：
   - `worldMap` canvas transform
   - `worldFog` canvas transform
   - `worldActor` canvas transform
   - `mainHud` canvas transform
   - 每层 canvas 是否被 clear/repaint
5. 再决定修复方向：
   - 如果 sprite 实际在 `worldActor`，优先修正 actor layer 与 map layer transform/redraw 时序；
   - 如果 sprite 实际在 `mainHud` 或旧路径，先删除/迁移重复绘制点；
   - 如果是 context stale，修正 projection source，而不是继续局部加 offset。
