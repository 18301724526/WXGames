# 文明火种 — 战争系统设计草案 v0.1

> 状态：基础概念草案，需与当前已实现的古典时代主动出征能力一起理解。
> 日期：2026-05-18

---

## 一、设计目标

战争系统第一阶段不直接做完整战争，而是先建立“城邦开始武装自己，并在边境时代面对外部压力”的基础循环。当前代码已经在古典时代落地了基础主动出征与世界点占领，因此本文档主要用于约束“士兵、训练、防御和威胁事件”的底层规则，不再代表“完全没有主动出征”。

核心体验：

```text
建造兵营
→ 兵营自动训练士兵
→ 进入边境时代并建造瞭望台
→ 士兵和瞭望台提供防御
→ 外部威胁事件检验防御
→ 防御成功获得奖励，防御失败承受轻微损失
```

边境时代阶段的目标不是让玩家马上出征，而是让玩家看到“兵力正在增长”，并理解兵营与防御压力之间的关系；真正的基础主动出征已经在古典时代世界系统中启用。

时代4与威胁事件的落地设计见 `docs/时代4与军事威胁系统设计_v0.1.md`。本文保留为战争系统基础概念与边界草案。

## 二、术语统一

### 2.1 单位名称

统一使用：**士兵**

不再区分：
- 守卫
- 战士
- warrior
- guard

原因：
- 玩家理解成本最低：兵营产士兵，士兵打仗。
- 当前阶段只需要一个军事单位。
- 后续进攻、防守、出征都可以沿用同一批士兵，不需要重新迁移概念。

### 2.2 士兵定位

| 项 | 设计 |
|------|------|
| 来源 | 兵营自动训练 |
| 当前用途 | 提供基础防御值，并作为古典时代出征与占领的人力来源 |
| 后续用途 | 出征、驻守、威胁事件、战斗结算 |
| 是否细分兵种 | 第一版不细分 |
| 是否占普通人口 | 第一版不占 |

第一版士兵不属于人口职业，不从农民/学者/工匠中分配，也不占人口上限。这样可以避免战争系统刚接入时立刻打乱已有资源生产节奏。边境时代新增瞭望台作为建筑侧防御来源，让新增建筑有明确用途。

## 三、兵营设计

### 3.1 兵营新定位

兵营从“静态加成建筑”调整为“自动训练士兵的军事建筑”。

当前版本兵营已有：
- 建筑 id：`barracks`
- 解锁时代：城邦时代（时代3）
- 建造页显示
- 独立军事标签页显示士兵、防御与训练进度

后续实现时，兵营卡片应表达：

```text
每 X 秒训练 1 名士兵
士兵上限 +Y
当前士兵 Z / Y
```

### 3.2 推荐数值

| 兵营等级 | 训练间隔 | 士兵上限 | 防御换算 |
|------|------:|------:|------|
| Lv1 | 30秒 / 1士兵 | 5 | 每名士兵 +1 防御 |
| Lv2 | 20秒 / 1士兵 | 10 | 每名士兵 +1 防御 |

当前 `barracks.maxLevel` 为 2，所以第一版只需要支持 Lv1 和 Lv2。

### 3.3 是否消耗资源

第一版建议：**训练士兵不消耗资源**。

理由：
- 城邦时代刚开放战争概念，先让玩家建立反馈感。
- 如果训练还消耗食物/木材/知识，容易和刚完成的城邦进阶资源压力叠在一起。
- 后续战争时代可以再引入“维持费”“装备消耗”“训练消耗”。

## 四、军事状态

推荐新增后端状态：

```js
military: {
  soldiers: 0,
  soldierCap: 5,
  trainingProgress: 0,
  defense: 0
}
```

字段说明：

| 字段 | 含义 |
|------|------|
| `soldiers` | 当前士兵数量 |
| `soldierCap` | 当前士兵上限，由兵营等级计算 |
| `trainingProgress` | 当前训练进度，单位秒 |
| `defense` | 当前防御值，第一版等于士兵数量 × 每兵防御 |

派生规则：

```text
soldierCap = 兵营等级对应上限
trainingInterval = 兵营等级对应训练间隔
defense = soldiers × defensePerSoldier
threatDefense = watchtowerLevel × 2
威胁检定总防御 = defense + threatDefense
```

如果没有建造兵营：

```js
military: {
  soldiers: 0,
  soldierCap: 0,
  trainingProgress: 0,
  defense: 0
}
```

## 五、训练规则

### 5.1 自动训练

当玩家已建造兵营，且士兵未达上限时：

```text
trainingProgress += elapsedSeconds
每达到 trainingInterval：
  soldiers += 1
  trainingProgress -= trainingInterval
```

当士兵达到上限：

```text
trainingProgress = 0
不再继续累计
```

### 5.2 离线收益

离线期间也应训练士兵。

第一版可以直接按离线秒数计算：

```text
可训练数量 = floor((trainingProgress + offlineSeconds) / trainingInterval)
soldiers = min(soldierCap, soldiers + 可训练数量)
trainingProgress = 未满上限时的余数；满上限时归零
```

士兵训练不受离线收益效率 80% 影响。因为它不是资源收益，而是建筑生产进度。

## 六、前端展示

### 6.1 建造页兵营卡片

兵营卡片显示：

```text
士兵 3 / 5
训练进度 18 / 30秒
防御值 3
```

按钮区域仍然保留建造/升级逻辑。

### 6.2 军事标签页

当前版本已有独立军事标签页。第一版展示：

```text
军事
士兵：3 / 5
防御：5
下一名士兵：18 / 30秒
```

士兵不放入顶部资源摘要栏，避免资源与军事概念混杂。

## 七、威胁事件

### 7.1 第一版事件类型

边境时代的战争事件只做防御型威胁事件。威胁事件复用事件队列，类型为 `threat`。主动出征与世界点占领已经在古典时代由 `TerritoryService` 负责，本文不重复展开地图与占领 UI。

示例：

| 事件 | 触发时代 | 推荐防御需求 | 成功结果 | 失败结果 |
|------|------|------:|------|------|
| 边境试探 | 边境时代 | 2 | 获得少量食物/知识 | 损失少量食物 |
| 夜袭预警 | 边境时代 | 4 | 获得木材/知识 | 损失少量木材 |
| 盗匪勒索 | 边境时代 | 3 名士兵 | 获得食物 | 损失少量食物 |

第一版失败惩罚必须轻，不应导致文明倒退或卡死。

### 7.2 检定规则

最简单版本：

```text
如果 (士兵防御 + 瞭望台边境防御) >= event.requiredDefense：
  成功
否则：
  失败
```

暂不建议加入随机数。早期系统需要让玩家明确知道“士兵更多就更安全”。

## 八、后续扩展边界

边境时代这一层仍不做：
- 敌方单位
- 战斗动画
- 士兵兵种细分
- 人口职业式军事分配
- 金属资源
- 装备/武器系统
- 复杂伤亡公式

后续可以扩展为：

```text
士兵总数 = 可用士兵 + 出征中士兵
可用士兵提供防御
出征中士兵参与战斗，出征期间不提供本城可用兵力
```

但第一版先不拆驻守/出征，全部士兵默认参与防御。

## 九、实现拆解建议（后续）

后续真正开发时，建议拆成以下小步：

1. 新增 `military` 状态与 normalize 兼容。
2. 新增兵营效果计算：士兵上限、训练间隔、防御换算。
3. 在资源 tick 或独立 military tick 中推进训练。
4. 客户端状态返回 `military`。
5. 建造页兵营卡片展示士兵与训练进度。
6. 文明页展示军事状态区。
7. 新增第一批防御型威胁事件。
8. 为训练、离线训练、事件检定补测试。

## 十、当前结论

战争系统 v0.1 的核心结论：

```text
兵营自动训练士兵。
士兵是唯一军事单位。
士兵第一版不占普通人口。
士兵第一版只提供防御。
边境时代只做防御压力事件。
古典时代已经开放侦察、世界点占领和基础主动出征。
更复杂的兵种、装备、外交和长期战争仍留待后续版本。
```
