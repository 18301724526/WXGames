# 文明火种 — 战争系统设计草案 v0.1

> 状态：设计草案，仅用于讨论和后续实现拆解。本文档不代表当前版本已实现。
> 日期：2026-05-17

---

## 一、设计目标

战争系统第一版不直接做完整战争，而是先建立“城邦开始武装自己”的基础循环。

核心体验：

```text
建造兵营
→ 兵营自动训练士兵
→ 士兵提供防御
→ 外部威胁事件检验防御
→ 防御成功获得奖励，防御失败承受轻微损失
```

第一版目标不是让玩家马上出征，而是让玩家看到“兵力正在增长”，并理解兵营与防御压力之间的关系。

---

## 二、术语统一

### 2.1 单位名称

统一使用：**士兵**

不再区分：
- 守卫
- 战士
- warrior
- guard

原因：
- 玩家理解成本最低：兵营产士兵，士兵打仗。
- 当前阶段只需要一个军事单位。
- 后续进攻、防守、出征都可以沿用同一批士兵，不需要重新迁移概念。

### 2.2 士兵定位

| 项 | 设计 |
|------|------|
| 来源 | 兵营自动训练 |
| 当前用途 | 提供防御值 |
| 后续用途 | 出征、驻守、威胁事件、战斗结算 |
| 是否细分兵种 | 第一版不细分 |
| 是否占普通人口 | 第一版不占 |

第一版士兵不属于人口职业，不从农民/学者/工匠中分配，也不占人口上限。这样可以避免战争系统刚接入时立刻打乱已有资源生产节奏。

---

## 三、兵营设计

### 3.1 兵营新定位

兵营从“静态加成建筑”调整为“自动训练士兵的军事建筑”。

当前版本兵营已有：
- 建筑 id：`barracks`
- 解锁时代：城邦时代（时代3）
- 建造页显示，不新增军事标签页

后续实现时，兵营卡片应表达：

```text
每 X 秒训练 1 名士兵
士兵上限 +Y
当前士兵 Z / Y
```

### 3.2 推荐数值

| 兵营等级 | 训练间隔 | 士兵上限 | 防御换算 |
|------|------:|------:|------|
| Lv1 | 30秒 / 1士兵 | 5 | 每名士兵 +1 防御 |
| Lv2 | 20秒 / 1士兵 | 10 | 每名士兵 +1 防御 |

当前 `barracks.maxLevel` 为 2，所以第一版只需要支持 Lv1 和 Lv2。

### 3.3 是否消耗资源

第一版建议：**训练士兵不消耗资源**。

理由：
- 城邦时代刚开放战争概念，先让玩家建立反馈感。
- 如果训练还消耗食物/木材/知识，容易和刚完成的城邦进阶资源压力叠在一起。
- 后续战争时代可以再引入“维持费”“装备消耗”“训练消耗”。

## 四、军事状态

推荐新增后端状态：

```js
military: {
  soldiers: 0,
  soldierCap: 5,
  trainingProgress: 0,
  defense: 0
}
```

字段说明：

| 字段 | 含义 |
|------|------|
| `soldiers` | 当前士兵数量 |
| `soldierCap` | 当前士兵上限，由兵营等级计算 |
| `trainingProgress` | 当前训练进度，单位秒 |
| `defense` | 当前防御值，第一版等于士兵数量 × 每兵防御 |

派生规则：

```text
soldierCap = 兵营等级对应上限
trainingInterval = 兵营等级对应训练间隔
defense = soldiers × defensePerSoldier
```

如果没有建造兵营：

```js
military: {
  soldiers: 0,
  soldierCap: 0,
  trainingProgress: 0,
  defense: 0
}
```

## 五、训练规则

### 5.1 自动训练

当玩家已建造兵营，且士兵未达上限时：

```text
trainingProgress += elapsedSeconds
每达到 trainingInterval：
  soldiers += 1
  trainingProgress -= trainingInterval
```

当士兵达到上限：

```text
trainingProgress = 0
不再继续累计
```

### 5.2 离线收益

离线期间也应训练士兵。

第一版可以直接按离线秒数计算：

```text
可训练数量 = floor((trainingProgress + offlineSeconds) / trainingInterval)
soldiers = min(soldierCap, soldiers + 可训练数量)
trainingProgress = 未满上限时的余数；满上限时归零
```

士兵训练不受离线收益效率 80% 影响。因为它不是资源收益，而是建筑生产进度。

## 六、前端展示

### 6.1 建造页兵营卡片

兵营卡片显示：

```text
士兵 3 / 5
训练进度 18 / 30秒
防御值 3
```

按钮区域仍然保留建造/升级逻辑。

### 6.2 文明页或资源顶部详情

第一版不新增军事标签页。

推荐在文明页增加一个小型军事状态区：

```text
军事
士兵：3 / 5
防御：3
下一名士兵：18 / 30秒
```

如果未来资源顶部 bar 的详情窗承载全局状态，也可以把士兵作为“非资源状态”放在详情窗下方，但不建议放进顶部资源摘要栏，避免资源与军事概念混杂。

## 七、威胁事件

### 7.1 第一版事件类型

战争第一版只做防御型威胁事件，不做主动出征。

示例：

| 事件 | 触发时代 | 推荐防御需求 | 成功结果 | 失败结果 |
|------|------|------:|------|------|
| 边境试探 | 城邦时代 | 2 | 获得少量食物/知识 | 损失少量食物 |
| 夜袭预警 | 城邦时代 | 4 | 获得木材/知识 | 损失少量木材 |
| 部落围观 | 城邦时代后段 | 6 | 获得声望类预留奖励 | 士兵损失 1 |

第一版失败惩罚必须轻，不应导致文明倒退或卡死。

### 7.2 检定规则

最简单版本：

```text
如果 defense >= event.requiredDefense：
  成功
否则：
  失败
```

暂不建议加入随机数。早期系统需要让玩家明确知道“士兵更多就更安全”。

## 八、后续扩展边界

第一版不做：
- 主动出征
- 敌方单位
- 战斗动画
- 士兵兵种细分
- 军事标签页
- 人口职业式军事分配
- 金属资源
- 装备/武器系统
- 复杂伤亡公式

后续可以扩展为：

```text
士兵总数 = 驻守士兵 + 出征士兵
驻守士兵提供防御
出征士兵参与战斗，出征期间不提供防御
```

但第一版先不拆驻守/出征，全部士兵默认参与防御。

## 九、实现拆解建议（后续）

后续真正开发时，建议拆成以下小步：

1. 新增 `military` 状态与 normalize 兼容。
2. 新增兵营效果计算：士兵上限、训练间隔、防御换算。
3. 在资源 tick 或独立 military tick 中推进训练。
4. 客户端状态返回 `military`。
5. 建造页兵营卡片展示士兵与训练进度。
6. 文明页展示军事状态区。
7. 新增第一批防御型威胁事件。
8. 为训练、离线训练、事件检定补测试。

---

## 十、当前结论

战争系统 v0.1 的核心结论：

```text
兵营自动训练士兵。
士兵是唯一军事单位。
士兵第一版不占普通人口。
士兵第一版只提供防御。
城邦时代只做防御压力，不做主动战争。
战争时代后续再扩展出征、伤亡、装备和更复杂事件。
```
