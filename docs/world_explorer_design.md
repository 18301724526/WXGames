# 地图探索队伍玩法设计

## 目标

把现有的方向侦察，升级为“派出一个小人队伍在地图上移动并解锁地块”的探索玩法。地图仍然保持探索生成：迷雾之外不保存真实地形，只有当路线需要经过或揭示某个坐标时，服务端才按世界种子生成地块并写入存档。

## 核心原则

- 服务端拥有探索任务、路线、时间进度和地块生成的权威状态。
- 前端只负责输入、动画播放、地图增量合并和表现，不在本地决定真实地形。
- 迷雾之外没有真实 tile 数据，不下发地形、据点和资源信息。
- 每次探索只返回增量 tile，避免整张地图重建影响流畅度。
- 旧的方向侦察接口先保留，新的探索队伍可以并行接入，降低迁移风险。

## 新账号出生地图

新账号不再只生成首都单格，而是生成首都周围半径 2 的已知区域：

- `0,0`：首都地块，`visibility = controlled`。
- 半径 1-2：近郊地块，`visibility = scouted`。
- 半径 1：安全陆地圈，避免海洋或河流贴脸。
- 半径 3 及更远：不生成 tile，前端按缺失 tile 绘制战争迷雾。

这让玩家进入游戏时能看到首都和近郊，但不会提前泄露外部地图。

## 探索任务

新增 `exploreMissions` 存档字段，每个任务保存：

- `id`：任务 ID。
- `mode`：`random` 或 `manual`。
- `status`：`active`、`ready`、`cancelled`。
- `origin`：出发坐标和城市/据点信息。
- `route`：服务端确认后的不重复路线。
- `revealedTileIds`：已经揭示过的 tile。
- `stepDurationSeconds`：每步移动耗时。
- `startedAt`、`nextStepAt`、`completesAt`：时间推进字段。

### 随机探索

随机探索由服务端生成路线：

- 从出发点开始。
- 优先选择未探索、靠近迷雾边缘的相邻格。
- 本次路线内不重复经过同一格。
- 每条路线限制步数，默认先用 8 步。
- 如果遇到死路，允许选择次优未知邻居或提前结束。

### 手动探索

手动探索由前端提交目标点：

- 服务端校验目标合法。
- 目标必须在合理探索距离内。
- 服务端用简单路径算法生成路线。
- 路线上的真实地形仍由服务端生成。

第一版手动探索可以先限制为直线/折线路线，后续再升级为避障寻路。

## 地块揭示

探索队伍移动时，服务端按任务进度揭示地块：

- 每到达路线上的一步，就生成并揭示当前坐标。
- 可选：每步揭示半径 1，用于更宽的视野。
- 响应返回 `newTiles` 和更新后的 mission。
- 前端把 `newTiles` 合并到已有 `worldMap.tiles`。

## 前端流畅度策略

- 队伍动画由前端本地按路线播放，不依赖每帧网络请求。
- 服务端只在开始任务、轮询任务状态、任务到达下一步时返回增量。
- 地图缓存只因新增 tile 失效，不做整张地图重建。
- 小人移动、迷雾边缘、已揭示 tile 可以分别作为表现层处理。

## 分阶段落地

1. 文档和步骤记录。
2. 服务端出生地图半径 2 初始化。
3. 服务端探索任务模型、随机路线和手动路线。
4. 服务端行动接口和客户端状态输出。
5. 前端 API 和基础按钮入口。
6. 前端小人路线表现、增量渲染优化。
7. 逐步替换旧方向侦察入口。

第一轮开发先完成 1-5，让探索玩法可以从接口和基础 UI 跑通。
