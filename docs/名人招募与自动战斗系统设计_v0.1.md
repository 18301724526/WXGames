# 名人招募与自动战斗系统设计 v0.1

更新时间：2026-05-25 02:06:27 +08:00  
代码基线：`0.1.112`，提交 `d7db0b7 Add tech tree zoom gestures`  
状态：设计草案；兵营百人制底座已于 `0.1.113` 落地

## 1. 核心定位

名人系统不是抽卡系统，也不是养卡系统。名人应当是文明发展、事件、寻访和战争扩张过程中自然出现的人物。玩家获得的不是一张卡，而是一个被当前文明经历生成出来的独立人物。

硬约束：

- 不做卡池、十连抽、碎片合成、保底、星级升阶。
- 名人没有等级。强度来自出身、属性、技能组合、经历、状态、带兵规模和适配场景。
- 名人生成后必须写入玩家云端存档，成为该玩家独有的人物。
- 技能由基础效果模块组合生成，但技能名字必须符合实际效果。
- 名人可以作为领队带兵出征，也可以作为城市发展人物派驻。
- 战斗是自动回合制，玩家不在战斗过程中操作。
- 队伍兵力是战斗承伤池，近似等于名人所带队伍的生命值。

## 2. 获得方式

名人来源应当围绕世界行为，而不是消费入口。

第一版建议保留三类来源：

| 来源 | 触发方式 | 玩家感受 | 是否可重复 |
| --- | --- | --- | --- |
| 事件投奔 | 普通事件、特殊事件、威胁事件后续 | 有人被文明吸引而来 | 可重复 |
| 寻访发现 | 消耗时间执行寻访，不消耗付费道具 | 主动寻找人才 | 可重复 |
| 战后归附 | 出征、占领、威胁处理后出现 | 战争和探索带来人物 | 可重复 |

后续可扩展：

- 攻略遗迹后发现隐居者。
- 某类科技解锁对应人才池。
- 城市宜居度或方针影响投奔倾向。
- 特定地理环境影响名人的出身与技能偏向。

## 3. 生成与存档

名人第一次生成时只存在于服务端结果中。玩家选择接纳、完成试炼或触发归附后，后端把名人写入云端存档。

建议存档结构：

```js
famousPeople: [
  {
    id: "fp_...",
    name: "陆骁",
    title: "山道突骑",
    eraBorn: 5,
    source: {
      type: "seek",
      eventId: "seek_border_veteran",
      cityId: "capital",
      territoryId: null,
      seed: "..."
    },
    archetype: "vanguard",
    roles: ["military"],
    attributes: {
      command: 72,
      force: 81,
      strategy: 42,
      governance: 28,
      craft: 16,
      charisma: 55
    },
    traits: ["mountain-born", "reckless"],
    skills: [
      {
        id: "skill_...",
        name: "血刃连袭",
        type: "assault",
        effects: [
          { key: "lifesteal", value: 0.18 },
          { key: "combo", chance: 0.22, times: 1 }
        ]
      }
    ],
    status: {
      assigned: "idle",
      cityId: null,
      missionId: null,
      woundedUntil: null,
      fatigue: 0,
      loyalty: 66
    },
    createdAt: "2026-05-25T02:06:27+08:00",
    generatorVersion: "famous-person-v0.1"
  }
]
```

### 3.1 名人不升级

名人没有等级，因此不显示 Lv、经验条、升星、突破。

可成长内容只保留低养成味的经历变化：

- 经历记录：参与过哪些战斗、事件、城市建设。
- 状态变化：受伤、疲劳、忠诚波动。
- 称号变化：长期行为触发称号替换或追加。
- 技能觉醒可以作为后期系统，但不能做成刷材料升级技能。

第一版不做名人成长，先让“生成、加入、派出、战报”闭环成立。

## 4. 属性模型

建议第一版属性分成六项，兼容发展与战争：

| 属性 | 作用方向 | 第一版用途 |
| --- | --- | --- |
| 统率 command | 带兵、阵型、减损 | 提高可带兵力和队伍稳定性 |
| 武勇 force | 直接伤害 | 提高普通攻击与突击技能伤害 |
| 谋略 strategy | 技能、控制、战术 | 提高计策技能触发和特殊效果 |
| 治理 governance | 城市、民生 | 派驻城市时影响资源或宜居描述 |
| 工巧 craft | 工业、打造 | 后续装备打造、工程建筑 |
| 魅力 charisma | 寻访、归附、士气 | 影响投奔、战后归附和士气 |

第一版实现战斗时可以只使用：

```text
统率：影响最大带兵和减损
武勇：影响物理伤害
谋略：影响策略技能
魅力：影响士气或归附
```

治理、工巧先进入数据结构，后续接城市派驻和打造系统。

## 5. 技能生成

技能不是固定死表，而是基础效果模块随机组合生成。每个技能至少有一个主效果，可以有一个副效果。高稀有组合不靠星级表达，而靠效果组合本身的价值和触发限制表达。

### 5.1 基础效果池

第一版战斗效果：

| key | 效果 | 标签 |
| --- | --- | --- |
| `lifesteal` | 造成伤害后按比例恢复己方兵力 | 血、饮血、回生、噬 |
| `combo` | 普通攻击或技能后概率追加攻击 | 连袭、追击、双斩、疾攻 |
| `counter` | 受到攻击后概率反击 | 反击、回锋、逆斩 |
| `shield` | 回合开始获得临时护盾 | 坚壁、护阵、铁壁 |
| `armorBreak` | 降低敌方防御或减伤 | 破甲、裂阵、摧锋 |
| `burn` | 持续造成伤害 | 火、焚、烈焰 |
| `poison` | 持续削弱或伤害 | 毒、蚀、瘴 |
| `moraleUp` | 提升己方伤害或触发率 | 鼓舞、振军、战歌 |
| `heal` | 恢复兵力 | 治、复、援 |
| `ambush` | 首回合额外伤害或控制 | 伏击、奇袭、暗袭 |

城市发展效果后续单独接入：

| key | 效果 | 标签 |
| --- | --- | --- |
| `foodPolicy` | 派驻时改善粮食相关描述 | 耕、稻、渠 |
| `woodPolicy` | 派驻时改善木材相关描述 | 林、斧、材 |
| `knowledgePolicy` | 派驻时改善知识相关描述 | 书、策、学 |
| `habitabilityPolicy` | 派驻时改善宜居倾向 | 居、安、治 |

### 5.2 技能命名规则

技能名由效果标签生成，不允许与效果脱节。

命名输入：

```text
主效果
副效果
技能类型
名人定位
时代语感
```

示例：

| 效果组合 | 合理名称 |
| --- | --- |
| 吸血 + 连击 | 血刃连袭、饮血追击、噬血双斩 |
| 护盾 + 反击 | 铁壁回锋、守阵逆斩、坚壁反击 |
| 破甲 + 灼烧 | 裂甲火攻、焚阵摧锋 |
| 士气 + 治疗 | 振军回春、战歌复阵 |
| 伏击 + 中毒 | 瘴影伏击、毒刃奇袭 |

命名要求：

- 主效果决定核心词。
- 副效果决定修饰词。
- 军事技能名字偏短、有动作感。
- 治理技能名字偏政策、方略、制度感。
- 不显示“获得吸血并有概率连击”这种程序描述作为技能名。

### 5.3 平衡限制

避免随机组合失控：

- 每个技能最多 2 个核心战斗效果。
- `lifesteal + combo` 可以出现，但数值必须打折。
- 控制类效果第一版不做硬控，只做减伤、减士气或延迟伤害。
- 同一名人第一版最多 1 个主动战斗技能 + 1 个被动特质。
- 技能强度由触发率、回合限制、兵力条件调节。

## 6. 兵营与招募迁移

`0.1.113` 前，兵营仍是个位数士兵上限：

```text
Lv1: 5 士兵
Lv2: 10 士兵
```

这不适合名人带队战斗。新系统需要迁移到百人制。

### 6.1 百人制原则

- 招募或训练士兵的最小有效单位是 100。
- UI 展示用“100、200、500”这种规模，不再用 1、3、6 这种个位数。
- 名人出征至少带 100 士兵。
- 队伍兵力就是自动战斗承伤池。
- 后续战报写“损失 27 士兵”，而不是“损失 1 士兵”。

### 6.2 兵营上限建议

第一版建议：

| 兵营规模 | 士兵上限 | 训练节奏 |
| --- | ---: | --- |
| 未建兵营 | 0 | 无 |
| 兵营 1 | 300 | 每 30 秒训练 10 士兵 |
| 兵营 2 | 600 | 每 25 秒训练 20 士兵 |
| 兵营 3 | 1000 | 每 20 秒训练 30 士兵 |
| 后续规模 | 开放曲线 | 继续提升上限和训练速度 |

说明：

- 这个表不等同于最终数值，只是把单位从“个位数士兵”改成“军队规模”。
- 如果保留建筑开放等级曲线，兵营不能再出现 2 级后完全不成长的问题。
- 时代进阶条件也要同步迁移，例如士兵 3 改成兵力 300，士兵 6 改成兵力 600。

### 6.3 招募方式

“招募”不应像人口招募一样点一下加一个人。军事招募建议改成三种节奏：

| 方式 | 用途 | 第一版建议 |
| --- | --- | --- |
| 兵营自动训练 | 默认补兵 | 保留，但产出单位变成 10/20/30 |
| 手动征募 | 玩家缺兵时主动补充 | 最少 100 兵起 |
| 任务/事件奖励 | 前期引导补齐 | 奖励按百人制发放 |

手动征募第一版可以暂不实现，但要在数据模型里预留：

```text
recruitBatchMin = 100
recruitCost = food + wood/iron
recruitTime = 若干秒
```

如果先不做消耗，则手动征募也不要出现按钮，避免玩家误解。

## 7. 队伍与出征

名人带兵出征时，队伍建议这样表达：

```js
expedition: {
  leaderId: "fp_...",
  soldiers: 300,
  originCityId: "capital",
  targetId: "site_...",
  status: "marching",
  createdAt: "...",
  resolveAt: "..."
}
```

出征规则：

- 每支队伍必须有一名领队名人。
- 每支队伍最低 100 士兵。
- 士兵从当前城市可用兵力中锁定。
- 战斗结束后返回剩余士兵，伤亡从城市兵力中扣除。
- 名人不会死亡，第一版失败后进入疲劳或受伤状态。

第一版为了平滑，可以保留“无名领队”的兼容策略，但新系统 UI 应鼓励玩家选择名人。

## 8. 自动回合制战斗

战斗过程自动结算，前端可播放简化动画，也可以先展示战报。

### 8.1 战斗状态

```js
battle: {
  round: 1,
  maxRounds: 8,
  attacker: {
    leaderId: "fp_...",
    soldiers: 300,
    maxSoldiers: 300,
    morale: 100,
    shield: 0,
    buffs: []
  },
  defender: {
    templateId: "tribe_camp",
    soldiers: 260,
    maxSoldiers: 260,
    morale: 90,
    shield: 0,
    buffs: []
  },
  log: []
}
```

### 8.2 回合流程

```text
战前：应用名人特质、地形、兵力差、士气
每回合开始：处理持续效果、护盾、士气
攻击方行动：普通攻击或技能触发
防守方行动：普通攻击或技能触发
回合结束：检查败退、记录伤亡
达到最大回合：按剩余兵力和士气判定胜负
```

### 8.3 伤害模型草案

第一版保持易调：

```text
基础伤害 = 兵力规模系数 + 名人武勇/谋略修正
最终伤害 = 基础伤害 * 技能倍率 * 士气修正 * 地形修正
实际伤害先扣护盾，再扣兵力
吸血 = 实际造成兵力伤害 * 吸血比例
```

玩家界面不展示公式，只展示：

- 优势明显
- 势均力敌
- 略显吃力
- 风险较高

### 8.4 战报

战报应该能让玩家理解发生了什么，但不堆公式。

示例：

```text
第 1 回合：陆骁率先冲阵，血刃连袭触发，敌军损失 42 士兵，己方恢复 8 士兵。
第 2 回合：敌军守势收缩，己方攻势受阻。
第 3 回合：陆骁再次追击，敌军败退。

结果：胜利
己方：300 -> 247
敌方：260 -> 0
收获：粮食、木材、一次投奔线索
```

## 9. 名人生成模板

第一版不需要做几百个名人，但生成器要有足够可扩展的骨架。

### 9.1 定位模板

| archetype | 定位 | 属性倾向 | 技能倾向 |
| --- | --- | --- | --- |
| `vanguard` | 突击领队 | 武勇、统率 | 连击、吸血、破甲 |
| `guardian` | 防守领队 | 统率、魅力 | 护盾、反击、减损 |
| `tactician` | 谋略领队 | 谋略、统率 | 灼烧、中毒、伏击 |
| `warden` | 城市治理 | 治理、魅力 | 宜居、粮食、秩序 |
| `artisan` | 工巧人才 | 工巧、治理 | 工业、打造、维护 |
| `scholar` | 知识人才 | 谋略、治理 | 知识、研究、寻访 |

### 9.2 来源影响

| 来源 | 倾向 |
| --- | --- |
| 边境威胁后投奔 | 军事、防守、反击 |
| 寻访山地 | 武勇、伏击、采矿/石料相关 |
| 寻访平原 | 农业、统率、粮食相关 |
| 遗迹探索 | 谋略、知识、特殊技能 |
| 部落归附 | 魅力、统率、混合技能 |

## 10. 与现有系统的连接

### 10.1 事件系统

新增事件结果类型：

```js
{ type: "famousPersonCandidate", generator: "border_defender" }
{ type: "famousPersonJoin", candidateId: "..." }
```

第一版可以先做“事件给候选名人，玩家点击接纳后加入”。

### 10.2 领土/侦察系统

侦察发现地点后，可以生成：

- 寻访线索
- 战后归附概率
- 特定地点人才倾向

古典时代之后，世界地点不只给资源和分城，也可能给人物。

### 10.3 任务系统

主线任务可以引导玩家：

```text
完成第一次寻访
接纳第一位名人
让名人率队出征
查看第一份战报
```

任务奖励不应该变成抽卡券，只能是资源、线索、声望、寻访机会等世界行为入口。

## 11. 第一版落地范围

建议第一版只做最小闭环：

1. 后端增加名人存档字段与生成器。
2. 增加 3 个来源模板：事件投奔、寻访发现、战后归附。
3. 增加 6 到 10 个基础技能效果模块。
4. 增加技能命名器。
5. 兵营数值迁移到百人制。
6. 出征选择名人和兵力。
7. 自动战斗结算和战报。
8. 前端只做名人列表、寻访入口、出征选择、战报弹窗。

暂不做：

- 名人等级。
- 装备系统。
- 复杂兵种。
- 名人死亡。
- 多名人组队。
- 技能升级。
- 抽卡入口。

## 12. 需要同步调整的旧规则

当前系统里这些规则需要在实现时同步迁移：

| 旧规则 | 新方向 |
| --- | --- |
| 兵营 Lv1 上限 5 | 兵营 Lv1 上限建议 300 |
| 兵营 Lv2 上限 10 | 兵营 Lv2 上限建议 600 |
| 每次训练 1 士兵 | 每次训练一小队，例如 10/20/30 |
| 时代条件士兵 3/6 | 迁移为兵力 300/600 |
| 任务奖励士兵 +3/+6 | 迁移为兵力 +300/+600 |
| 占领固定派 1 士兵 | 迁移为最低 100 兵，并逐步要求名人领队 |
| 地点防御 3/4/5 | 迁移到百人制或抽象战力 |

## 13. 待确认问题

这些问题不阻塞第一版设计，但实现前建议确认：

- 名人是否允许重复姓名，还是必须全服/单用户唯一。
- 寻访是否消耗资源，还是只消耗时间和次数。
- 第一版是否允许没有名人的队伍继续占领旧地点。
- 名人受伤后是不能出征，还是降低战斗表现。
- 战斗过程先做纯战报，还是同时做简化动画。
