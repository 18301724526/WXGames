## Tutorial Intro Anchor Regression, Same-Day Follow-Up

- Public H5 version checked:
  `deployedCommit = ec1f750ca1471a4b02b6afe63e6ad08647a5c1d2`,
  assets rewritten with `deploy-ec1f750ca147`.
- User-reported blocker:
  after resetting a test account, the first tutorial highlight can appear at a wrong position; tapping may flash the highlight to the capital briefly and then return to the wrong position, blocking tutorial progression. Refreshing the page tends to recover.
- Backend data check:
  online `test1 / 123456` reset/login responses contained valid capital data:
  `cityState.capitalCityId = capital`, capital tile such as `tile_24_16`, `siteId = capital`, and world origin such as `{ q: 24, r: 16 }`.
- Frontend anchor check after a fresh page load:
  `worldMapRenderer.getWorldSiteCanvasAnchor('capital', ...)` returned a valid live rect near
  `x = 222.49`, `y = 328.26`, `width = 84.89`, `height = 88.63`.
  `TutorialCanvasRenderer.resolveTutorialIntroTarget()` and the strong tutorial guide resolver matched the same live target.
- Diagnosis:
  the remaining failure was not "cannot get capital anchor" and not backend missing data.
  The broken path was same-page account reset. Calling the reset flow's map camera cleanup hit
  `RangeError: Maximum call stack size exceeded` in `CanvasTerritoryActionHandlers.resetWorldMapCamera()`.
- Root cause:
  `worldMapRenderer` and `worldActorLayerRenderer` form a cyclic renderer graph.
  The reset cleanup recursively followed renderer references without a visited set, so same-page reset could abort before stale map layer context and hit targets were safely cleared.

## Narrow Fix Prepared For Tutorial Reset Regression

- `frontend/js/platform/CanvasTerritoryActionHandlers.js`
  - Replaced recursive renderer cleanup with graph traversal guarded by a `seen` set.
  - Clears transient world-map contexts, cached layer render result, renderer hit targets, and actor-layer hit targets across `worldMapRenderer`, `worldMapLayerRenderer`, and `worldActorLayerRenderer`.
  - Keeps the existing camera-centering behavior unchanged: reset still recenters on the current state's capital and requests a forced render.
- `frontend/js/platform/CanvasTerritoryActionHandlers.test.js`
  - Added a regression test with the same cyclic renderer graph seen online:
    `worldMapRenderer.worldActorLayerRenderer -> worldActorLayerRenderer.worldMapRenderer`.
  - Expected result: account reset camera cleanup succeeds without stack overflow and clears stale contexts/hit targets from both renderers.
