# 文明火种 · 聚落时代策划案 v2.1

> 设计范围：农耕时代（时代1）→ 聚落时代（时代2）的完整闭环
> 文档权威：本文件为时代2设计的唯一权威来源，旧文档中与之冲突的部分以本文为准

---

## 一、时代定位

| 属性 | 值 |
|------|-----|
| 时代编号 | 2 |
| 时代名称 | **聚落时代** |
| 前置时代 | 农耕时代（时代1） |
| 后置时代 | 青铜时代（时代3，预留） |
| 核心新增 | **木材资源** + **工匠职业** + **伐木场建筑** |
| 不引入 | 金属、石材、铜矿（留给时代3） |
| 预计进阶时长 | **3–5 分钟** |

**命名理由**：农耕之后，人们开始聚集成更大的村落，用木材建造栅栏、仓库、伐木场。"聚落"比"青铜"更贴合"只新增木材"这个资源设定——青铜需要金属冶炼，而聚落只需要木头。

## 二、进阶条件

### 时代1 → 时代2

```javascript
cost: { food: 150, knowledge: 15 }
```

**时长推演**（玩家最优分配 2 farmer + 1 scholar）：

| 资源 | 起点 | 目标 | 净产出 | 所需时间 |
|------|------|------|--------|----------|
| food | 100 | 150 | 2 farmer × 1.0 − 3人口 × 0.2 = **1.6/s** | ~31秒 |
| knowledge | 0 | 15 | 1 scholar × 0.05 = **0.05/s** | ~300秒 = **5分钟** |

**瓶颈资源**：knowledge（由 scholar 数量决定）。玩家若分配 2 scholar，knowledge 产出翻倍至 0.1/s，时长压缩到 **2.5分钟**。合理范围 **3–5 分钟**。

## 三、强制引导流程

### 触发机制

当 `gameState.currentEra === 1` 且 `food >= 150` 且 `knowledge >= 15` 时：

1. `GameStateService.calculateEraProgress()` 返回 `canAdvance: true`
2. **前端自动检测条件满足** → 高亮进阶按钮，气泡提示"你的聚落已准备好扩张！"
3. 用户点击进阶 → 后端执行 `advanceEra` → `currentEra` 变为 `2`
4. **后端自动触发**：在 `advanceEra` 响应中，调用 `EventService.generateSpecialEvent(2)` 生成事件

> **强制原则**：条件满足时用户无法跳过引导。进阶按钮外的所有操作被 tutorial 锁定，直到完成全部引导步骤。

### 引导步骤

| 步骤 | 后端状态 | 前端高亮 | 气泡文案 | 用户操作 |
|------|----------|----------|----------|----------|
| 1 | `canAdvance: true` | **进阶时代按钮**（脉冲动画） | "你的聚落已准备好扩张！点击迈入新时代" | 点击进阶 |
| 2 | `currentEra: 2` | 文明标签页 | 时代切换动画 | 观看 |
| 3 | `eventQueue: [森林事件]` | **事件标签**（金色脉冲红点） | "森林里似乎有什么发现..." | 点击事件标签 |
| 4 | 事件弹窗打开 | 事件详情弹窗 | 故事文本 + 选项按钮 | 阅读 |
| 5 | 用户点击选项 | 上报 `claimEvent` | — | 点击"收集木材" |
| 6 | 后端发放 `wood: 20` | 获得资源动画（+20 🪵） | "获得了 20 木材！" | 观看 |
| 7 | `wood: 20` | **建筑标签**（红点） | "用新发现的木材建造伐木场" | 点击建筑标签 |
| 8 | `unlockedBuildings` 含 `lumbermill` | **伐木场卡片** | "伐木场产出木材，工匠负责采集" | 点击建造 |
| 9 | 建造成功 | **人口分配区** | "分配工匠去伐木场工作" | 分配 1 名 craftsman |
| 10 | `craftsmen >= 1` | 引导完成 | "聚落时代开启！继续建设吧" | 自由操作 |

## 四、事件系统机制

### 核心规则

| 规则 | 说明 |
|------|------|
| **非常驻** | 用户完成（点击选项领取奖励）后，事件从队列移除 |
| **挂机存活** | 游戏开着但不操作，事件一直存在；关闭重开，事件仍在 |
| **后端驱动** | 事件生成、奖励计算、状态判断全部在后端；前端只负责渲染 |

### 事件生命周期

```
时代进阶(到2) ──▶ 后端生成事件 ──▶ 写入 eventQueue
                                                    │
                                                    ▼
用户挂机/不操作  ◀──  事件一直存在  ◀──  前端轮询显示
                                                    │
                                                    ▼
用户点击选项  ──▶  上报 claimEvent  ──▶  后端校验+发放奖励
                                                    │
                                                    ▼
后端从 eventQueue 移除事件  ──▶  保存状态  ──▶  完成
```

### 特殊事件属性

| 属性 | 值 |
|------|-----|
| `type` | `"special"` |
| `status` | `"pending"`（未领取）/ `"claimed"`（已领取） |
| `expiresAt` | `null` — **永不自动过期** |
| `options` | 1 个选项（强制引导，无分支） |
| 前端展示 | 金色边框卡片 + 脉冲红点 |

### 事件数据结构（后端定义）

```javascript
{
  id: "evt_settlement_forest_001",
  type: "special",
  status: "pending",
  title: "森林低语",
  description: "聚落的猎人在北边的森林边缘发现了一片从未涉足的林地...",
  icon: "🌲",
  options: [
    {
      id: "opt_collect_wood",
      label: "收集木材",
      reward: { wood: 20 },
      action: "claim"
    }
  ],
  expiresAt: null,
  createdAt: "2026-05-16T10:00:00Z"
}
```

### API 设计

```
GET  /api/game/state
  → 返回 gameState.eventQueue（含事件列表）

POST /api/game/action
  body: { action: "claimEvent", eventId: "evt_xxx", optionId: "opt_xxx" }
  → 后端：校验事件存在 → 发放奖励 → 从 eventQueue 移除 → 保存状态
```

## 五、特殊事件设计："森林低语"

### 触发条件
聚落时代（时代2）首次进入时，后端自动生成并写入 `eventQueue`。

### 故事文本

> 聚落的猎人在北边的森林边缘发现了一片从未涉足的林地。高大的橡树和笔直的松木排列如墙，阳光从树冠缝隙洒下，照亮了满地掉落的干燥枯枝。
>
> 一位老猎人蹲下身，拾起一根树枝在石头上摩擦——火星溅起，燃起了篝火。
>
> "这些木头，"他说，"够我们建十座粮仓。"

### 选项

| 选项 | 效果 |
|------|------|
| **收集木材**（唯一选项） | 获得 **20 wood**，事件从队列移除 |

> 设计意图：强制引导，无选择分支。"唯一选项"降低认知负担，确保所有玩家获得相同初始资源，进入后续建造引导。

## 八、新建筑：伐木场 (lumbermill)

```json
{
  "id": "lumbermill",
  "name": "伐木场",
  "icon": "🪵",
  "category": "production",
  "unlockEra": 2,
  "maxLevel": 3,
  "buildCost": { "food": 50, "wood": 15 },
  "upgradeCosts": [
    { "food": 80, "wood": 30 },
    { "food": 150, "wood": 60 }
  ],
  "effects": {
    "perLevel": {
      "woodOutputBase": 2
    }
  }
}
```

| 属性 | 值 |
|------|-----|
| 建造消耗 | 50 食物 + **15 木材** |
| 升级 1→2 | 80 食物 + 30 木材 |
| 升级 2→3 | 150 食物 + 60 木材 |
| 产出 | 每级提供 **2 木材/秒基础产出**，乘以工匠数量 |
| 示例 | level 1 + 2 craftsmen = 2 × 2 = **4 木材/秒** |

**建造策略**：事件给 20 木材 → 花 15 建造伐木场 → 剩 5 木材（预留）。

> **成本原则**：伐木场**不免费**。除时代0的引导农田免费外，所有建筑均需消耗资源。时代2的木材让玩家首次体验"资源约束下的建造决策"。

## 十二、架构解耦方案

### 设计原则

| 原则 | 说明 |
|------|------|
| **前后端解耦** | 前端只负责显示，后端负责所有逻辑计算 |
| **开闭原则** | 新增时代/事件类型时，不改现有代码，只加配置 |
| **单一职责** | 每个文件只做一件事，代码行数控制在 100 行以内 |
| **职责边界清晰** | domain → 规则判断，calculators → 状态计算，actions → 业务动作，routes → 路由分发 |

### 现有问题诊断

| 文件 | 问题 | 违反原则 |
|------|------|----------|
| `backend/routes/gameRoutes.js` | 150+ 行，混杂路由 + 资源扣除 + 时代进阶 + 人口分配 + 引导校验 | 单一职责 |
| `frontend/app.js` | 300+ 行，状态管理 + 渲染 + API 调用 + 事件绑定 + 引导控制 | 单一职责 |

### 后端解耦

```
backend/
├── routes/
│   └── gameRoutes.js          → 只负责路由分发，所有逻辑委托给 ActionService
│
├── actions/                    → 新增：每个动作一个文件
│   ├── AdvanceEraAction.js   → 时代进阶
│   ├── AssignPopulationAction.js → 人口分配
│   ├── BuildBuildingAction.js  → 建筑建造
│   └── ClaimEventAction.js     → 事件领取
│
├── services/
│   ├── EventService.js         → 新增：事件生成、领取、过期
│   └── GameStateService.js     → 拆分：Normalize + ClientState + OfflineIncome
│
├── domain/                     → 纯数据结构 + 规则
│   ├── Event.js                → 事件数据结构和生命周期
│   └── Population.js           → 人口分配规则校验
│
└── calculators/
    └── EventRewardCalculator.js → 新增：事件奖励计算
```

### 前端解耦

```
frontend/
├── app.js                      → 拆分为 StateManager + Bootstrap
│
├── state/
│   └── GameStateManager.js     → 游戏状态管理
│
├── controllers/
│   ├── TutorialController.js   → 引导逻辑控制
│   ├── BuildingController.js   → 建筑交互控制

│   └── EventUIRenderer.js      → 新增：事件卡片 + 弹窗
│
└── api/

```

### 责任边界

| 职责 | 所属模块 | 禁止行为 |
|------|----------|----------|
| 游戏规则判断 | `domain/` + `validators/` | 不能调用数据库或 HTTP |
| 状态计算 | `calculators/` | 不能修改 gameState |
| 业务动作 | `actions/` | 不能直接操作数据库 |
| 数据持久化 | `services/` + `repository/` | 不能包含游戏规则 |
| HTTP 路由 | `routes/` | 不能包含业务逻辑 |
| 前端渲染 | `renderers/` | 不能修改状态 |
| 前端控制 | `controllers/` | 不能调用 DOM 操作 |

### 开闭原则示例

**新增时代3时，不改现有代码：**

```javascript
// EraConfig.js — 配置驱动
ERA_ADVANCEMENT = {
  2: { nextEra: 3, name: '青铜时代', cost: {...} },  // 新增配置即可
};

// EventService.js — 工厂模式
function generateForEraAdvance(toEra) {
  return this.eventFactory.createForEra(toEra);
  // 新增时代 → 在 EventFactory 注册新事件生成器 → 无需修改 EventService
}
```

**新增事件类型时，不改 EventService：**

```javascript
// 在 EventFactory 注册新类型
EventFactory.register('random', RandomEventGenerator);
EventFactory.register('daily', DailyEventGenerator);
// EventService 本身不需要修改
```

### 单一职责示例：人口分配拆分

**修改前**（gameRoutes.js 内联逻辑）：

```javascript
} else if (action === 'assign') {
  const mapping = { farmer: 'farmers', scholar: 'scholars', craftsman: 'craftsmen' };
  const key = mapping[target];
  if (gameState.population.unassigned < amount) {
    result = { success: false, error: '...' };
  } else {
    gameState.population.unassigned -= amount;
    gameState.population[key] += amount;
  }
}
```

**修改后**（拆分到独立文件）：

```javascript
// backend/domain/Population.js
function reassign(population, target, count) {
  const key = MAPPING[target];
  if (!key) return { error: 'INVALID_TARGET' };
  if (population.unassigned < count) return { error: 'INSUFFICIENT_POPULATION' };
  return {
    population: {
      ...population,
      unassigned: population.unassigned - count,
      [key]: population[key] + count,
    }
  };
}

// backend/actions/AssignPopulationAction.js
function execute(gameState, { target, count }) {
  const result = PopulationDomain.reassign(gameState.population, target, count);
  if (result.error) return { success: false, error: result.error };
  gameState.population = result.population;
  return { success: true };
}
```
