# 回合制战斗场景与技能冷却设计 v0.2

日期：2026-05-29

## 状态

本文是当前战斗系统的最新真相源，用于约束“战斗场景怎么播放、战报事件怎么被前端消费、旧硬判断占领如何退出主流程”。

但以下内容仍未定稿，不能按本文现有描述直接扩展实现：

- `speed` 属性的来源和数值公式。
- 武将/名人的最终属性体系。
- 技能生成、技能效果、冷却时长和数值平衡。

当前代码里的名人属性和技能机制来自早期方案，只能视为兼容底座；后续重做战斗深度时，需要先开新文档把属性、速度和技能规则重新定稿。

## 目标

本版战斗场景不再使用程序绘制的抽象小人作为最终表现方向，而改为资源驱动的战场演出：

- 战场是美术背景图。
- 士兵是透明 sprite sheet 序列帧资源。
- 小兵在场上跑位、接战、攻击、受击。
- 玩家只负责战前搭配，进入战场后自动播放黑盒战斗。
- 战斗内核仍是回合制；行动顺序和技能释放要由后端战报明确给出，前端只负责播放。

## 表现规则

`100兵=1个小兵`只用于画面表现和性能控制，不是战斗单位拆分。

示例：

```text
500 兵 = 5 个小兵表现
501 兵 = 6 个小兵表现
```

战斗结算仍按真实兵力计算，501 兵不会被拆成 6 个独立战斗单位。

## 资源规范

当前落地资源：

| 类型 | 路径 | 用途 |
| --- | --- | --- |
| 战场背景 | `frontend/assets/art/battle/battlefield-forest-camp.png` | 战斗场景底图 |
| 己方士兵帧表 | `frontend/assets/art/battle/soldier-player-sheet.png` | 己方小兵序列帧 |
| 敌方士兵帧表 | `frontend/assets/art/battle/soldier-enemy-sheet.png` | 敌方小兵序列帧 |

士兵帧表规格：

```text
4 行 x 4 列
第 1 行：idle 待机
第 2 行：move 跑动
第 3 行：attack / skill 攻击或技能
第 4 行：hit / defeated 受击或退场
```

后续增加兵种时沿用同一结构，避免渲染器按单个资源写死。

## 战斗内核

### 行动顺序

最终方向是每轮按速度排序：

```text
速度高的一方先行动
速度低的一方后行动
```

若速度相同，当前版本己方先行动。

注意：`speed` 当前还没有设计完成。暂时不要从武勇、统率、兵力、地形或技能里临时推导速度公式，也不要把速度字段写进武将最终属性模型。下一版需要单独确认：

- `speed` 是武将基础属性、战斗派生属性，还是只存在于战斗单位模板。
- 速度是否影响“每轮先后手”，还是进一步影响行动频率。
- 敌方单位的速度从地点模板、兵种模板还是战斗难度生成。
- 同速时是否继续保持己方优先，或改成种子随机/防守方优势。

### 技能释放

当前演出层按“每个单位最多一个主动战斗技能”播放，但技能系统本身仍是早期方案，尚未最终定稿。

规则：

```text
开局双方第一次行动必定释放技能。
技能释放后进入冷却。
之后轮到自己行动时：
  如果技能冷却结束，释放技能；
  否则执行普攻。
冷却结束不会插队，必须等到自己的行动回合才释放。
```

当前冷却以“自身行动次数”为单位递减，而不是以真实时间递减。这样可以保持纯回合制语义，前端动画只是战报事件的演出。

注意：这只是当前 `speed-skill-cooldown-v1` 战报的播放规则，不等同于最终技能系统。技能来源、技能标签、主动/被动边界、冷却数值、开局必放是否保留，都需要在武将属性与技能设计重做时一并确认。

### 战报事件

后端输出 `speed-skill-cooldown-v1` 战报。

关键字段：

```js
{
  system: "speed-skill-cooldown-v1",
  skillRules: {
    openingSkill: true,
    cooldownTicksOnOwnTurnOnly: true,
    fallbackAction: "basicAttack"
  },
  turns: [
    {
      actor: "attacker",
      target: "defender",
      action: "skill" | "basicAttack",
      skillName: "...",
      cooldownBefore: 0,
      cooldownAfter: 3,
      damage: 42,
      attackerSoldiersAfter: 474,
      defenderSoldiersAfter: 457
    }
  ]
}
```

前端只消费这些事件播放动画，不参与战斗计算。

### 旧占领结算边界

旧版“士兵数量大于地点防御即胜利”的硬判断只能作为历史兼容或 fallback，不能再作为有主地点战斗的主流程。后续有主地点占领应统一输出战报事件，由战斗场景播放；前端不能根据地点防御自行推断胜负。

## 前端演出

每个行动事件拆成两个视觉阶段：

```text
move：攻击方小兵向接战区跑动
impact：攻击方播放攻击/技能帧，目标方播放受击帧
```

技能事件会额外显示一次技能冲击特效和技能名。普攻只显示普通接战特效。

## 回退边界

本版保留了进入战斗场景、跳过、关闭、战报回放等通用框架。若需要回退表现层，可只替换 Canvas renderer 的战斗绘制实现；后端 `speed-skill-cooldown-v1` 战报结构应继续保留，因为它描述的是正确的战斗规则。
