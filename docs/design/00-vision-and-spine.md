# 系统设计：PVPVE + 三国志深度 —— 愿景与单一事实源脊柱

> 本目录（`docs/design/`）是对五大系统的完整设计蓝图：科技树重设计、外交、AI 势力+AI 城市、人物关系网+性格。北极星是**三国志系列的深度**，玩法框架是 **PVPVE**。强约束：**单一事实源，任何为效率牺牲底层实现的做法都不允许**。
>
> 状态：设计进行中（自主推进）。用户会在起床时叫停，届时整理进度接力到公司 codex 继续。每份文档末尾有「**待你确认的设计问题**」。

## 0. 为什么需要这些系统

现在的世界是**静态**的：中立空城 + 不动的野怪营地 + 一个只探索不占地的 AI 探路者。没有活的对手、没有人物之间的关系、没有外交、科技只解锁建筑。要成为 PVPVE + 三国志深度的游戏，缺的是**活的世界**：

- **活的对手**（PVPVE 的 E 和 P 之外的第三方）：AI 势力会发展城池、研究科技、征兵、扩张、开战、结盟。
- **活的人**：武将有性格，会在世界里认识别人，关系会随事件迭代；触发好友来投、背叛、投敌。
- **活的政治**：势力之间有好感度、有敌对/友好/中立/同盟/仇视，会因你的行为和彼此的性格而变化。
- **有深度的成长**：科技不只是解锁建筑，而是文明式的多路线、多种效果（兵种/加成/能力/政策/外交战斗修正）。

## 1. 三国志（光荣 RTK）北极星映射

游戏**已经具备**三国志的核心底子——武将六维属性直接对应 RTK 五维 + 速度：

| 本作字段 | 三国志对应 | 已存在 |
|---|---|---|
| `command` 统率 | 统率 | ✅ |
| `force` 武力 | 武力 | ✅ |
| `intelligence` 智力 | 智力 | ✅ |
| `politics` 政治 | 政治 | ✅ |
| `charisma` 魅力 | 魅力 | ✅ |
| `speed` 速度 | （机动） | ✅ |
| `quality` 品质 | （稀有度） | ✅ |
| `skills`/`abilityKit` | 特技/战法 | ✅ |
| `status.loyalty` 忠诚 | 忠诚度 | ✅（仅对玩家） |

**要新增的三国志要素**（本设计的核心）：

- **性格（性格/気性）**：每个武将一个性格，驱动行为（喜欢出游→到处认识人、刚直→易结仇、义气→重羁绊）。
- **相性（相性）**：一个隐藏兼容值，影响两人是否投缘、招降成功率、羁绊生成。
- **羁绊/关系（义兄弟・夫婦・主従・宿敌）**：人物间的有向关系 + 强度 + 历史。
- **在野武将（在野武将）**：不隶属任何势力、在世界上游历的武将，可被各势力登用。
- **势力（勢力）**：以君主为核心的一等实体，拥有城池、科技、外交、战略 AI。
- **外交（外交）**：势力间好感度 + 关系状态 + 同盟/宣战/停战。
- **登用/来投（登用）**：招募机制，受相性、好感、君主魅力、关系网影响——**这正是 ②b 捕获面板里"招降"要接的加成系统**。

## 2. 单一事实源脊柱：两个核心世界实体

所有五个系统都挂在**两个核心实体**上。把这两个实体的事实源钉死，其余系统都是它们之上的关系与规则层——这是"改一处不用改多处"的关键。

### 脊柱 A：势力（Faction）
一等实体，**玩家和 AI 用同一个模型**（关键：玩家只是 `factionId==='player'` 的一个势力，不是特例）。一个势力聚合：君主、城池、领土、科技、资源、外交立场、战略 AI 状态。

- 现状：`gameState.polity` 只是玩家身份（name/color/capitalCityName），**不是**一等势力实体，也没有其它势力。→ 需升级为势力实体 + 势力注册表。
- 依赖它的系统：外交（势力↔势力）、AI 城市（势力拥有城）、科技（每势力独立研究）、关系网（人属于势力）。
- **事实源位置**：取决于"共享世界 vs 每玩家世界"架构（正在勘察，见 §4）。无论哪种，势力都是一个**带 `factionId` 的实体注册表**，玩家势力和 AI 势力同构。

### 脊柱 B：人物（Person）
已存在（`gameState.famousPeople[]` + 武将六维/技能/忠诚）。要扩展为世界级的人物，带**性格**与**关系**：

- 现状：只有玩家自己的花名册（roster），没有世界人物注册表，没有性格/关系字段。
- 要加：`personality`（性格）、`compatibility`（相性种子）、`relationships`（有向 pairwise 关系，稀疏）、`factionId`（隶属势力，或"在野"）。
- 依赖它的系统：关系网（人↔人）、登用/来投（关系+相性驱动）、AI 势力（用人物做君主/武将）、②b 捕获招降（相性+关系加成）。
- **事实源位置**：人物是世界级事实（在野武将、AI 势力武将都要存），需要一个**世界人物注册表**，不能只在玩家 roster 里。

> **单一事实源铁律**：势力和人物各自**只有一个权威副本**；城池的归属是"城上的 `factionId`"（不另存一份势力→城列表，用查询派生）；关系是"存在人物身上的有向边"（不另存全局关系矩阵，稀疏边即事实）；外交好感度是"势力对上的有向条目"。任何"为了快"而复制一份的做法，都要用**查询/投影**替代。

## 3. 系统依赖图（建造顺序）

```
        ┌─────────────────────────────────────────┐
        │  脊柱 A: 势力实体 (Faction)  ← 先建，最底层 │
        │  脊柱 B: 世界人物注册表 + 性格 (Person)     │
        └───────────────┬─────────────────────────┘
            ┌───────────┼───────────┬──────────────┐
            ▼           ▼           ▼              ▼
     [外交/好感度]  [关系网/羁绊]  [每势力科技]   [AI 城市]
     势力↔势力      人↔人+迭代      Civ 式重设计   势力拥有城
            └───────────┴─────┬─────┴──────────────┘
                              ▼
                    [AI 势力决策循环]  ← 顶层，消费以上全部
                    扩张/建造/研究/征兵/开战/外交/登用
```

**建造顺序**（每一层都是前一层的单一事实源之上）：
1. **势力实体 + 世界人物注册表**（脊柱，先建，其余都依赖）。
2. **性格 + 相性**（人物属性扩展）。
3. **外交好感度**（势力↔势力关系）与**关系网**（人↔人关系）——同构的"有向关系 + 状态 + 迭代"，可共用一套关系基元设计。
4. **每势力科技** + **科技树 Civ 式重设计**（效果类型扩展）。
5. **AI 城市**（势力拥有会发展的城）。
6. **AI 势力决策循环**（顶层，消费全部）。

## 4. 架构定论：共享世界 PVPVE（决定了势力/AI 的事实源位置）

**已勘察定论（有代码证据）：这是共享世界 PVPVE。** 一个规范世界（`WorldMapConstants.DEFAULT_WORLD_SEED` 单种子 + canonical tile id），**多个真人玩家 + AI 势力共处其中**。

关键机制（`backend/repositories/`）：
- **每玩家状态**：`game_states` 表，`playerId` 主键（`GameStateRepository.js`）。一个 playerId 一份 gameState。
- **共享领土注册表**：`shared_world_territories` 表（`ownerPlayerId` + territory JSON）= **所有玩家已占领土的单一事实源**。
- **跨玩家可见**：`getClientProjectionForPlayer(playerId)` → `sharedWorldTerritories`（`excludePlayerId`），只读合并进该玩家的领土 DTO；**投影字段在存盘前被 strip**（canonical isolation，不污染玩家自己的 state）。
- **出生防撞**：`player_spawn_allocations` 表全局唯一约束，玩家不会撞点。
- **世界 tick**：`WorldWorkerService`（~5s）推进每个近期活跃玩家。

**对单一事实源的决定性结论（所有系统据此设计）：**
1. **真人玩家就是一个势力**（他的 `playerId`）。玩家和真人对手天然同构。
2. **AI 势力是额外的势力，必须活在共享世界状态**——新建 `ai_faction_state` 表/仓库，**与 `shared_world_territories` 平行**，**不放进任何玩家的 gameState**。
3. **AI 拥有的城/领土复用现有 `shared_world_territories` 注册表**，`ownerPlayerId = 'ai_faction_<id>'`（单一事实源，不另造一套城模型）。
4. **外交** + **世界人物注册表（在野武将/全部人物）** 同样是**共享世界状态**，每个玩家请求时经投影读取。
5. **AI 势力决策 + 关系网迭代的模拟只在 `WorldWorkerService` tick 里跑一次**（共享），绝不每玩家各跑一份。

> 由此，脊柱 A（势力）的事实源 = **共享 `ai_faction_state`（AI 势力）+ `game_states.playerId`（真人势力）+ `shared_world_territories`（城/领土归属）**；脊柱 B（人物）的世界注册表 = **新的共享人物表**。每个事实一份、经投影下发。这条架构线贯穿 `01`~`06` 全部文档。

## 6. 设计与开发纪律（贯穿全部）

1. **单一事实源**：每个事实只有一个权威副本；派生数据用查询/投影，不复制。
2. **玩家=势力**：玩家和 AI 走同一势力模型，不为玩家开特例分支。
3. **每阶段可测**：纯规则核（关系/外交/相性/科技效果）做成 pure 模块 + 单测；服务层接线；对抗性 review 后再部署。
4. **配置表驱动**：数值（性格类型、相性表、外交阈值、科技节点/效果、AI 权重）走 Excel 配置表（`config/tables/`），不硬编码。
5. **世界 tick 挂钩**：关系迭代 + AI 决策挂到现有 `advanceRuntimeState` / `WorldWorkerService` tick，不新起失控的调度。
6. **不偷懒**：宁可慢，也要底层实现正确；发现要改多处=事实源没钉好，回去重构。
