# 前端 UI 状态层与开发路线图文档检查报告

> 检查日期：2026-05-20  
> 检查对象：
> - `docs/前端UI状态层迁移方案_v0.2.md`
> - `docs/文明火种_开发路线图_v2.3.md`
> - 当前前端/后端代码与测试文件

## 一、总体结论

两份文档的技术主线基本一致：当前前端正在从直接 DOM 操作逐步迁移到 `UIStatePresenter` 生成纯 ViewState，再由 H5 DOM Renderer、Canvas Renderer 或小游戏入口消费同一份 UI 状态。

当前代码也基本符合这条路线：

- `frontend/js/state/UIStatePresenter.js` 已集中承载资源、人口、建筑、事件、军事、侦察、世界雷达、弹窗、导航、日志等 ViewState。
- H5 入口已经通过 `H5ShellAdapter`、各 UI Adapter 和 `H5CanvasAppShell` 收敛 DOM 依赖。
- `frontend/js/platform/` 下已经出现共享 Canvas 渲染基类、H5 Canvas 壳层和小游戏平台入口相关实现。

不过，两份文档已经出现明显的版本漂移和局部状态过期。建议先做一次文档收口，否则后续 AI 助手或接手开发者容易误判“当前版本、已完成范围、下一步优先级”。

## 二、发现的问题

### 1. 文档文件名、标题和元信息版本不一致

严重程度：中

现象：

- 原问题：`docs/前端UI状态层迁移方案_v0.1.md` 文件名是 `v0.1`，但正文标题是 `前端 UI 状态层迁移方案 v0.2`。
- 原问题：`docs/文明火种_开发路线图_v2.2.md` 文件名是 `v2.2`，但正文标题是 `v2.3`，元信息仍写 `版本：v2.2`。
- 当前处理：两份文档已经分别对齐为 `docs/前端UI状态层迁移方案_v0.2.md` 与 `docs/文明火种_开发路线图_v2.3.md`。

影响：

- 后续引用文档时无法判断哪个版本才是准版本。
- 远端仓库、部署记录和版本测试已经推进到 `0.1.13`，文档版本标识却停留在较早状态。

建议：

- 已采用推荐方案：升级文件名并同步正文版本信息。

### 2. 路线图“当前系统状态”日期过旧

严重程度：中

现象：

- 路线图第二章写的是“当前系统状态（截至 2026-05-18）”。
- 实际文档和代码已经推进到 2026-05-20。
- 当前已包含 Canvas HUD、H5 Canvas shell、小游戏平台入口和多个依赖注入收口项。

影响：

- AI 助手会倾向把 2026-05-18 当作真实当前状态。
- 后续开发可能重复检查或重复实现已经完成的迁移项。

建议：

- 更新为“当前系统状态（截至 2026-05-20）”。
- 增加一小节“0.1.10 ~ 0.1.13 已完成内容”，把 Canvas HUD、设置菜单、日志面板、小游戏入口、依赖注入收口统一列进去。

### 3. 项目结构描述缺少当前关键目录

严重程度：中

路线图中的结构描述仍停留在早期：

```plain
frontend/js/
  api/
  config/
  controllers/
  services/
  state/
  ui/
```

当前实际还需要补充：

```plain
frontend/js/platform/
  CanvasGameRenderer.js
  H5CanvasGameRenderer.js
  H5CanvasRuntime.js
  H5CanvasAppShell.js
  MiniGameCanvasRenderer.js
  MiniGameApp.js
  PlatformRuntime.js

frontend/minigame/
  game.js
```

影响：

- 新接手者会忽略 Canvas 和小游戏迁移主线。
- 路线图第 6 章已经提到 Canvas HUD，但项目结构没有对应入口，前后不完整。

建议：

- 在路线图 1.2 代码结构中补齐 `platform/` 和 `minigame/`。
- 在关键文件清单中把 `Canvas HUD` 从 `app.js (initCanvasHUD)` 改为 `app.js -> H5CanvasAppShell -> CanvasGameRenderer`。

### 4. 资源栏 DOM/CSS 删除记录已收口

严重程度：已处理

此前检查发现 `frontend/style.css` 仍残留 `.resource-panel`、`.resource-card`、`.resource-value` 等资源 DOM 样式。当前已完成清理：

- `frontend/index.html` 不再包含 `resourcePanel`、`resourceDetailModal` 和资源详情关闭按钮。
- `frontend/style.css` 不再包含资源 DOM 面板、卡片、图标和值文本选择器。
- `ResourceRenderer.js`、`ResourceDetailModalAdapter.js` 及对应测试文件已删除。
- 资源条和资源详情面板均由 `CanvasGameRenderer` 绘制，H5 与小游戏入口通过 Canvas 命中目标打开/关闭。

验收口径：

- 使用 `resource-art.test.js`、`h5-canvas-runtime.test.js`、`shared-canvas-renderer.test.js` 和 `minigame-platform.test.js` 锁定资源栏不再回退到 DOM 路径。

### 5. 测试说明过期

严重程度：中

迁移方案中写“55 个测试全部通过”，但当前测试数量已经变化。

实际抽查版本测试：

```plain
node --test frontend/tests/stage5-version.test.js frontend/tests/version-number.test.js
```

结果：

- `stage5-version.test.js` 通过，确认当前版本为 `0.1.13`。
- `version-number.test.js` 失败，因为该旧测试仍期待 `0.1.1`。

影响：

- 文档中的“测试全部通过”不能直接作为当前验收依据。
- 旧测试文件会误导后续 CI 或人工检查。

建议：

- 删除或重命名旧的 `frontend/tests/version-number.test.js`，或改成不再硬编码 `0.1.1`。
- 文档中的测试说明改为“以当前 `backend/package.json` 版本和 `stage5-version.test.js` 为准”。

### 6. 路线图中的平台定位需要补充说明

严重程度：低到中

路线图 1.1 写：

```plain
平台：H5 网页版（非微信小游戏 Canvas SDK）
```

这个描述对早期版本是准确的，但现在仓库已经有小游戏入口和 `PlatformRuntime`。目前更准确的说法应该是：

```plain
当前正式运行平台仍是 H5 网页版；
代码层已经开始为微信/抖音小游戏 Canvas 入口做兼容迁移。
```

影响：

- 若只看路线图，会以为小游戏方向还没有进入代码实现。
- 与 `frontend/minigame/game.js` 和 `frontend/js/platform/*` 现状不一致。

建议：

- 保留“H5 是当前正式部署平台”。
- 增加“小游戏 Canvas 入口为迁移预研/渐进落地中”的状态说明。

## 三、建议的文档收口顺序

1. 统一版本标识
   - `前端UI状态层迁移方案_v0.2.md`
   - `文明火种_开发路线图_v2.3.md`

2. 更新路线图当前状态
   - 日期改到 2026-05-20。
   - 当前版本补到 `0.1.13`。
   - 增加 Canvas HUD 和小游戏入口状态。

3. 补齐项目结构和关键文件清单
   - 增加 `frontend/js/platform/`。
   - 增加 `frontend/minigame/`。
   - Canvas HUD 文件链路改成 `app.js -> H5CanvasAppShell -> CanvasGameRenderer`。

4. 校准 Canvas HUD 迁移记录
   - 明确哪些 DOM 已删除。
   - 明确哪些 CSS 还有遗留。
   - 把“底部 Tab 栏 DOM 删除”状态写清楚：当前 Canvas 已绘制并接管入口，但 H5 DOM tab 仍存在作为页面结构/兼容层。

5. 清理或更新旧测试描述
   - 处理 `frontend/tests/version-number.test.js` 对 `0.1.1` 的旧断言。
   - 文档不要写固定“55 个测试”，改写为“当前测试以 `node --test frontend/tests/*.test.js` 和 `backend npm test` 为准”。

## 四、建议的下一步开发重点

结合两份文档和当前代码，下一阶段建议继续沿着 Canvas HUD 迁移做，而不是回头大改业务逻辑：

1. 顾问弹窗迁移到 Canvas 面板。
2. 城市切换菜单迁移到 Canvas 下拉列表。
3. 继续按模块删除已迁移功能的 DOM 方法，避免 H5 人工测试出现双路径歧义。
4. 事件弹窗迁移到 Canvas 面板。
5. 再考虑中间页面内容逐步 Canvas 化。

这与当前路线图附录 B 的待迁移列表一致，也和 `UIStatePresenter` 已经沉淀出的 ViewState 能力匹配。

## 六、结论

文档不是方向错，而是需要一次“版本和事实状态收口”。

建议优先修正文档版本、当前状态日期、项目结构和 Canvas HUD 迁移记录。资源栏与资源详情面板已经完成 Canvas 接管和 DOM 删除，后续继续做顾问弹窗、城市切换菜单和事件弹窗的 Canvas 接管。
