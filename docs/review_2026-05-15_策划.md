## 一、严重问题（需立即修复）

### 1.1 前后端科技定义完全脱节 ⭐⭐⭐

| 维度 | 前端 | 后端 |
|------|------|------|
| 科技ID | `fireMaking`, `writing`, `agriculture`, `animalHusbandry`, `metallurgy`, `calligraphy`, `potteryWheel`, `irrigation`, `bronze`, `iron`, `geometry`, `philosophy` | `钻木取火`, `文字`, `农业`, `畜牧`, `冶金`, `书写法`, `陶轮`, `灌溉`, `青铜器`, `铁器`, `几何`, `哲学` |
| 科技名映射 | `techNameMap` 做反向映射（后端中文名→前端ID） | 直接用中文名作为键 |
| 科技数量 | 12个 | 12个（但key完全不同） |

**问题**：前端 `canAdvanceEra()` 检查 `requiredTechs: ['fireMaking', 'writing']`，但后端 `calculateEraProgress()` 检查 `requiredTechs: ['钻木取火', '文字']`。虽然 `syncFromServer()` 做了反向映射把后端状态转回前端 `techState`，但**前端本地时代进阶检查**（`canAdvanceEra`、`calculateEraProgress`）仍然用前端ID做判断，而服务器返回的 `eraProgress` 虽然正确，**前端却不直接使用后端返回的 `canAdvance`**——前端 `advanceEra()` 里调用后端 API，但 `renderUI()` 里 `canAdvanceEra()` 仍是前端本地计算。

**更隐蔽的问题**：后端 `TECHNOLOGIES` 中 `畜牧`（era 1）解锁条件需 `农业`（era 0）完成，但前端 `animalHusbandry` 的 `era: 1`、`prereq: 'agriculture'`——如果后端和前端对 era 的理解不一致，可能导致某些科技在前端显示"可研发"但后端拒绝。

### 1.2 前后端事件系统双轨并行，数据定义完全不同 ⭐⭐⭐

前端 `app.js` 的 `events` 对象：
- 6个事件：`harvest`, `famine`, `beastAttack`, `ruins`, `visitors`, `celestial`
- 选项结构：`optionA`/`optionB`，`effects` 为固定数值（`food: 200`, `knowledge: 150`）
- 前端自己渲染事件面板、自己处理选择逻辑

后端 `server.js` 的 `EVENTS` 对象：
- 13个事件（6通用 + 7时代专属）
- 选项结构：`options.A`/`options.B`，使用百分比+保底双轨制（`foodPercent: 0.20`, `foodMin: 30`）
- 有 `weight`, `condition`, `era`, `expiresAt`
- 有 `negativeStreak` 保护、首事件100%触发、队列上限、过期自动执行惩罚

**问题**：前端 `chooseEventOption` 调用后端 API，但前端事件展示（`renderPendingEvents`）用的是**前端自己的 `events` 定义**。如果后端触发了一个前端没有定义的事件（比如 `fireUnstable`），前端 `renderPendingEvents` 里 `const evt = this.events[eventId]` 会返回 `undefined`，导致渲染空白或崩溃。

更严重：`syncFromServer` 同步了 `eventQueue`，但 `pendingEvents` 只是字符串ID列表，前端渲染时依赖 `this.events[eventId]`——**后端新增了7个时代专属事件，前端完全不认识**。

### 1.4 开局状态仍有多处偏离 v2.0 设计 ⭐⭐

- `resources.wood: 100`, `stone: 100`, `metal: 100` → **应为 0**（MVP开局无这些资源）
- `buildings.farm: 0` → **应为 1**（影响食物产出净增长）

### 1.5 人口上限公式前后端不一致 ⭐⭐

后端 `sanitizeGameState`：
```js
const baseMax = (gameState.buildings.house || 0) * 4;
const techBonus = (gameState.techEffects?.houseCapacity || 0);
const maxPop = baseMax + techBonus;
```

前端 `syncFromServer` 取 `maxPop: serverPop.maxPop ?? serverPop.max ?? 5`。

## 三、数值设计问题

### 3.1 食物消耗与饥饿机制

后端 `tickPlayer`：
```js
const foodConsumption = (gameState.population.total || 0) * 0.2 * deltaSeconds;
gameState.resources.food -= foodConsumption;
```

前端 `tick` 注释写"前端不计算资源，由服务端驱动"，但 `calculateMultipliers` 里：
```js
const happiness = 100 + s.buildings.house * 5 + s.buildings.temple * 15;
s.happiness = happiness;
```

前端自己重算幸福度，但幸福度是后端算的。**如果后端 `getDefaultGameState` 初始 `happiness: 80`，前端 sync 后会显示 80，但前端 tick 会把它改成 100+...**。不过前端不保存，5秒后 heartbeat 又会被后端覆盖。闪烁问题。

### 3.2 知识产出"研发期间归零"

后端 `calculateResourceOutput`：
```js
const knowledgeOutput = scholarKnowledgeOutput + popBaseKnowledgeOutput;
```

知识产出**始终**计算，不分研发与否。但后端 `updateResearchProgress`：
```js
researching.progress += output.knowledge * deltaSeconds;
```

所以后端逻辑是：知识产出同时用于**库存增长**和**研发进度**。这会导致知识"双花"——1点知识既增加库存又推进研发。

前端 `tick`：
```js
let knowledgeGain = 0;
if (!s.activeResearch) {
    knowledgeGain = knowledgePerSec * multipliers.global;
}
```

前端理解是：研发期间知识**不增加库存**。但后端实际做的是**两者同时增加**。设计意图需明确：
- 方案A（前端理解）：研发期间知识全部投入研发，库存不增
- 方案B（后端实现）：知识同时增加库存和推进研发（相当于白给）
- 方案C：知识 50% 研发、50% 库存（折中）

### 3.3 离线收益只算了食物和知识

后端 `calculateOfflineIncome`：
```js
return { food, knowledge, wood, offlineHours }
```

前端 `showOfflineModal` 只展示 `food` 和 `knowledge`。如果离线期间产出了 wood，玩家看不到。

## 四、待确认问题（需 Coordinator/策划决策）

1. **科技定义统一**：前后端科技 ID/名称/前置关系需要完全统一。建议统一用中文名（后端），前端 `techNameMap` 只保留映射不做逻辑判断。

2. **事件系统统一**：后端事件系统更丰富（13个 vs 6个），建议前端删除本地 `events` 定义，改为从后端获取事件模板，或至少把后端所有事件同步到前端。

3. **建筑配置加载**：前端是否应该 `fetch('shared/buildingConfig.json')` 加载配置，而不是硬编码？

4. **知识"双花"问题**：后端同时把知识加到库存和研发，这是设计意图还是 bug？

5. **食物消耗惩罚**：v2.0 设计了饥饿惩罚（产出×0.5、每60秒-1人），但后端 `tickPlayer` 里只有 `food -= total*0.2`，没有饥饿状态判定。

6. **人口上限公式**：后端 `baseMax = house * 4`，但前端硬编码 `maxPop: 5`，后端默认 `population.max: 5`，`house: 1`——`1*4=4≠5`。统一为 `house*4+1` 还是 `house*4`？
