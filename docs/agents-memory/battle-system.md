Building a new entity-based battle (每个士兵独立实体, 主将带兵, 竖版左右对冲, 独立战斗场景) to replace the placeholder 1v1 formula combat. Perf validated (~6000 individual sprites @60fps on an iPhone; rendering is cheap, targeting was the cost — solved by target stickiness).

**Architecture (deterministic core shared front+back):**
- `shared/battleSimCore.js` (UMD): authoritative, deterministic, headless sim. createBattle/step (frontend per-frame) + simulate (backend run-to-end). Movement = steering blend (speed jitter + formation cohesion + separation); orders via `ORDER_BEHAVIORS` registry; skills via `EFFECT_REGISTRY` (data-driven, rage/cooldown, auto/manual via `skill` input-stream entries); morale (general death → ½ damage + seeded rout). Default order = 待命 (no auto-clash). Extension fields: `unitType`(兵种)/`action`(招牌动作槽)/`slotOff`.
- `shared/battleAI.js` (UMD): behavior-tree AI deciding squad orders (hp-low→retreat / enemy-near→soldierAttack / else→advance). Pure, deterministic, reusable backend+frontend.
- `backend/services/battle/BattleSimService.js`: attribute→stat conversion (PLACEHOLDER, to be balanced/config-driven), buildSetup/resolve, `applyCasualtiesToFormationSnapshot`, and `resolveBattle` (full loop: snapshot+attributes→battle→survivors written back).
- `frontend/tools/battle-scene-lab.html` (at https://kodagame.top/tools/battle-scene-lab.html): playable demo — real core, real battle sprites, 军令 UI, skills, AI enemy, result panel. This is what's tested on phone.

**Authority/anti-cheat model:** frontend renders the show; backend re-runs the deterministic core with true stats + backend-issued seed + the recorded input stream → authoritative result. So changing attack power client-side does nothing. Tune `cohesion`/`speedJitter`/`separation*` in core DEFAULTS for the feel dial.

**Sprites are temporary** (only idle/move/attack/die, single soldier set keyed by side); real 兵种/状态/招牌动作 plug in via the registries — never hardcode lists. See [[deploy-urls]].

**DECISION (user, June 23): the current "auto-resolve + passive replay" is the WRONG model — it's a non-interactive cutscene. Rebuild as interactive + decoupled.** The dev is firm: battle must NOT be coupled to march/explore (it currently piggybacks explore missions: 10s steps, mission.combat, resolve-on-arrival). Battle should be its own domain exchanging only **squad status (待命/行军中/撤退中/战斗中) + setup/result** with the world layer; 军令/skills are battle INPUTS (inputStream), never touched by the march layer. Plan + phases in [docs/world-combat-interactive-architecture.md](../../docs/world-combat-interactive-architecture.md). Target flow: attack → backend opens a battle SESSION (issues seed + setup, squad→战斗中, NOT resolved) → client plays interactively (军令/skills/rage, lab-style entity scene) recording the inputStream → submit → backend re-sims `BattleSimService.resolveBattle(setup, inputStream)` authoritatively → applies result. `battleSimCore`/`BattleSimService` are already the pure black box this needs. Phases: (1) backend session service + start/resolve actions, (2) frontend interactive scene (port lab UI) replacing the passive overlay, (3) make squads first-class (decouple from explore mission), (4) data-driven skills/兵种. Keep the auto-resolve path only as an unattended fallback.

**Battle camera (zoom/pan) SHIPPED (codex-battle, commit d5a1772b):** dev feedback — units looked tiny/blurry because the **fixed 9:16 portrait arena (720×1280)** is letterboxed into the short stage (panel eats ~37% height) → scene scale ~0.31 → ~10px soldiers. Decision: **DO NOT change the aspect / "fill width"** (it's a mobile portrait game; sprites must stay proportional to the scene or they'd be giants/ants across devices). Instead added a **camera** (pinch/wheel zoom + drag pan; default zoom 1.6, range [1,5]). Architecture mirrors the existing `WorldMapRuntimeCameraPolicy` split — clean layering, no coupling: **`frontend/js/domain/BattleCameraPolicy.js`** = pure no-DOM math (computeFit/getViewTransform/zoomAt/panBy/clampCamera), unit-tested; `BattleCanvasRenderer` only reads the transform + clips to the stage (stashes `entityBattle._viewFit` for the input layer); `CanvasGameAppBattleScene` holds `entityBattle.camera` state + `entityBattleZoom/entityBattleDrag` controllers; both input routers translate `wheelZoom`/`pinchZoom`/`drag` → `actionController.handle({type:'entityBattleZoom'|'entityBattleDrag'})` (same pattern as techTreeZoom/worldMapDrag) and add `entityBattle` to `hasBlockingOverlayOpen`. The overlap geometry (foot-anchored sprites + y-sort painter draw) was already correct — it only failed to read because the scene was too small; zoom fixes that. Note: sprite size = `arenaBase × fitScale × cameraZoom × SPRITE_SCALE(2)`, fully proportional/device-consistent. Still needs in-browser playtest.
