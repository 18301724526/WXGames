owner 复盘(2026-07-13,教程解耦十层挂死夜之后):教程解耦选了"最安全但最麻烦"的路(半迁移+双引擎并存+逐步等价验证),事后看**暴力解耦可能更优**——①游戏逻辑清零该功能→②独立写引擎(对干净接口)→③一次接入。双引擎并存窗口是十层 livelock 病的温床;等价性仪式(逐字节基线 diff)对"引导 UX"这种非圣物内容过重。

**Why**:生产级纪律(main 每 commit 可发布+行为等价)用在 dev 阶段产品上成本倒挂;dev 测试服能接受功能下线数周。

**How to apply**:未来解耦立项(如 [[tutorial-coupling-step6]] 余簇)先问分水岭三题:①该功能在测试服停摆数周可接受吗?②其精确行为是圣物吗(游戏规则/存档=是;UX 流程=否)?③耦合面宽吗?三者答"可停摆+非圣物+宽耦合"→**暴力解耦优先**(清零段自身是易验收中间态);答"不可停摆或行为圣物"→等价迁移。注意:清零段的工作量(发放防作弊/规则改键/存档迁移)任何路线都逃不掉,暴力法省的是**并存窗口成本与等价仪式成本**,不是沼泽本身;接入日会一次性撞上底层架构病(查询 owner/镜像类),预算要留。关联 [[refactor-no-debt-for-safety]](生产级纪律条款,适用范围应限"不可停摆或圣物行为"的解耦)。
