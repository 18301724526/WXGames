As of 2026-06-29 (Codex→Claude handoff; resets the [[architecture-refactor]] client direction) the frontend ECS work is **REAL BitECS, NOT a self-rolled architecture wearing an ECS shell**. The user is emphatic and repeated it: do real ECS, **not 套个壳的自研架构**. A green test or a tidy folder name is NOT acceptance.

**Only-real-ECS rule.** `bitecs@0.4.0` is the ONLY approved ECS core, touched ONLY through the single seam `frontend/js/ecs/core/EcsCoreBoundary.js` (or the generated `frontend/js/ecs/runtime/EcsModeRuntimeBundle.js` from `npm run build:ecs-runtime`). A module "counts as ECS" ONLY when: entity id from `addEntity(world)`; authoritative state in `defineComponent(...)` TypedArray columns (`Component.field[entity]`); reads via `defineQuery`; mutations inside NAMED systems that write the component arrays; renderer/platform/input/sync get **read-only** projections/snapshots, never writable component ownership; tests assert the component storage + system/query behavior. If a test still passes against a POJO owner, it is NOT ECS.

**REJECTED even if tests pass — this is the 套壳 the user forbids:** owner / snapshot / projection / model / resolver / facade / **bridge / adapter / wrapper** objects standing in for component storage; class instance / POJO / host mirror / shell mirror / `globalThis` / renderer cache as authoritative state; App+Shell+CanvasShell+Runtime+Renderer+StateManager all writing one fact; **a bridge that forwards writes both ways**; renderer writing authoritative state; a bundle that hides self-written ECS storage; and **just moving a file into `ecs/` and calling it migrated**. Self-rolled ECS cores (EcsCore/EntityStore/QueryEngine/etc.) and non-bitecs ECS packages (ecsy/miniplex/koota/…) are guard-banned.

**The ONE accepted template = `frontend/js/ecs/foundation/WorldClock.js` (+ .test.js).** Copy its shape: one `defineComponent` whose arrays ARE the truth; `createWorld`+`addEntity`+`addComponent` → a plain handle with NO methods; `defineQuery` to resolve entities; named systems (`runClockSyncSystem`/`runClockAdvanceSystem`) that ONLY mutate `Component.field[entity]` and return booleans; a frozen value-copy projection (`getClockSnapshot`); `getShared` returns the real world handle (cached on a global), not a wrapper. Tests assert `Clock.<field>[entity]` directly AND that the handle has no `getEpochNowMs`/`updateFromPayload` methods (actively rejecting the wrapper anti-pattern).

**Per-slice migration method:** name the old source of truth + every writer → define the BitECS components → write FAILING tests that inspect component arrays/queries → systems that mutate only components → add a projection for renderer/platform compat → rewire old public APIs to call BitECS directly → delete/demote the old owner to read-only → add a guard banning new owner/mirror writes for that fact. **No batch may end with two writable sources of truth.** Slice order: WorldClock (done) → **fog facts (DONE 2026-07-04: FogVisibility 组件早已真 bitecs；[[fog-facts-slice]] 补上 FogRevealModel 揭示强度切片+烘焙退役)** → battle overlay (NEXT) → mode/modal → input routing. Do NOT convert world-map renderer authority or tutorial flow first (too many boundaries; they hide ECS mistakes). Fog gets fog-specific components — no shared generic component.

**Status caveat (don't be fooled by green guards):** only WorldClock is real BitECS today. `ecs/owner/` (BattleOwner, FogOwner), `ecs/projection/`, `ecs/snapshot/`, the `*Model/*State/*Snapshot` modules are still POJO owner/wrapper = migration debt the standard itself flags. Even the new `check-frontend-ecs-fog-owner` guard freezes fog into a FogOwner (an Owner = a rejected way-station). The guard suite (EcsCoreBoundary seam, core-guard, boundary-skeleton, platform-boundary, host-bridge-retired, fog-owner) enforces SEAMS + layout + bitecs-pinned-0.4.0, but does NOT yet enforce "state lives in components" — a POJO owner can still pass the layout gate.
