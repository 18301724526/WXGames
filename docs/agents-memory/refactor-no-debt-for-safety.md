On the architecture refactor program, the user FORBIDS deferring or keeping any debt / impure / unhealthy architecture *for safety's sake*. "It's the riskiest surgery, so leave it for later / wrap it in a permanent proxy / just guard it without deleting" is exactly the reasoning they reject. Concretely this made **P3 Axis A** (remove `CanvasGameShell extends CanvasGameApp`, collapse the triple-host live-state mirror to ONE owner) the correct next step — its only deferral reason in the handoff doc was "riskiest / only partly unit-tested," which is debt-preservation-for-safety.

**Why:** the program's north star is "fix any bug without a global search"; preserving a mirror/facade for safety just relocates the debt and keeps the guard green while both copies move together (violates 铁律 #2 "remove/derive, don't just guard").

**How to apply:** distinguish two kinds of safety — (1) preserving debt to feel safe = BANNED (keeping inheritance, a permanent `shell→lastGame` proxy, guard-only-no-delete); (2) tests + verify to make a deletion observable = REQUIRED. A transitional getter+setter proxy is allowed ONLY as scaffolding that is deleted at the end; the end state must be a single state owner with `extends` gone. When recommending a next step, pick the option that fully eliminates the debt.

**Verification model (user-refined, 2026-06-29 — this is HOW to be safe without preserving debt):** the incremental safety net is **characterization tests + read-proof of behavior-equivalence**, NOT live testing. LIVE/game testing is a ONE-SHOT FINAL pass for only the residue tests cannot encode (real pixels / animation smoothness / browser frame-order). Do NOT ask the user to live-verify mid-refactor. Every refactor step must be **read-provably behavior-preserving** (dead-write deletion, pure substitution, provably-same-value accessor) + pinned by a characterization test. **Needing mid-way human verification is a RED FLAG** that the step wasn't read-provable — meaning either (a) you patched / added non-locality (deviation — revert & redo), or (b) the equivalence depends on coupling that isn't readable yet (decouple FIRST so it becomes readable; never paper over with a live check). If a step's equivalence is genuinely un-read-provable, surface it as a STRUCTURAL FINDING ("coupled to the point of unreadability, must decouple X first"), don't hide it behind a live test. Corollary the user asserted (and it's right): a correct architecture makes ~all bugs deterministic/必现 and root-causable by reading code; if a final-state failure can't be attributed by reading, the architecture is still wrong (= you left/created non-locality). Rationale: reading proves equivalence for LOCAL/pure changes (no oracle needed); an oracle (test, or one-shot live) is needed only where equivalence isn't locally evident — and that not-evident-ness is itself the coupling debt to remove. "Don't run the game until the whole refactor is done" is the user's standing instruction and follows from this.

**2026-06-29 update — the proxy MECHANISM is retired, the PRINCIPLE stands.** The getter/setter `HOST_PROXIED_FIELDS` proxy I used for Axis A was deleted by office commit `65821c0c` because under the new [[bitecs-ecs-standard]] a write-forwarding shell proxy / single-POJO-owner is itself a rejected "壳". The de-mirroring INTENT (kill the multi-writer mirror) is still correct and is the necessary PRE-STEP, but the endpoint is now BitECS `defineComponent` arrays, not a single POJO owner. So: keep using characterization-tests + read-proof as the safety net, but stop reaching for proxy/owner/wrapper collapses — migrate the fact into a real BitECS component per the WorldClock template instead.
