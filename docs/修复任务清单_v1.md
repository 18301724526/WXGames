# 文明火种 - 前端后端修复任务清单

**来源**：策划审阅报告（2026-05-15）  
**执行方**：主程（Codex）  
**原则**：前端只显示，后端是唯一计算和决策来源

---

## 任务1：[P0] 前端 GameConfig.js 移除硬编码建筑参数

**文件**：`frontend/js/config/GameConfig.js`  
**问题**：`maxLevel`、`description`、`effectText` 全部写死在前端，与后端 `shared/buildingConfig.json` 双轨并行。

**修复要求**：
- 删除 `BUILDINGS` 对象中所有 `maxLevel`、`description`、`effectText` 字段
- 前端 `BUILDINGS` 只保留纯展示字段：`id`、`name`、`icon`
- `maxLevel` 判定权移回后端（后端通过 `buildingCosts` 返回 `null` 表示已满级）
- `description` / `effectText` 不再从前端 `GameConfig.js` 读取

**验收标准**：
- 前端 `GameConfig.js` 中 `BUILDINGS` 对象只含 `id`、`name`、`icon`
- 后端修改 `farm.maxLevel` 从 4 改到 5 时，前端不需要改任何代码也能正确显示

---

## 任务2：[P0] 前端 BuildingUIRenderer 移除硬编码解锁列表

**文件**：`frontend/js/ui/BuildingUIRenderer.js`  
**问题**：`getVisibleIds()` 里写死了 `if (state.currentEra === 1) return ['farm', 'house']`，前端自己决定显示哪些建筑。

**修复要求**：
- 删除 `getVisibleIds()` 方法中所有硬编码 era 判断
- 直接使用后端返回的 `state.unlockedBuildings` 数组作为渲染依据
- 教程 Step 5 的锁定态（house不可点）通过后端 `tutorial` 字段 + 建筑卡片的 `disabled` 逻辑处理，不是通过硬编码隐藏卡片

**验收标准**：
- 后端在 `GET /api/game/state` 的 `gameState.unlockedBuildings` 中返回 `['farm', 'house', 'workshop']` 时，前端自动显示3张卡片
- 农耕时代解锁建筑变更不需要改前端代码

---

## 任务3：[P0] 前端建筑效果文案动态化

**文件**：`frontend/js/ui/BuildingUIRenderer.js`  
**问题**：`config.effectText` 是死值（永远"食物产出 +50%"），不随等级变化。

**修复要求**：
- `BuildingUIRenderer.render()` 中不再读取 `config.effectText`
- 改为从后端返回的 `state.buildingEffects` 动态计算：
  - `foodOutputMultiplier = 1.5` → "食物产出 +50%"
  - `foodOutputMultiplier = 2.0` → "食物产出 +100%"
  - `foodOutputMultiplier = 2.5` → "食物产出 +150%"
- 同理处理：`populationCap`、`happinessBonus`、`knowledgeOutputMultiplier` 等所有 `buildingEffects` 字段

**验收标准**：
- 农田从 Lv1 升到 Lv2 后，卡片效果文案自动从 "+50%" 变为 "+100%"
- 文案计算逻辑不在前端硬编码，全部从 `state.buildingEffects` 读取

## 任务4：[P1] 前端"已满级"判定移回后端

**文件**：`frontend/js/domain/BuildingState.js`  
**问题**：`getActionLabel()` 用前端 `buildingConfig.maxLevel` 判断"已满级"，阈值与后端可能不一致。

**修复要求**：
- 后端 `GET /api/game/state` 返回的 `buildingCosts[id]` 中：
  - 可建造/升级时 → 返回成本对象 `{ food: 50 }`
  - 已满级时 → 返回 `null` 或 `{ maxLevel: true }`
- 前端 `getActionLabel()` 改为判断 `state.buildingCosts[id] === null` 时显示"已满级"
- 删除前端对 `buildingConfig.maxLevel` 的任何读取

**验收标准**：
- 后端把 `farm.maxLevel` 从 4 改到 3，前端不需要改代码，按钮在 Lv3 时自动显示"已满级"

---

## 任务5：[P1] 删除遗留单机引擎 game.js

**文件**：`frontend/game.js`  
**问题**：这是旧版完全独立的单机游戏引擎，有自己的 tick 循环、计算公式、存档系统。虽然 index.html 没加载它，但存在被误引用的风险。

**修复要求**：
- 删除 `frontend/game.js` 或重命名为 `game.old.v1.js` 并加注释 `"已废弃，不要引用"`

**验收标准**：
- `frontend/` 目录下不存在独立的 `game.js` 文件

---

## 任务6：[P1] 后端 tick 补充人口增长机制

**文件**：`backend/server.js`  
**问题**：`setInterval` 只更新食物和知识，没有人口自然增长逻辑。玩家造完民居后人口上限6，但实际永远只有3人。

**修复要求**：
- tick 中每 1 秒给 `population.growthProgress += 1`
- 当 `growthProgress >= 120`（120秒，参数化）时：
  - `growthProgress = 0`
  - 如果 `population.total < population.max`：
    - `population.total += 1`
    - `population.unassigned += 1`
- 人口增长周期参数化：`GameConfig.population.growthInterval = 120`（秒）

**验收标准**：
- 造完民居后约120秒，总人口从3自动增长到4
- 增长时 `unassigned` 同步+1，玩家可立即分配

---

## 任务7：[P1] 后端食物产出公式参数化

**文件**：`backend/calculators/ResourceTickCalculator.js`  
**问题**：基础数值（农民产出1.0、知识0.05、消耗0.2/人）目前可能硬编码在代码里。

**修复要求**：
- 将以下数值抽到 `backend/config/GameConfig.js`：
  - `baseFoodPerFarmer = 1.0`
  - `baseKnowledgePerPerson = 0.05`
  - `foodConsumptionPerPerson = 0.2`
  - `populationGrowthInterval = 120`
- `ResourceTickCalculator` 从 `GameConfig` 读取，不再写死

**验收标准**：
- 改 `GameConfig.baseFoodPerFarmer` 从 1.0 到 1.2，前端不需要改代码，产出自动变化

---

## 任务8：[P2] 食物消耗可视化（tooltip）

**文件**：`frontend/js/ui/ResourceUIRenderer.js`（如不存在则新建，或在现有渲染逻辑中实现）  
**问题**：当前只显示净产出 "+3.9/s"，玩家看不到拆解。

**修复要求**：
- 资源面板增加 tooltip / 展开态：
  - 农民产出：`farmers × baseFoodPerFarmer × multiplier`
  - 人口消耗：`- total × consumption`
  - 净增长：`net`
- 文案格式示例："产出 4.5/s | 消耗 -0.6/s | 净 +3.9/s"

**验收标准**：
- 点击或长按食物面板时显示完整拆解
- 所有数字来自后端 `state.resources` 和 `state.buildingEffects`

## 依赖关系

```
任务3 依赖 任务1（effectText动态化需要先移除硬编码）
任务4 依赖 后端 API 格式调整（buildingCosts返回null表示已满级）
任务6 依赖 任务7（人口增长周期参数化）
```

**建议执行顺序**：
1. 先执行任务5（删除遗留文件，降低风险）
2. 执行任务7（参数化，为后续打基础）
3. 执行任务1 + 任务2 + 任务3（前端去硬编码，一组改完）
4. 执行任务4（已满级判定后端化）
5. 执行任务6（人口增长）
6. 最后任务8（UI增强）

---

## 全局约束

1. **前端不得引入任何计算逻辑**：所有数值（成本、效果、上限、解锁列表）必须来自后端 API
2. **前端不得硬编码建筑ID以外的任何建筑参数**：`maxLevel` / `cost` / `effect` 全部后端说了算
3. **后端参数化所有数值**：避免硬编码在 calculator 中
4. **新增时代或建筑时只改配置文件**：不改 JS 代码（开闭原则验证）
