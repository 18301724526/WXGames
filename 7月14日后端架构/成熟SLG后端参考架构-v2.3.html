成熟 SLG 后端实现参考与验收基准 v2.3

实现级标准：规定 placement CAS、handoff 线性化、提交前 fencing、可恢复命令、核心表约束、持久事件水位、断线续传、机器判定容量合同和可执行恢复。单机与多机只能改变部署映射，不得改变这些语义。

ARCHITECTURE VERDICT

REFERENCE_V2_3_STATUS = NOT_PROVEN（等待第四轮对抗审核），CURRENT_IMPLEMENTATION = NON_CONFORMING。v2.3 已吸收第三轮交叉质询裁决（REVISE_TO_V2_3）的全部 P0/P1/P2 修订：handoff 最终 fencing 谓词与激活 CAS、授权 epoch 的 admission 线性化边界、restore 单例准入、PITR × 外部消费对账与 seq 不重用、restore 九步执行语义、协议与数据面收尾。机器数量不参与正确性判定。C1-C7、D1-D7、R1-R6 中任意一项失败即不通过；O1-O6 决定能否进入对外测试或生产。规范缺少执行语义标为 UNDER_SPECIFIED，当前代码未实现标为 IMPLEMENTATION_ONLY_GAP；只有两条规范要求无法同时满足时才标为 SPEC_CONTRADICTION。

Correctness Gates

  • C1 每个可变 aggregate 只有当前 owner_epoch 的 holder 可写；每笔事务在数据库内验证 fencing。
  • C2 服务端推导完整 owner 集合并校验所有引用实体权限；客户端 ownerKey 不构成授权。
  • C3 同一 owner 串行；多 owner 预声明、按唯一顺序加锁，动态发现 owner 必须回滚并重路由。
  • C4 命令携带所有 touched owner 的 expectedVersions；任一冲突整体拒绝并重新求值。
  • C5 调度器、后台任务和管理接口不得绕过命令入口直接改权威状态。
  • C6 placement 仅通过数据库单行 CAS 分配新 epoch；handoff 具有唯一线性化点、超时、回收主体和提交前最终 fencing。
  • C7 M_max 只控制 admission；原子业务不得因容量阈值自动变成 Saga，拆分必须由持久 execution plan 和产品语义证明。

Data + Event Gates

  • D1 领域行、aggregate 版本、命令终态、定时任务、持久事件和 outbox 在同一事务提交。
  • D2 短期 outbox 与客户端续传事件流职责分离；snapshot cut 使用已提交事件水位。
  • D3 消费者具有 per-stream cursor、缺口检测、重放、退避、最大重试和 DLQ。
  • D4 缓存、搜索索引、读模型均不是权威来源，可被完整重建。
  • D5 付费货币、充值、退款和补偿进入不可变 ledger；幂等键不能代替业务唯一约束。
  • D6 job 完成 CAS 与领域写入同事务；旧 token/epoch 影响零行时整个事务回滚。
  • D7 Saga confirm/compensate 受状态、版本和 dedupe key 守卫；迟到消息不能越过补偿终态。

Release + Recovery Gates

  • R1 release manifest 锁定后端、前端、配置、DB schema、协议和事件 schema。
  • R2 线上运行不可变制品并原子切换；Git worktree 不承担线上目录。
  • R3 API、协议、事件和数据库迁移遵循兼容矩阵与 expand-migrate-contract。
  • R4 权威数据由 DB/PITR 和持久流恢复；基础设施由版本配置重建，密钥来自 secret manager。
  • R5 恢复先 fence 全部 writer，再做 integrity、replay、未决工作流检查和玩法 smoke。
  • R6 release/restore 具有持久 run、唯一执行租约、逐步幂等 guard、超时、证据 digest、信任根和显式流量重开授权。

Operational Gates

  • O1 健康检查覆盖 owner lease、scheduler、outbox、事件流、DB、磁盘和恢复 lag。
  • O2 容量结果通过版本化 JSON Schema 和确定性 evaluator 校验，绑定联合负载、runId、单位、release/hardware 与原始证据 digest。
  • O3 后端仅私网可达，凭据不入代码和日志，权限最小化并可审计。
  • O4 生产与非生产隔离；单机部署时也必须有资源预算和故障边界。
  • O5 按 endpoint、账号、session、owner、AOI 和成本设置可测量限制，不使用万能固定数字。
  • O6 gate 只有 PASS/FAIL/NOT_PROVEN/NOT_APPLICABLE；缺字段、跨 run 拼接、未知 writer 或无审批 NA 一律不得 PASS。
01

一次权威写入到底怎样发生

成熟实现不是“多个服务都能写同一份状态”。每条命令必须具有稳定 ID、服务端推导的 owner 集合、当前 fencing epoch、全量 expectedVersions 和可查询终态；重试只恢复同一 commandId，不重新创造一次业务事实。

命令状态机与未知提交：admission 小事务以业务唯一键插入 ACCEPTED，并同事务把 credential/session/authz 三类 epoch 快照写入 receipt（授权判定的线性化点）；执行者以 CAS 写入 IN_PROGRESS(execution_token, execution_lease_until)。领域事务重新锁定 receipt，并把领域写入、heads、jobs、stream/outbox 和 terminal receipt 一起提交。执行者崩溃后，recovery guardian 只能 CAS 接管已过期 token；连接在 COMMIT 附近断开时，客户端仅查询同一 commandId。因此允许“receipt 存在但尚无领域写入”，禁止“领域写入已提交但 receipt 非终态”。
外部副作用边界：支付、通知、邮件和第三方 API 不得在可回滚领域事务中直接调用。领域事务只写 transactional outbox；接收方按稳定业务键去重，必要时使用 reserve/confirm/compensate。handoff 或 lease 撤销只能阻止尚未提交的数据库写，不能被宣称为能够撤销已经发生的外部事实。
当前实现差距：backend/application/commands/CommandCommitter.js:50 先持久化领域状态，:110 再单独记录命令结果，尚未满足 D1 的原子终态要求。与此同时，WorldWorkerService.executeInternalCommand():231-256 已进入统一 CommandExecutionPipeline；不能把“worker 完全绕过管线”写成事实。
02

不同类型写入采用不同一致性边界

SLG 的关键不是把所有操作塞进一个玩家 JSON。单 owner、有限多 owner 和长事务工作流必须采用三种明确边界，并且共享同一 fencing、幂等和审计语义。

写入类型 典型命令 owner 与事务 冲突/失败处理 明确禁止
单玩家 / 单城市 升级建筑、科技、征兵、领取奖励、资源消费 player:{id} 或 city:{id} 邮箱；事务验证 owner_leases.epoch，并更新领域行、head、terminal receipt、jobs、stream event 与 outbox。 expectedVersions 冲突即拒绝；相同 commandId 返回原终态；余额约束与业务唯一键共同防重复。 API 与 worker 分别读取旧快照后覆盖整行玩家状态。

Owner Handoff Rule

active → draining → transferring → active。placement 以数据库 CAS 创建 handoff_id 和 drain_deadline；draining 拒绝新 admission，但 in-flight 事务仍可提交。旧 holder 的最终 fencing 谓词明文为：state='active' OR (state='draining' AND handoff_id=:handoff_id AND statement_timestamp() < drain_deadline)，配合 holder_id、owner_epoch、lease_until 匹配与行数断言（0 行即整个事务 raise 并回滚）。线性化点是同一事务内校验旧 epoch/state、递增 epoch 并写入新 holder 的成功 handoff CAS；committed_watermarks 由 placement 在该 CAS 同一事务内从 stream_heads 原子记录，新 holder 只能从该记录值接管。transferring → active 激活是独立 CAS：执行主体是新 holder，前置谓词为 state='transferring' AND holder_id=:new_holder AND owner_epoch=:new_epoch；激活未在 transfer 超时前完成时，placement recovery 以 CAS 回滚为旧 holder active（恢复原 holder_id）或强制递增 epoch 指派其他 holder，禁止 owner 永久停留在 transferring。旧 holder 的每笔事务在 COMMIT 前持有 lease 行锁并最终校验，deadline 后由 placement recovery CAS 回收。

Owner Set Rule

路由前计算并记录 owner_set_hash、每个 owner 的服务端 expectedVersion 来源和 plan_attempt。owner_set_hash 的规范化构造钉死为：owner_key 按字节序（memcmp）升序排序、去重后以 \n 连接的 SHA-256——两个 plan 相同 owner 集合必得同一 hash。执行者只按 receipt.plan_attempt 当前值读取对应 plan；plan_attempt 只能经 receipt 行上带 status/execution_token 前置谓词的 CAS 单调递增。执行阶段发现未声明 owner 时不得临时补锁；事务回滚，旧 plan 标记 superseded，重新授权并以同一 commandId 生成新 execution plan。plan 行本体不可变：BEFORE UPDATE/DELETE 触发器只放行 state/superseded_by 列变更，其余列一律拒绝。M_max 是 admission 参数，不得改变业务原子性。

Saga Rule

只有产品允许中间状态可见、每一步可幂等且存在补偿/人工收敛语义时才能使用 Saga。超过 M_max 但要求原子可见的命令必须拒绝、重塑分区或改业务规则，不能自动拆分。execution plan、reservation、confirm/compensate dedupe key 和状态守卫全部持久化；迟到 confirm 在 compensating/completed 后必须失败。

Hot State Rule

资源、建筑、军队、行军、地块归属使用窄行和明确类型。JSON 只用于低频、不可索引或不可变结果，不允许一行玩家状态包含几十个 JSON 字段并被整行重写。

Partition Rule

所有世界游戏表携带 realm_id；地图和战斗再携带 zone/cell。分区键从主键、唯一约束、路由、事件 stream、迁移到恢复保持一致。合服/拆服需要显式 ID 与游标迁移方案，不能从复合键“自动正确”推导；该方案登记为 v2.3+ 专项设计债，起点采纳 realm_migration_cut 表方案（迁移 cut 持久化：源/目标 realm、ID 映射、各 stream 游标截断点、ledger 结转与验收证据）。

Time Rule

持久化 finish_at/due_at 和状态，不每秒写倒计时。客户端用 serverTime 本地渲染；到期由 scheduled job 重新提交确定性命令。

Constraint Rule

幂等处理网络重投，数据库唯一键、CHECK、FK、CAS 和不可变 ledger 处理业务重复与并发。两者不能互相替代；重复奖励、重复占领、重复入盟必须由 domain constraint 最终阻止。

Knowledge Rule

player_tile_vision 表示“玩家知道什么”，不是“世界现在是什么”。AOI 推送必须取当前权威世界状态与当前可见权限的交集，权限撤销会使订阅和缓存立即失效。

History Rule

战斗、经济、事件和 Saga 历史必须携带解释它们所需的 ruleset/config/event schema 版本。版本制品的保留周期不得短于相应审计和回放周期。

CREATE TABLE owner_leases (
  realm_id    BIGINT NOT NULL,
  owner_key   TEXT NOT NULL,
  owner_epoch BIGINT NOT NULL,
  holder_id   TEXT NOT NULL,
  state       TEXT NOT NULL CHECK
    (state IN ('active','draining','transferring')),
  lease_until TIMESTAMPTZ NOT NULL,
  handoff_id  UUID,
  drain_deadline TIMESTAMPTZ,
  transfer_to TEXT,
  committed_watermarks JSONB NOT NULL DEFAULT '{}',
  updated_at  TIMESTAMPTZ NOT NULL,
  PRIMARY KEY (realm_id, owner_key),
  UNIQUE (realm_id, handoff_id)
);

CREATE TABLE aggregate_heads (
  realm_id BIGINT NOT NULL,
  owner_key TEXT NOT NULL,
  version BIGINT NOT NULL DEFAULT 0,
  committed_event_seq BIGINT NOT NULL DEFAULT 0,
  updated_at TIMESTAMPTZ NOT NULL,
  PRIMARY KEY (realm_id, owner_key)
);

-- Placement handoff CAS: one DB row is the authority. The SAME
-- transaction atomically records committed_watermarks from
-- stream_heads; the new holder may only resume from this value.
UPDATE owner_leases
SET owner_epoch = owner_epoch + 1, holder_id = :new_holder,
    state = 'transferring', lease_until = :new_lease,
    committed_watermarks = (
      SELECT COALESCE(jsonb_object_agg(stream_key, committed_seq), '{}')
      FROM stream_heads
      WHERE realm_id = :realm AND stream_key = ANY(:owner_streams)),
    updated_at = now()
WHERE realm_id = :realm AND owner_key = :owner
  AND owner_epoch = :observed_epoch
  AND state = 'draining' AND handoff_id = :handoff_id
RETURNING owner_epoch;
-- assert affected_rows = 1, else the handoff CAS lost.

-- transferring -> active activation CAS, executed by the NEW holder
-- after loading the recorded committed_watermarks:
UPDATE owner_leases
SET state = 'active', lease_until = :new_lease, updated_at = now()
WHERE realm_id = :realm AND owner_key = :owner
  AND state = 'transferring' AND holder_id = :new_holder
  AND owner_epoch = :new_epoch
RETURNING owner_epoch;
-- assert affected_rows = 1; 0 rows => activation lost, stop serving.
-- Not activated before :transfer_deadline => placement recovery CAS
-- rolls back to the old holder (state='active') or forces a new epoch.

-- Final fencing inside the domain transaction (self-contained).
-- Step 1: lock the lease row and hold the lock through COMMIT.
--   Final fencing predicate: state='active' OR (state='draining'
--   AND handoff_id=:handoff_id AND now < drain_deadline).
SELECT owner_epoch
FROM owner_leases
WHERE realm_id = :realm AND owner_key = :owner
  AND holder_id = :holder AND owner_epoch = :epoch
  AND lease_until > statement_timestamp()
  AND (state = 'active'
    OR (state = 'draining' AND handoff_id = :handoff_id
        AND statement_timestamp() < drain_deadline))
FOR UPDATE;
-- assert returned rows = 1;
-- 0 rows => raise and roll back every domain write in this transaction.

-- Step 2: aggregate CAS under the held lease lock.
UPDATE aggregate_heads h
SET version = version + 1, updated_at = now()
WHERE h.realm_id = :realm AND h.owner_key = :owner
  AND h.version = :expected_version
RETURNING h.version;
-- assert affected_rows = 1, else raise and roll back.

-- domain state + terminal status + result. A status query never
-- advances state and is the only client recovery path for unknown
-- COMMIT; it must therefore read linearizably from the authoritative
-- store (or declare a bounded max staleness in the release manifest)
-- and is on the authoritative-read inventory.

Session / refresh 状态机：账号 credential_version、设备 session_epoch、授权 authz_epoch 分别处理密码/凭据撤销、单设备会话撤销和权限变化。refresh family 语义钉死为 per-device：每个设备 session 一个 refresh_family_id，refresh token 采用 family + rotation CAS，旧 refresh 重放撤销该设备 family 并递增该 session 的 session_epoch（撤销半径=单设备，不触发全账号踢出）。sessions 表持有双 hash 列 access_token_hash 与 refresh_token_hash，各自 UNIQUE。多设备迁移矩阵：密码重置/凭据泄露 → credential_version++（全设备失效）；单设备登出、refresh 重放、设备策略拒绝 → 该 session_epoch++（单设备失效）；角色/权限/联盟职位变化 → authz_epoch++（账号级，session 不失效）。请求和 stream HELLO 必须同时校验三类 epoch，撤销传播有产品 SLO 和可观测证据，不能依赖 access token 自然过期。
授权线性化边界（撤销语义=读法 a）：授权判定的线性化点是 admission 小事务把三类 epoch 快照进 command_receipts 的时刻；撤销的正确性边界 = admission 时点授权 + 有界传播 SLO。SLO 是撤销传播延时的上界证据，不是授权契约——普通命令在 SLO 窗口内以旧 epoch 提交属声明内行为，须有 stale-commit 证据计数。以下受管制命令类别必须在领域事务 commit 点复核 epoch：支付/经济类、管理员操作、权限授予类（联盟任命、转让等）；这些命令在领域事务内重读 sessions 行（普通读即可，撤销写方提交后的下一次重读必见）并断言与 admission 快照一致，不一致即回滚拒绝。网关的 session/epoch 缓存条目年龄超过撤销 SLO 时一律 fail-closed 拒绝，不得以陈旧缓存放行。
授权过滤与旧包：服务端无法保证已经在网络中的旧包“零到达”。每次订阅/重订阅生成递增 subscriptionGeneration，权限变化递增 permissionEpoch；客户端在应用前丢弃旧 generation/epoch。filtered/AOI 客户端流的 envelope streamSeq 一律是 visible seq；source stream 的序号不得出网，只作为服务端审计元数据。visible seq 采用持久化路径（二选一已钉死）：filtered stream 的 visible_seq → source event_id 映射行与 visible_head_seq 在授权过滤投影事务中同事务持久化，服务端崩溃后从持久映射续传，不做基于权限历史的确定性重算——P5 的续传承诺由此在 filtered 流上可验收。服务端主动收缩权限后，必须在受影响的 ordered stream 上发送 in-band SUBSCRIPTION_INVALIDATED 控制事件（活性与缓存卫生要求，不是泄漏门禁）：客户端收到后失效本地缓存并以新 generation 重订阅；空闲流也必须能在撤销 SLO 内收到该事件。
动态 owner 与 expectedVersion：客户端版本只表达其已知 projection。服务端授权阶段必须为追加引用从权威 heads 取得 expectedVersion 并记录来源；执行期发现新 owner 时原事务回滚，旧 plan 标记 superseded，重新授权后生成新 plan_attempt。不得在持锁中临时追加 owner 或把服务端新发现版本伪装成客户端前置条件。

    evidence_digest = :chained_step_digest, updated_at = now()

  AND current_step = :this_step

-- Sub-steps of STREAMS_REPLAYED after a PITR cut at T1:
-- (1) For EVERY consumer_group, compare its acked_seq against the
--     restored stream heads:

FROM consumer_cursors c
JOIN stream_heads h USING (realm_id, stream_key)

-- Server enforcement: on startup after a restore, and before the
-- gateway re-admits writes, the server verifies the token signature
-- against the trust root, matches restoreId to the completed
-- restore_runs row and evidenceChainDigest to its recorded chain.
-- Missing or invalid token => the server stays in read-only mode
-- (fail-closed); no operator flag can bypass this check.

-- Signer key lifecycle (applies to release + restore signing keys):
-- validity window: every key declares not_before / not_after;
--   signatures are valid only for tokens/manifests issued inside it.
-- overlap: rotation keeps old + new keys verifiable for a declared
--   overlap window so in-flight manifests/tokens stay valid.
-- revocation: a revoked key immediately blocks NEW switch/rollback/
--   reopen decisions; already-running instances keep running but
--   their manifests must be re-signed within the declared grace
--   window or the release becomes ineligible for re-selection.
--   Rollback targets (vN-1) signed by the revoked key must be
--   re-signed BEFORE the rollback path is considered available.
-- trust root: verification keys live in the secret manager /
--   dedicated trust store, independent of the runtime directory
--   and of the artifacts they verify.
