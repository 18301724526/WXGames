# 容量合同确定性判定器规范 v2.2

判定器版本：`slg-capacity-evaluator-v2.2`  
结构输入：`容量合同-v2.2.schema.json`  
状态：`REFERENCE_V2_2_STATUS = NOT_PROVEN`，等待第三轮机械攻击

## 1. 职责边界

JSON Schema 负责字段存在性、类型、枚举、基础范围和部分条件约束。本判定器负责 JSON Schema 无法表达的跨字段、数组聚合、证据引用、数学关系和结果优先级。

任何报告只有同时满足以下条件才可进入判定：

1. 按 JSON Schema 校验通过。
2. 判定器实现的制品 digest 与合同中的 `evaluator.algorithmDigest` 一致。
3. 输入、schema、判定器和所有 evidence artifact 都使用 RFC 8785 JSON Canonicalization Scheme 后计算 SHA-256；非 JSON 原始制品按原始 bytes 计算。
4. release manifest 验签通过，且 digest 与合同 `release.manifestDigest` 相同。

结构校验失败或判定器 digest 不匹配时，输出 `NOT_PROVEN`，不得尝试“尽量解析”。

## 2. 输出与退出码

判定器只输出一个规范化 JSON：

```json
{
  "schemaVersion": "slg-capacity-evaluation-v2.2",
  "runId": "uuid",
  "result": "PASS|FAIL|NOT_PROVEN|NOT_APPLICABLE",
  "reasonCodes": ["..."],
  "gateResults": [{ "gateId": "...", "result": "...", "reasonCodes": ["..."] }],
  "inputDigest": "sha256:...",
  "evaluatorDigest": "sha256:..."
}
```

退出码：`0=PASS`，`2=FAIL`，`3=NOT_PROVEN`，`4=NOT_APPLICABLE`，`5=invalid invocation/internal evaluator error`。内部错误不得转换为 PASS 或业务 FAIL。

## 3. 结果优先级

优先级从高到低：

1. 任一正确性、守恒、fencing、receipt、cursor、授权、release identity、restore 或最终 checksum 不变量失败：`FAIL`。
2. 没有观察到不变量失败，但 scope、阈值、联合负载、证据、digest、inventory、阶段或判定器输入不完整：`NOT_PROVEN`。
3. gate 明确不适用于该部署，并具有非空理由、审批人、审批证据和替代保证：该 gate 可为 `NOT_APPLICABLE`。
4. 只有所有适用 gate 都为 PASS、所有 NA 审批有效、最终状态收敛，整体结果才为 `PASS`。

`FAIL` 不能被平均值、其他 PASS gate 或后续 retry 覆盖。`NOT_PROVEN` 不能被“未观察到失败”升级。

## 4. 规范化与证据绑定

判定器必须：

- 校验所有 `evidence.artifacts[].runId` 等于根 `runId`。
- 校验每个 stage/gate 的 `evidenceRefs` 都唯一引用同一合同中的 artifact；缺失或悬空引用为 `NOT_PROVEN`。
- 重新读取 artifact bytes 并计算 digest；不可访问、digest 不符或读取错误为 `NOT_PROVEN`。
- 校验 artifact 的时间戳位于 run 的 `[startedAt, completedAt]`，预先生成的固定 fixture/manifest 只能作为 environment/release 输入，不得伪装成测量证据。
- 禁止引用另一个 run 的聚合摘要；跨 run 基线只能作为比较项，不得满足本 run required condition。
- 校验 `rawEvidenceDigest` 是按 evidenceId 排序后的 `(evidenceId,digest)` 规范化清单 digest。

## 5. Workload 语义校验

### 5.1 Mix

分别对 `commandMix`、`jobMix`、`consumerMix`：

- type 必须唯一并存在于 inventory。
- 使用 decimal 精确求和；与 1 的差不得大于 `floatingPointTolerance`。
- 任一 type 在 runtime trace 中零命中时，相关 scope 为不完整，结果 `NOT_PROVEN`。
- runtime 出现 inventory/mix 外未知 type 时，结果 `NOT_PROVEN`；若未知写入破坏不变量，结果 `FAIL`。

### 5.2 Phase

phase 必须严格按 `warmup → steady → burst → fault → drain → cooldown` 排列，各出现一次。每个 evidence window：

- start < end。
- 位于 run 时间范围内。
- 相邻 phase 不重叠；若产品明确允许采样重叠，必须使用不同 evidence stream 并在 gate 中声明，不能复用同一 sample 同时证明 steady 和 drain。
- durationSeconds 与 evidence window 差值在采样时钟误差阈值内。

### 5.3 形状关系

- `totalPlayers >= concurrentActiveUsers`。
- `connections >= concurrentActiveUsers`，除非 workload 明确为无连接后台模型；此时相关 client/socket gate 必须有 NA 审批。
- `touchedOwnersP99 <= touchedOwnersMax`。
- `mMax` 必须绑定同 release 的容量参数；超过 mMax 的原子命令计入 reject，不得静默转 Saga。
- hotOwnerFraction、realm/zone skew、AOI density、data age 和 retention 必须能从原始 trace/fixture 重算。

## 6. 分布与单位校验

所有 percentile 对象必须满足：

```text
0 <= p50 <= p95 <= p99 <= max
```

单位映射固定：

| 字段 | 允许单位 |
|---|---|
| arrivalRate | requests_per_second / events_per_second |
| serviceTime | ms |
| queueDepth | items / bytes（若两者都需要，必须拆为两个 metric） |
| queueOldestAge | ms |
| utilization | ratio，且所有值位于 [0,1] |

字段单位错误、分位数逆序、样本数不足、只保留成功样本或采样丢失率超过产品阈值时为 `NOT_PROVEN`。已知丢失样本集中在错误/超时路径时为 `FAIL`，因为证据选择破坏了结论真实性。

## 7. Condition 求值

每个 condition 的 `source` 只能是：

- `PRODUCT_SLO`：threshold 的来源。
- `MEASURED`：observed 的来源。

不得使用 MODEL_ASSUMPTION 产生 required condition；模型假设只能出现在评审附录，`modelAssumptionsUsed=true` 的 gate 不得 PASS。

运算符按 IEEE 754 有限数执行，但先拒绝 NaN/Infinity。`eq/neq` 对非整数使用 `floatingPointTolerance`；其他运算符直接比较。每个 condition 必须保存 threshold 和 observed 的原始 sample/evidenceRef，不能只保存格式化字符串。

## 8. Gate 唯一判定

对每个 gate 重新计算，不信任输入的 `result`：

```text
if any correctness invariant failed:
  FAIL
else if !scopeComplete
     or !evidenceValid
     or modelAssumptionsUsed
     or any required input/condition/evidence missing:
  NOT_PROVEN
else if any required condition evaluates false:
  FAIL
else if !allInvariantsPassed or !allSlosPassed or !drainPassed:
  FAIL
else:
  PASS
```

输入 `gate.result` 必须等于重算结果，否则整体 `NOT_PROVEN` 并报告 `DECLARED_RESULT_MISMATCH`。如果矛盾项包含被隐藏的不变量失败，整体仍按优先级判 FAIL。

NOT_APPLICABLE 只能在 gate 的 scope inventory 确认路径不存在时使用；“尚未实现”“没有测”“单机所以不管”不是 NA 理由。审批证据必须来自 release 责任人和架构责任人，且 digest 纳入 evidence chain。

## 9. Stage 与共享资源

每个 stage 必须从同一 trace 重算 arrival/service/queue/utilization/deadline。判定器至少检查：

- `rho = arrivalRate.mean * serviceTime.mean` 使用相同 class/window/unit，并与实测 utilization 交叉验证。
- queue oldest age、bytes/items、reject 和 retry amplification 来自同一 phase。
- stage p99 不允许直接相加冒充端到端 p99；端到端 deadline 必须由同一 request/command/job trace 计算。
- 共享 CPU、DB pool、WAL/disk、network、GC、socket buffer 的 resource center 都有独立 evidence；缺少任一实际瓶颈中心为 NOT_PROVEN。
- 同一 owner/zone/alliance 与均匀 owner 分开报告，不得用混合平均隐藏热点串行极限。

## 10. Fault、Drain 与最终状态

- 每个 fault 的实际开始/结束时间和目标必须存在于 fault timeline artifact。
- fault 未成功注入或目标未受影响时，对应 gate 为 NOT_PROVEN。
- drain 阶段必须同时满足：新流量按合同继续、前台 SLO 不失败、backlog 单调收敛、deadline 内归零或回到 steady bound。
- 通过丢弃权威事件、缩短 retention、跳过 poison item、清除 receipt/job/Saga 或停止前台流量得到的 drain 为 FAIL。
- `expectedChecksum` 来自 fault 前固定 fixture + 已接受命令事实的确定性重算；不能在测试结束后用 observed 值回填。
- unresolved receipt/job/Saga、cursor gap、ledger imbalance、owner violation 任一非零时整体 FAIL。

## 11. 整体结果

判定器重新计算整体结果：

```text
if any gate == FAIL or final correctness invariant failed:
  FAIL
else if any applicable gate == NOT_PROVEN:
  NOT_PROVEN
else if all gates are NOT_APPLICABLE:
  NOT_PROVEN
else if every applicable gate == PASS and every NA approval is valid:
  PASS
```

输入 `overallResult` 与重算结果不同则输出重算结果，并增加 `DECLARED_OVERALL_RESULT_MISMATCH`。不得为了兼容旧报告降低结果。

## 12. 判定器自身验收

发布判定器前必须有 mutation tests，至少证明以下坏合同不能 PASS：

- overall PASS 但一个 gate FAIL/NOT_PROVEN。
- unresolved receipt/job/Saga 非零或 backlog 未排空。
- mix 总和不为 1、重复/缺失 phase、时间窗倒置。
- p99 < p95、单位错误、utilization > 1。
- condition observed 不满足 operator 但声明 PASS。
- evidenceRef 悬空、跨 run、digest 不匹配或 artifact 不可读取。
- inventory 有 unknown/zero-hit writer。
- fault 未实际生效、drain 通过丢事件实现、最终 checksum 回填 observed。
- evaluator 自身 digest 与合同不符。

这些 mutation fixtures、期望结果和判定器二进制/源码 digest 必须进入 release manifest。判定器没有通过自身 mutation suite 前，所有容量报告保持 NOT_PROVEN。
