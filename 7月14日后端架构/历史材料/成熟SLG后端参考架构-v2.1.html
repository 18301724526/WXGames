成熟 SLG 后端实现参考与验收基准 v2.1

实现级标准：规定 fenced owner、原子命令结果、核心表约束、持久事件水位、断线续传、容量合同和恢复证据。单机与多机只能改变部署映射，不得改变这些语义。

ARCHITECTURE VERDICT

REFERENCE_V2_STATUS = NOT_PROVEN，CURRENT_IMPLEMENTATION = NON_CONFORMING。机器数量不参与正确性判定。C1-C5、D1-D5、R1-R5 中任意一项失败即不通过；不得以“用户少、冲突概率低、增加重试、提高 busy timeout”代替证明。O1-O5 决定该架构能否进入对外测试或生产运行。

Correctness Gates

  • C1 每个可变 aggregate 只有当前 owner_epoch 的 holder 可写；每笔事务在数据库内验证 fencing。
  • C2 服务端推导完整 owner 集合并校验所有引用实体权限；客户端 ownerKey 不构成授权。
  • C3 同一 owner 串行；多 owner 预声明、按唯一顺序加锁，动态发现 owner 必须回滚并重路由。
  • C4 命令携带所有 touched owner 的 expectedVersions；任一冲突整体拒绝并重新求值。
  • C5 调度器、后台任务和管理接口不得绕过命令入口直接改权威状态。

Data + Event Gates

  • D1 领域行、aggregate 版本、命令终态、定时任务、持久事件和 outbox 在同一事务提交。
  • D2 短期 outbox 与客户端续传事件流职责分离；snapshot cut 使用已提交事件水位。
  • D3 消费者具有 per-stream cursor、缺口检测、重放、退避、最大重试和 DLQ。
  • D4 缓存、搜索索引、读模型均不是权威来源，可被完整重建。
  • D5 付费货币、充值、退款和补偿进入不可变 ledger；幂等键不能代替业务唯一约束。

Release + Recovery Gates

  • R1 release manifest 锁定后端、前端、配置、DB schema、协议和事件 schema。
  • R2 线上运行不可变制品并原子切换；Git worktree 不承担线上目录。
  • R3 API、协议、事件和数据库迁移遵循兼容矩阵与 expand-migrate-contract。
  • R4 权威数据由 DB/PITR 和持久流恢复；基础设施由版本配置重建，密钥来自 secret manager。
  • R5 恢复先 fence 全部 writer，再做 integrity、replay、未决工作流检查和玩法 smoke。

Operational Gates

  • O1 健康检查覆盖 owner lease、scheduler、outbox、事件流、DB、磁盘和恢复 lag。
  • O2 versioned workload manifest 声明负载形状、阶段容量、队列、deadline 和停止条件。
  • O3 后端仅私网可达，凭据不入代码和日志，权限最小化并可审计。
  • O4 生产与非生产隔离；单机部署时也必须有资源预算和故障边界。
  • O5 按 endpoint、账号、session、owner、AOI 和成本设置可测量限制，不使用万能固定数字。
01

一次权威写入到底怎样发生

成熟实现不是“多个服务都能写同一份状态”。每条命令必须具有稳定 ID、服务端推导的 owner 集合、当前 fencing epoch、全量 expectedVersions 和可查询终态；重试只恢复同一 commandId，不重新创造一次业务事实。

命令状态机与未知提交：RECEIVED → ACCEPTED → IN_PROGRESS → COMMITTED | REJECTED_FINAL | FAILED_FINAL。领域事务必须把终态 receipt 与领域写入一起提交。连接在 COMMIT 附近断开时，客户端以 commandId 查询或重试；服务端只返回已存终态，或继续恢复同一命令。相同 ID 不同 payload hash 必须拒绝。ACK 仅报告命令终态和事件水位，不作为第二条状态应用通道。
当前实现差距：backend/application/commands/CommandCommitter.js:50 先持久化领域状态，:110 再单独记录命令结果，尚未满足 D1 的原子终态要求。与此同时，WorldWorkerService.executeInternalCommand():231-256 已进入统一 CommandExecutionPipeline；不能把“worker 完全绕过管线”写成事实。
02

不同类型写入采用不同一致性边界

SLG 的关键不是把所有操作塞进一个玩家 JSON。单 owner、有限多 owner 和长事务工作流必须采用三种明确边界，并且共享同一 fencing、幂等和审计语义。

写入类型 典型命令 owner 与事务 冲突/失败处理 明确禁止
单玩家 / 单城市 升级建筑、科技、征兵、领取奖励、资源消费 player:{id} 或 city:{id} 邮箱；事务验证 owner_leases.epoch，并更新领域行、head、terminal receipt、jobs、stream event 与 outbox。 expectedVersions 冲突即拒绝；相同 commandId 返回原终态；余额约束与业务唯一键共同防重复。 API 与 worker 分别读取旧快照后覆盖整行玩家状态。
同 realm / zone 多实体 开始行军、占领地块、战斗结算、双方资源转移 服务端预声明完整 owner 集合；仅当 owner 数不超过 capacity contract 的 M_max 时，按全局锁顺序在同一数据库分区事务内更新 army、march、tile、battle。 锁序固定为 receipt → owner_leases → aggregate_heads/stream_heads → 按表序与主键排序的领域行。动态 owner 发现导致整体回滚和重路由。 跨多个进程各写一半后再“尽量补齐”。
跨 zone / realm 跨服迁移、跨区邮件、联盟协作、排行榜奖励 使用持久 Saga：reserve → dispatch → confirm，失败进入 compensate；每个 step 通过稳定 commandId 进入对应 owner runtime，每一步仍是本地事务。 实例、步骤和 reservation 全部持久化；超时进入 compensating，无法自动收敛进入 escalated/manual_review，禁止静默遗留半完成状态。 跨数据库双写、无状态机的分布式“同时成功”。
定时推进 建筑完成、行军到达、体力恢复、活动结算 scheduled_jobs 以稳定 job:{jobId} 命令进入 owner runtime；claim 必须原子写入 lease_token/lease_epoch，旧 lease 无权完成任务。 按 due_at、priority class、job_id 稳定出队；指数退避、最大尝试和 DLQ 有界；重复投递由 receipt 去重。 timer worker 直接 UPDATE 玩家、地块或行军状态。
读模型 / 推送 排行榜、地图视野、战报、在线状态、前端 projection 消费持久事件流后更新 projection；outbox 仅是短期 relay spool。客户端与后台消费者分别维护 per-stream cursor。 重复事件幂等，缺口阻止 cursor 前移并触发重放；超出保留窗口才使用带 committed watermark 的 snapshot。 把 Redis、缓存或 projection 当成资源余额和领地归属的权威来源。

Owner Handoff Rule

active → draining → transferring → active。draining 后拒绝新命令并排空 mailbox；旧 holder 释放或被撤销后递增 owner_epoch，新 holder 从已提交 watermark 接管。旧 epoch 的任何写入必须在数据库内失败。

Owner Set Rule

路由前计算并记录 owner_set_hash。执行阶段发现未声明 owner 时不得临时补锁；事务回滚，重新授权并重路由同一 commandId。M_max 是版本化容量参数，不是写死的通用数字。

Saga Rule

超过 M_max、跨分区或包含外部系统的工作必须拆为可补偿步骤。reservation 具有到期时间、业务唯一键和守恒校验；人工仲裁是显式终态，不允许无限重试。

Hot State Rule

资源、建筑、军队、行军、地块归属使用窄行和明确类型。JSON 只用于低频、不可索引或不可变结果，不允许一行玩家状态包含几十个 JSON 字段并被整行重写。

Partition Rule

所有世界游戏表携带 realm_id；地图和战斗再携带 zone/cell。分区键从主键、唯一约束、路由、事件 stream、迁移到恢复保持一致。合服/拆服需要显式 ID 与游标迁移方案，不能从复合键“自动正确”推导。

Time Rule

持久化 finish_at/due_at 和状态，不每秒写倒计时。客户端用 serverTime 本地渲染；到期由 scheduled job 重新提交确定性命令。

Constraint Rule

幂等处理网络重投，数据库唯一键、CHECK、FK、CAS 和不可变 ledger 处理业务重复与并发。两者不能互相替代；重复奖励、重复占领、重复入盟必须由 domain constraint 最终阻止。

Knowledge Rule

player_tile_vision 表示“玩家知道什么”，不是“世界现在是什么”。AOI 推送必须取当前权威世界状态与当前可见权限的交集，权限撤销会使订阅和缓存立即失效。

History Rule

战斗、经济、事件和 Saga 历史必须携带解释它们所需的 ruleset/config/event schema 版本。版本制品的保留周期不得短于相应审计和回放周期。

owner lease + fenced aggregate CAS

CREATE TABLE owner_leases (
  realm_id    BIGINT NOT NULL,
  owner_key   TEXT NOT NULL,
  owner_epoch BIGINT NOT NULL,
  holder_id   TEXT NOT NULL,
  state       TEXT NOT NULL CHECK
    (state IN ('active','draining','transferring')),
  lease_until TIMESTAMPTZ NOT NULL,
  transfer_to TEXT,
  updated_at  TIMESTAMPTZ NOT NULL,
  PRIMARY KEY (realm_id, owner_key)
);

CREATE TABLE aggregate_heads (
  realm_id BIGINT NOT NULL,
  owner_key TEXT NOT NULL,
  version BIGINT NOT NULL DEFAULT 0,
  committed_event_seq BIGINT NOT NULL DEFAULT 0,
  updated_at TIMESTAMPTZ NOT NULL,
  PRIMARY KEY (realm_id, owner_key)
);

-- Inside the domain transaction, never only in memory:
UPDATE aggregate_heads h
SET version = version + 1, updated_at = now()
FROM owner_leases l
WHERE h.realm_id = :realm AND h.owner_key = :owner
  AND l.realm_id = h.realm_id AND l.owner_key = h.owner_key
  AND l.holder_id = :holder AND l.owner_epoch = :epoch
  AND l.state = 'active' AND l.lease_until > statement_timestamp()
  AND h.version = :expected_version
RETURNING h.version;

command receipt terminal state

CREATE TABLE command_receipts (
  realm_id BIGINT NOT NULL,
  command_id UUID NOT NULL,
  session_id UUID NOT NULL,
  client_seq BIGINT NOT NULL,
  payload_hash BYTEA NOT NULL,
  owner_set_hash BYTEA NOT NULL,
  expected_versions JSONB NOT NULL,
  status TEXT NOT NULL CHECK (status IN (
    'accepted','in_progress','committed',
    'rejected_final','failed_final'
  )),
  result JSONB,
  result_versions JSONB,
  event_watermarks JSONB,
  terminal_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ NOT NULL,
  PRIMARY KEY (realm_id, command_id),
  UNIQUE (realm_id, session_id, client_seq)
);

-- The same transaction that changes domain rows must update
-- in_progress -> committed/rejected_final and persist result.
-- A status query by command_id is the recovery path for an unknown COMMIT.

scheduled job lease + stable claim

CREATE TABLE scheduled_jobs (
  realm_id BIGINT NOT NULL,
  job_id UUID NOT NULL,
  owner_key TEXT NOT NULL,
  due_at TIMESTAMPTZ NOT NULL,
  priority_class SMALLINT NOT NULL,
  command_type TEXT NOT NULL,
  payload JSONB NOT NULL,

    (status IN ('pending','leased','completed','dead_letter')),
  lease_token UUID,
  lease_epoch BIGINT NOT NULL DEFAULT 0,
  lease_until TIMESTAMPTZ,

  max_attempts INTEGER NOT NULL,

);

CREATE INDEX scheduled_jobs_due
  ON scheduled_jobs
  (realm_id, status, next_attempt_at, due_at, priority_class, job_id);

  stream_key TEXT NOT NULL,
  stream_seq BIGINT NOT NULL,
  event_id UUID NOT NULL,
  owner_key TEXT NOT NULL,
  owner_version BIGINT NOT NULL,

  realm_id BIGINT NOT NULL,
  event_id UUID NOT NULL,
  destination TEXT NOT NULL,
  stream_key TEXT NOT NULL,
  stream_seq BIGINT NOT NULL,
  status TEXT NOT NULL,

  ON outbox_events (realm_id, status, next_attempt_at)

P1 Session Epoch密码重置、账号冻结、显式登出和风险处置递增 credential/session epoch；每个请求校验。多设备是明确策略，不是偶然共享 token。
P2 Intent + Authorization客户端只提交意图。服务端推导 owner，并校验 payload 中 city/army/tile/alliance 等每个引用实体的归属与操作权限。
P3 Stable Retry超时复用 commandId，并可查询终态。网络幂等只处理重投；奖励、支付、占领和入盟仍必须有 domain business key/constraint。
P4 One State ChannelACK 只解决命令生命周期；客户端 projection 只应用 ordered event/snapshot，禁止 ACK patch 与 EVENT patch 竞争覆盖。
P5 Per-stream Resume玩家、AOI、聊天等 stream 各自维护 cursor、head、retainedFrom 和 gap 状态；慢流不能阻塞其他流恢复。
P6 Committed Snapshot Cutsnapshot 返回 per-stream committed watermarks；应用后仅接收 seq > watermark。cut 不依赖 relay 的 published_at。
P7 AOI Permissionviewport、zone、实体密度和 fanout 有 endpoint-specific budget；推送取权威世界状态与当前 visibility 的交集，权限 epoch 变化立即失效订阅。
P8 Version Matrixrelease manifest 声明客户端协议、配置、ruleset、事件 schema 和服务端读写范围；支持几个旧版本属于产品发布策略，不是固定常数。
P9 Lightweight HeartbeatPING/PONG 只维护连接、serverTime 和 presence TTL；不得 1Hz 加载完整玩家状态、地图或做源码 hash。
P10 Server Authority倒计时由 serverTime + finishAt 本地显示；seed、结算、奖励和运动真值由服务端生成。缺失客户端报告不能改变权威结果。
P11 Transport EquivalenceWebSocket、SSE 和 HTTP fallback 必须保持相同 commandId、event seq、cursor、错误和恢复语义。
P12 Resource Budgets按 endpoint、IP、account、session、player、owner、命令类型、body 深度/解压后大小和 AOI 成本分别限额并输出明确拒绝原因。

06

发布、恢复、安全和运行证据

正确的数据路径如果无法兼容发布、隔离故障和可重复恢复，仍然不是可运营架构。所有数值阈值由产品 SLO 和实测填写，不从示例复制。

领域强制设计失败 / 恢复语义唯一可接受证据
Immutable Releasemanifest 固定 backend、frontend、config、DB schema、protocol、event schema 的 digest 与兼容范围；制品 build once。原子切换完整 release；失败回滚到另一份完整 manifest，不在运行目录 git checkout 或逐文件修补。签名 manifest、制品 digest、切换 trace、健康 gate 和完整回滚演练。
Expand-Migrate-Contractexpand 阶段兼容旧新 reader/writer；迁移可断点续跑并对账；旧版本清零后 contract。任一阶段失败均停在仍兼容状态；禁止 schema、代码、事件 producer 同时做不可逆切换。版本兼容矩阵、migration checksum、行数/守恒对账、旧 reader 数和回滚测试。
Data Restore权威 DB 使用备份/PITR；持久事件流与对象数据按各自恢复方案；基础设施由版本配置和制品重建。恢复顺序为 fence writers → stop scheduler/relay → restore → integrity → replay → reconcile → smoke → issue new epochs。定期全量演练，报告实际 RPO/RTO、stream gap、未决 receipt/job/Saga 和最终 checksum。
Writer Fencing恢复、failover、迁移和部署前撤销/递增 owner epoch；旧 holder、旧 scheduler lease、旧 DB primary 无写权。提交结果未知通过 receipt/status 恢复；旧 epoch 写、旧 lease complete 和旧主提交必须确定失败。split-brain、GC pause、进程重叠、DB failover 和 restore-orphan-writer 故障注入。
Durability Profile逐存储声明 process crash、OS/power loss、disk corruption、replica lag 的保证和确认点。不得把进程 OOM/kill 与主机掉电混为同一丢失语义；ACK 只在声明的 durable commit point 后返回。按故障类型重复 crash/power/failover 测试，核对所有 ACKed commandId 与恢复后终态。
Secrets + Identity密钥来自 secret manager，服务到服务使用可轮换身份与最小权限；日志、trace、备份不含明文秘密。轮换支持明确重叠窗口和撤销；secret manager 不可用时按 endpoint 拒绝，不回退到代码默认值。secret scan、权限矩阵、轮换演练、撤销时延、审计日志和敏感字段泄漏测试。
Health + Runbooks健康覆盖 owner lease/epoch、scheduler lag、outbox/event lag、DB/replica、disk、snapshot、restore/replay 和证书。liveness 不等于 correctness；假活、cursor 停滞、磁盘趋满和恢复分叉必须在用户错误前触发隔离或降级。每个信号的故障注入、告警时延、自动化边界、值班 runbook 和人工升级记录。
