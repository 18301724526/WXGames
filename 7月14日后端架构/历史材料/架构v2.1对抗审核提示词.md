# SLG 架构 v2.1 第二轮对抗审查提示词

主评审对象：`docs/slg-backend-reference-architecture.html`

本轮目标不是确认 v2.1 “已经完善”，而是攻击它新增的实现级规范，寻找仍然存在的矛盾、竞态、不可实现条款、证据漏洞和容量逃生口。

固定原则：

```text
REVIEW_ROUND = 2
REVIEW_MODE = ADVERSARIAL
REFERENCE_V2_STATUS = NOT_PROVEN
CURRENT_IMPLEMENTATION = NON_CONFORMING
BURDEN_OF_PROOF = ARCHITECTURE
MACHINE_COUNT_IS_NOT_CORRECTNESS = TRUE
SMALL_SCALE_IS_NOT_A_WAIVER = TRUE
CONSENSUS_IS_NOT_PROOF = TRUE
FOCUS = BREAK_V2_1_AND_FIND_MISSING_FOUNDATIONS
```

所有评审席共同遵守：

1. 不得以“当前用户少”“单机先够用”“以后再拆”“多重试几次”作为通过理由。
2. 不得因为 v2.1 新增了字段或表名就判定闭环；必须验证状态机、锁顺序、故障点、恢复路径和可执行测试。
3. 数值阈值必须来自产品 SLO 和隔离测量。不得把示例数字写成通用正确答案。
4. 发现源码事实时，先用 CodeGraph 定位并引用当前行号；不得把推测写成 FACT。
5. 每条保留要求必须落到 SQL/约束、事务时间线、协议 trace、容量公式、故障注入或恢复报告之一。
6. 只读评审。除指定报告外，不得修改产品代码、配置、数据库、进程、服务器或架构文档。
