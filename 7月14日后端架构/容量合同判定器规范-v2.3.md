# 容量合同确定性判定器规范 v2.3

判定器版本：`slg-capacity-evaluator-v2.3`  
结构输入：`容量合同-v2.3.schema.json`  
状态：`REFERENCE_V2_3_STATUS = NOT_PROVEN`，等待第四轮机械攻击

## 1. 职责边界

JSON Schema 负责字段存在性、类型、枚举、基础范围和部分条件约束。本判定器负责 JSON Schema 无法表达的跨字段、数组聚合、证据引用、数学关系和结果优先级。

任何报告只有同时满足以下条件才可进入判定：

1. 按 JSON Schema 校验通过。
2. 判定器实现的制品 digest 与合同中的 `evaluator.algorithmDigest` 一致。
3. 输入、schema、判定器和所有 evidence artifact 都使用 RFC 8785 JSON Canonicalization Scheme 后计算 SHA-256；非 JSON 原始制品按原始 bytes 计算。
4. release manifest 验签通过，且 digest 与合同 `release.manifestDigest` 相同。

结构校验失败或判定器 digest 不匹配时，输出 `NOT_PROVEN`，不得尝试“尽量解析”。

## 2. 输出与退出码

判定器只输出一个规范化 JSON：

```json
{
  "schemaVersion": "slg-capacity-evaluation-v2.3",
  "runId": "uuid",
  "result": "PASS|FAIL|NOT_PROVEN|NOT_APPLICABLE",
  "reasonCodes": ["..."],
  "gateResults": [{ "gateId": "...", "result": "...", "reasonCodes": ["..."] }],
  "inputDigest": "sha256:...",
  "evaluatorDigest": "sha256:..."
}
```

退出码：`0=PASS`，`2=FAIL`，`3=NOT_PROVEN`，`4=NOT_APPLICABLE`，`5=invalid invocation/internal evaluator error`。内部错误不得转换为 PASS 或业务 FAIL。

## 3. 结果优先级

优先级从高到低：

1. 任一正确性、守恒、fencing、receipt、cursor、授权、release identity、restore 或最终 checksum 不变量失败：`FAIL`。
2. 没有观察到不变量失败，但 scope、阈值、联合负载、证据、digest、inventory、阶段或判定器输入不完整：`NOT_PROVEN`。
3. gate 明确不适用于该部署，并具有非空理由、审批人、审批证据和替代保证：该 gate 可为 `NOT_APPLICABLE`。
4. 只有所有适用 gate 都为 PASS、所有 NA gate 的审批链完整有效（验证步骤见 §11）、最终状态守恒复算通过（§10.1），整体结果才为 `PASS`。

`FAIL` 不能被平均值、其他 PASS gate 或后续 retry 覆盖。`NOT_PROVEN` 不能被“未观察到失败”升级。

## 4. 规范化与证据绑定

判定器必须：

- 校验所有 `evidence.artifacts[].runId` 等于根 `runId`。
- 校验每个 stage/gate 的 `evidenceRefs` 都唯一引用同一合同中的 artifact；缺失或悬空引用为 `NOT_PROVEN`。
- 重新读取 artifact bytes 并计算 digest；不可访问、digest 不符或读取错误为 `NOT_PROVEN`。
- 校验 artifact 的时间戳位于 run 的 `[startedAt, completedAt]`，预先生成的固定 fixture/manifest 只能作为 environment/release 输入，不得伪装成测量证据。
- 禁止引用另一个 run 的聚合摘要；跨 run 基线只能作为比较项，不得满足本 run required condition。
- 校验 `rawEvidenceDigest` 是按 evidenceId 排序后的 `(evidenceId,digest)` 规范化清单 digest。

## 5. Workload 语义校验

### 5.1 Mix

分别对 `commandMix`、`jobMix`、`consumerMix`：

- type 必须唯一并存在于 inventory。
- 使用 decimal 精确求和；与 1 的差不得大于 `floatingPointTolerance`。
- 任一 type 在 runtime trace 中零命中时，相关 scope 为不完整，结果 `NOT_PROVEN`。
- runtime 出现 inventory/mix 外未知 type 时，结果 `NOT_PROVEN`；若未知写入破坏不变量，结果 `FAIL`。

### 5.2 Phase

phase 必须严格按 `warmup → steady → burst → fault → drain → cooldown` 排列，各出现一次。每个 evidence window：

- start < end。
- 位于 run 时间范围内。
- 相邻 phase 不重叠；若产品明确允许采样重叠，必须使用不同 evidence stream 并在 gate 中声明，不能复用同一 sample 同时证明 steady 和 drain。
- durationSeconds 与 evidence window 差值在采样时钟误差阈值内。

### 5.3 形状关系

- `totalPlayers >= concurrentActiveUsers`。
- `connections >= concurrentActiveUsers`，除非 workload 明确为无连接后台模型；此时相关 client/socket gate 必须有 NA 审批。
- `touchedOwnersP99 <= touchedOwnersMax`。
- `workload.mMax` 必须精确等于 release manifest 容量参数中声明的 `mMax`（等值校验见 §5.4，不相等即 `NOT_PROVEN`）；超过 mMax 的原子命令计入 reject，不得静默转 Saga。
- hotOwnerFraction、realm/zone skew、AOI density、data age 和 retention 必须能从原始 trace/fixture 重算。

### 5.4 Mandatory deployment profile 校验

判定器不接受实现方自选测试范围。执行主体是判定器本身，在 §8 的 gate 求值之前完成本节全部校验：

- 合同必须携带 `deploymentProfile`，且其 `profileId` 与 digest 与 release manifest 中登记的部署 profile 一致；缺失或不一致为 `NOT_PROVEN`（reasonCode `MANDATORY_PROFILE_MISSING`）。
- profile 的三张 required 清单各有精确的成员判定谓词，与 v2.3 schema deploymentProfile 描述一致：`requiredGateIds` 每个 id 必须在 `gates[].gateId` 中恰好出现一次；`requiredStageClasses` 每个 class 必须被至少一个 `stages[].class` 覆盖；`requiredFaultScenarioIds` 每个 id 必须作为某个 `stages[].scenarioId` 出现，且该 stage 的 `phase = fault` 并携带 fault-phase trace 证据（其 `evidenceRefs` 指向时间窗落在对应 fault phase window 内的 trace artifact）——fault 对象本身无 scenarioId 字段，成员判定一律以 `stages[].scenarioId` 为准，不得用自由字符串对 `workload.faults` 做无定义的成员判定。判定器逐清单计算集合差 `required − present`，任一差集非空即 `NOT_PROVEN`（reasonCode `MANDATORY_PROFILE_INCOMPLETE`），差集内容作为证据字段写入判定输出。requiredGateIds 中的 gate 以 `NOT_APPLICABLE` 出现仍计为出现，但其审批链必须按 §11 验证有效，否则 `NOT_PROVEN`。
- gate id 在整个合同内必须唯一；重复 id 为 `NOT_PROVEN`（reasonCode `DUPLICATE_GATE_ID`），不得以“取第一个”或“合并”方式尽量解析。
- `workload.mMax` 必须精确等于 release manifest 容量参数字段中声明的 `mMax`（v2.3 起 release 侧为必填 typed 字段，切断 capacityContractId 循环引用）；不相等、release 侧字段缺失或不可验签为 `NOT_PROVEN`（reasonCode `MMAX_BINDING_MISMATCH`）。
- 单 stage + 单自命名 gate + 空 faults 这类结构合法但覆盖为零的合同，必然因 required 清单缺席落入 `NOT_PROVEN`；不存在自选范围获得 PASS 的路径。

## 6. 分布与单位校验

所有 percentile 对象必须满足：

```text
0 <= p50 <= p95 <= p99 <= max
0 <= mean <= max
sampleCount >= 1
```

v2.3 起 percentiles 对象的 `mean` 与 `sampleCount` 为 schema 必填字段（`failedCount`/`droppedCount` 可选）；缺失即结构校验失败，按 §1 输出 `NOT_PROVEN`。`sampleCount` 低于产品声明的最小样本阈值时，引用该分布的 gate 为 `NOT_PROVEN`（reasonCode `INSUFFICIENT_SAMPLES`）。

单位映射固定：

| 字段 | 允许单位 |
|---|---|
| arrivalRate | requests_per_second / events_per_second |
| serviceTime | ms |
| queueDepth | items / bytes（若两者都需要，必须拆为两个 metric） |
| queueOldestAge | ms |
| utilization | ratio，且所有值位于 [0,1] |

字段单位错误、分位数逆序、样本数不足、只保留成功样本或采样丢失率超过产品阈值时为 `NOT_PROVEN`。已知丢失样本集中在错误/超时路径时为 `FAIL`，因为证据选择破坏了结论真实性。

## 7. Condition 求值

每个 condition 的 `source` 只能是：

- `PRODUCT_SLO`：threshold 的来源。
- `MEASURED`：observed 的来源。

不得使用 MODEL_ASSUMPTION 产生 required condition；模型假设只能出现在评审附录，`modelAssumptionsUsed=true` 的 gate 不得 PASS。

运算符按 IEEE 754 有限数执行，但先拒绝 NaN/Infinity。`eq/neq` 对非整数使用 `floatingPointTolerance`；其他运算符直接比较。每个 condition 必须保存 threshold 和 observed 的原始 sample/evidenceRef，不能只保存格式化字符串。

## 8. Gate 唯一判定

对每个 gate 重新计算，不信任输入的 `result`：

```text
if any correctness invariant failed:
  FAIL
else if !scopeComplete
     or !evidenceValid
     or modelAssumptionsUsed
     or any required input/condition/evidence missing:
  NOT_PROVEN
else if any required condition evaluates false:
  FAIL
else if !allInvariantsPassed or !allSlosPassed or !drainPassed:
  FAIL
else:
  PASS
```

输入 `gate.result` 必须等于重算结果，否则整体 `NOT_PROVEN` 并报告 `DECLARED_RESULT_MISMATCH`。如果矛盾项包含被隐藏的不变量失败，整体仍按优先级判 FAIL。

NOT_APPLICABLE 只能在 gate 的 scope inventory 确认路径不存在时使用；“尚未实现”“没有测”“单机所以不管”不是 NA 理由。审批证据必须来自 release 责任人和架构责任人，且 digest 纳入 evidence chain。

## 9. Stage 与共享资源

每个 stage 必须从同一 trace 重算 arrival/service/queue/utilization/deadline。判定器至少检查：

- `rho = arrivalRate.mean * serviceTime.mean`：`mean` 与 `sampleCount` 直接取自 percentiles 对象的必填字段（§6），判定器不得从格式化字符串或未定型 artifact 推断均值；两个 mean 必须使用相同 class/window/unit。判定器由此唯一复算 rho，与实测 utilization 交叉验证，并与该 stage evidenceRefs 指向的同一 `traceDigest` 原始 trace artifact 交叉验证——从 trace 重算的 mean/sampleCount 与合同字段之差超过 `floatingPointTolerance` 为 `NOT_PROVEN`（reasonCode `TRACE_FIELD_MISMATCH`）。任一参与分布的 `sampleCount` 低于最小样本阈值即样本不足，相关 gate 为 `NOT_PROVEN`。
- queue oldest age、bytes/items、reject 和 retry amplification 来自同一 phase。
- stage p99 不允许直接相加冒充端到端 p99；端到端 deadline 必须由同一 request/command/job trace 计算。
- 共享 resource center 的证据齐全性按 §9.2 校验；`deploymentProfile.requiredResourceCenterTypes` 声明的类型缺任一即为 NOT_PROVEN（reasonCode `RESOURCE_CENTER_UNCOVERED`）。
- 同一 owner/zone/alliance 与均匀 owner 分开报告，不得用混合平均隐藏热点串行极限。

### 9.1 联合 trace 校验

边际分布相同不代表相关峰值同时发生；独立 stage percentile 的任何组合都不能证明联合负载。判定器执行：

- 每个声明相关峰值/联合负载的 gate（含 reconnect storm × authz cache miss × snapshot build 这类联合场景）必须引用同一 `scenarioId` 与同一 `traceDigest` 的证据；判定器解析这些 gate 的全部 evidenceRefs，校验对应 artifact 均携带同一 (scenarioId, traceDigest) 对，且其时间窗落在同一 phase window 内。
- 来自不同 scenario 的证据拼接——scenarioId 不同、traceDigest 不同、或时间窗不重叠却声称同时发生——为 `NOT_PROVEN`（reasonCode `CROSS_SCENARIO_STITCHING`，判定输出列出冲突的 (gateId, scenarioId, traceDigest) 三元组作为证据字段）。
- 联合峰值 gate 的每个 condition 的 observed 值必须能从该联合 trace 单独复算；无法从同一 trace 复算的 observed 为 `NOT_PROVEN`。

### 9.2 Resource center 校验

- 集合差的左操作数是 typed 字段 `deploymentProfile.requiredResourceCenterTypes`（v2.3 schema 起为 deploymentProfile 必填数组，`uniqueItems`，items 复用 `resourceCenter.type` 的同一枚举 `cpu/gc/db_pool/lock/wal_disk/network/socket/client`，纳入 deploymentProfile 的 required 清单）；判定器计算集合差 `requiredResourceCenterTypes − {resourceCenters[].type}`，差集非空即 `NOT_PROVEN`（reasonCode `RESOURCE_CENTER_UNCOVERED`，与 schema resourceCenter 描述使用同一 reasonCode，本规范不再使用别名 `RESOURCE_CENTER_MISSING`；差集内容作为证据字段写入判定输出）。合同只提交 1 个 center 而 profile 要求 8 类时，其余 7 类即差集，判定器有确定输入可算，不依赖“实际瓶颈”的事后判断。
- 每个 center 必须有各自独立的 evidence artifact（独立 evidenceRefs；不得多个 center 共用同一聚合摘要冒充独立观测）；evidence 缺失、悬空或 digest 不符按 §4 处理为 `NOT_PROVEN`。
- 缺任一 required center 类型（`requiredResourceCenterTypes` 声明但合同 `resourceCenters` 未覆盖）时，依赖该 center 的容量结论不可证明，相关 gate 与整体均不得 PASS。

## 10. Fault、Drain 与最终状态

- 每个 fault 的实际开始/结束时间和目标必须存在于 fault timeline artifact。
- fault 未成功注入或目标未受影响时，对应 gate 为 NOT_PROVEN。
- drain 阶段必须同时满足：新流量按合同继续、前台 SLO 不失败、backlog 单调收敛、deadline 内归零或回到 steady bound。
- 通过丢弃权威事件、缩短 retention、跳过 poison item、清除 receipt/job/Saga 或停止前台流量得到的 drain 为 FAIL。
- `expectedChecksum` 来自 fault 前固定 fixture + 已接受命令事实的确定性重算；不能在测试结束后用 observed 值回填。
- unresolved receipt/job/Saga、cursor gap、ledger imbalance、owner violation 任一非零时整体 FAIL（复算方式见 §10.1，不信任声明值）。

### 10.1 最终状态守恒复算

执行主体是判定器本身。判定器不信任 finalState 中的任何布尔（如 `backlogDrained`）或“全零”计数声明，必须用 v2.3 schema finalState 的 typed 守恒字段逐条复算：

- accepted-to-terminal 守恒方程（字段取自 `finalState.commandConservation`，v2.3 schema 必填 typed 对象，`additionalProperties:false`，仅有 `accepted/committed/terminal/pending` 四个计数）：`commandConservation.accepted = commandConservation.terminal + commandConservation.pending`，且 `commandConservation.committed <= commandConservation.terminal`；drain gate 额外要求 `commandConservation.pending = 0`。拒绝与终态失败等全部出口都计入 `terminal`，schema 之外不存在 `rejected`/`terminalFailed` 独立字段，判定器不得按不存在的字段判“守恒字段缺失”。
- 权威事件守恒方程（字段取自 `finalState.eventConservation`）：`eventConservation.produced = eventConservation.applied + eventConservation.pending + eventConservation.dropped`，且 `eventConservation.dropped = 0`（含 run 期间 retention 收缩造成的丢弃）。
- cleanup/delete 断言（字段取自 `finalState.cleanupConservation`）：`cleanupConservation.cleanupCount = 0` 且 `cleanupConservation.deleteCount = 0`；任何通过清理积压、删除 receipt/job/Saga/authoritative event、缩短 retention 得到的“排空”直接暴露在这两个字段上，等价于 §10 的丢弃式 drain。
- `finalState.cursorGap.count`、`finalState.ledgerImbalance.count`、`finalState.ownerViolation.count` 与 `finalState.dlqUnresolved.count` 逐一复算为 0。
- 上述每个守恒对象（`commandConservation`/`eventConservation`/`cleanupConservation` 与各 conservationCounter）自带必填 `evidenceRefs`；判定器按 §4 重取 artifact bytes，从原始计数来源复算各字段值并与合同值比对。

任一守恒方程不平、任一断言字段非零、或合同字段与 artifact 复算值不一致时，整体 `FAIL`（reasonCode `CONSERVATION_VIOLATION`，判定输出携带不平方程的名称与两侧数值作为证据字段）。守恒字段缺失、evidenceRefs 悬空或 artifact 不可读取为 `NOT_PROVEN`。`backlogDrained:true` + 全零 unresolved 计数的声明组合在本节复算通过之前没有任何证明力。

## 11. 整体结果

判定器重新计算整体结果。本节与 v2.3 schema 的 overall PASS 分支对齐：schema 的 PASS 分支放行“全部适用 gate 为 PASS + 带完整审批链（reason + approvedBy + 审批证据 digest）的 NOT_APPLICABLE gate”的合同，如实申报 PASS 且含有效 NA 的合同结构校验通过；审批链的机械验证由本节执行。

```text
if any gate == FAIL or final correctness invariant failed or §10.1 conservation failed:
  FAIL
else if any applicable gate == NOT_PROVEN:
  NOT_PROVEN
else if all gates are NOT_APPLICABLE:
  NOT_PROVEN
else if any NOT_APPLICABLE gate's approval chain is missing or invalid:
  NOT_PROVEN
else if every applicable gate == PASS and every NA gate's approval chain is valid:
  PASS
```

NA 审批链验证步骤（判定器逐 NA gate 执行，任一步失败则该 NA 无效，整体 `NOT_PROVEN`，reasonCode `NA_APPROVAL_INVALID`）：

1. `reason` 非空，且不属于 §8 的禁用理由（“尚未实现”“没有测”“单机所以不管”）。
2. `approvedBy` 必须同时包含 release 责任人与架构责任人两个不同主体（双责任人），且身份与 release manifest 中登记的责任人一致；单人、重复主体或身份不符即无效。
3. 审批证据 artifact 的 digest 必须已纳入 evidence chain——出现在 `rawEvidenceDigest` 的按 evidenceId 排序的 (evidenceId, digest) 清单中，并按 §4 重取 bytes 复算 digest 一致；缺失、悬空或 digest 不符即无效。
4. NA gate 若属于 §5.4 的 requiredGateIds，其替代保证声明必须存在于审批证据中；缺失即无效。

输入 `overallResult` 与重算结果不同则输出重算结果，并增加 `DECLARED_OVERALL_RESULT_MISMATCH`；但 mismatch 通道不得产出 `PASS`——若重算结果为 PASS 而申报值与之不符，错报本身即证据一致性破坏，整体降为 `NOT_PROVEN`。由此不存在“故意错报 overallResult 绕过 schema PASS 分支、再由 mismatch 通道换取重算 PASS”的路径。不得为了兼容旧报告降低结果。

## 12. 判定器自身验收

发布判定器前必须有 mutation tests，至少证明以下坏合同不能 PASS：

- overall PASS 但一个 gate FAIL/NOT_PROVEN。
- unresolved receipt/job/Saga 非零或 backlog 未排空。
- mix 总和不为 1、重复/缺失 phase、时间窗倒置。
- p99 < p95、单位错误、utilization > 1。
- condition observed 不满足 operator 但声明 PASS。
- evidenceRef 悬空、跨 run、digest 不匹配或 artifact 不可读取。
- inventory 有 unknown/zero-hit writer。
- fault 未实际生效、drain 通过丢事件实现、最终 checksum 回填 observed。
- evaluator 自身 digest 与合同不符。

v2.3 新增 fixtures（含期望为 PASS 的正样例，验证判定器不过度拒绝）：

- 含带完整审批链（reason + 双责任人 approvedBy + 审批证据 digest 在 evidence chain 内）NOT_APPLICABLE gate、其余适用 gate 全 PASS 的合同：期望整体 `PASS`（验证 v2.3 schema PASS 分支放行有效 NA 与 §11 对齐）。
- NOT_APPLICABLE gate 审批链缺失或不完整（缺双责任人之一、或审批证据 digest 不在 evidence chain）：期望 `NOT_PROVEN`。
- 单 stage + 单自命名 gate + 空 faults + 自填 mMax 的结构合法合同：期望 `NOT_PROVEN`，reasonCode 含 mandatory profile 缺失（`MANDATORY_PROFILE_MISSING`/`MANDATORY_PROFILE_INCOMPLETE`，对应事故链 IC-5）。
- finalState 守恒不平（`commandConservation.accepted ≠ terminal + pending`、`commandConservation.committed > terminal`、`eventConservation.dropped > 0`，或 `cleanupConservation.cleanupCount`/`deleteCount` 非零）：期望 `FAIL`（reasonCode `CONSERVATION_VIOLATION`）。
- 守恒方程各侧字段与 schema 完全一致、复算平衡的正样例合同：期望不因守恒复算被拒（验证 §10.1 使用的字段路径与 v2.3 schema `$defs.commandConservation`/`$defs.eventConservation` 一致，不引用 schema 外字段）。
- percentiles 缺 `mean`/`sampleCount`：期望 schema 结构校验拒绝（按 §1 走 `NOT_PROVEN` 通道，不得尽量解析）。
