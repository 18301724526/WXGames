# wxgame 后端架构 v2.3 当前实现迁移路线图

版本：2026-07-14  
基准：`main@07f02da9` 及其后仅文档提交  
状态：`REFERENCE_V2_3_STATUS = NOT_PROVEN`（等待第四轮对抗审核）  
当前裁决：`CURRENT_IMPLEMENTATION = NON_CONFORMING`

## 1. 文档职责

`成熟SLG后端参考架构-v2.3.html` 定义目标语义，本文件只定义当前 wxgame 如何迁移到目标语义。当前代码缺少目标表或组件属于 `IMPLEMENTATION_ONLY_GAP`，不得据此降低目标标准；目标规范内部无法同时满足的要求才属于 `SPEC_CONTRADICTION`。

迁移不以“看起来像成熟架构”为完成标准。每一阶段都必须有可执行 DDL、事务 trace、故障注入、回滚路径和最终数据对账。单服务器部署不豁免 fencing、原子 receipt、持久 job/stream 或恢复语义。

## 2. 已核实的当前事实

- 已有统一 `CommandExecutionPipeline`，API 和 world worker 的主要权威写入正在向同一入口收敛。
- `CommandIdempotencyStore.begin()` 先持久化 `in_progress`；领域提交后再单独写 terminal result。崩溃窗口的真实结果是残留 `COMMAND_IN_FLIGHT`，不是“完全没有 receipt”。
- 主持久化仍以大型 `game_states` 行及多个 JSON 字段为主，热点写放大和细粒度约束能力不足。
- 已有 migration id/checksum/lock，缺少跨版本 expand-migrate-contract 政策、兼容矩阵和发布证据。
- 当前没有 v2.3 定义的 placement epoch、持久 client stream、完整 scheduled job lease、Saga、经济 ledger、restore coordinator 和容量判定器。
- 当前 worker 停止路径不构成 v2.3 fencing；进程生命周期不能替代数据库写权限。

## 3. 不可违反的迁移原则

1. 每次只迁移一个可证明的权威写入纵切面，旧新路径不得同时成为同一 aggregate 的权威 writer。
2. expand 阶段先增加可兼容结构；migrate 阶段可断点续跑并对账；旧 reader/writer 清零后才 contract。
3. 双写只能用于可校验 shadow，不得让两个结果共同决定业务状态。shadow 失败不修改权威结果。
4. 每阶段必须定义回滚到哪一个完整 release manifest；禁止逐文件手工回退。
5. correctness gate 失败立即停止。吞吐、重试或 busy timeout 不能掩盖原子性、守恒或 fencing 失败。
6. 不先承诺并发数字。容量结论必须在对应阶段用 `容量合同-v2.3.schema.json` 产生可复算证据。

## 4. 阶段依赖图

```text
M0 writer inventory + invariants + release identity
  -> M1 atomic receipt + transaction boundary
    -> M2 placement CAS + owner fencing + handoff
      -> M3 narrow authoritative aggregates
        -> M4 scheduled jobs + durable stream + outbox
          -> M5 session/protocol/resume
            -> M6 Saga + premium ledger
              -> M7 restore/release/capacity qualification
```

任何阶段发现上一阶段不变量没有持续成立，回退到上一阶段，不允许绕过依赖并行宣布完成。

## 5. M0：写入清单、业务不变量与发布身份

### 交付

- 静态和运行时 writer inventory：route、command、worker、admin、migration、repair script、consumer、client state writer。
- 每种命令的 owner set、领域表、业务唯一键、expectedVersion 来源、外部副作用和最终 projection 清单。
- 基线 release manifest：backend、frontend、config、schema、protocol、event schema digest。
- 当前数据库的脱敏 production-shape fixture、权威 checksum 规则和 restore rehearsal 基线。

### 退出门禁

- 未知或测试未命中的 runtime writer 数为 0。
- 所有付费、奖励、占领、入盟、行军、建筑完成路径都有明确业务不变量。
- 基线备份可在隔离环境恢复，且最终 checksum 可重复。

## 6. M1：原子 receipt 与命令恢复所有权

### 交付

- 新 `command_receipts` 和 `command_execution_plans` expand migration。`command_receipts` 必须含 `admission_credential_version`、`admission_session_epoch`、`admission_authz_epoch` 三列（FL-13 前半）。
- admission 小事务：稳定 commandId、payload hash、session/clientSeq 唯一约束；由 admission 事务本身在写入 receipt 的同一语句集内快照三类 epoch，任一 epoch 不可得即 fail-closed 拒绝 admission，不落 receipt。
- `accepted -> in_progress` execution token/lease CAS 和 recovery guardian。
- repository 提供显式事务上下文；领域行、heads、jobs/events/outbox、terminal receipt 在同一事务提交。
- statusUrl 只查询状态；客户端 retry 复用原 commandId。

### 强制故障注入

- admission 后、执行前崩溃。
- execution lease 期间进程暂停并由 guardian 接管。
- 领域 COMMIT 前、COMMIT 返回未知、COMMIT 后 ACK 前崩溃。
- terminal receipt 写入失败。

### 退出门禁

- 不存在领域状态已提交但 receipt 非终态的可达状态。
- 残留 in-progress 可由唯一 guardian 在 lease 到期后收敛。
- 同 commandId 不同 payload 永久拒绝；相同 payload 返回同一终态。
- 每条 accepted receipt 的三列 admission epoch 快照非空且等于 admission 时点 sessions 行的权威值（抽样比对可复核）；后续阶段（M5）以该快照为撤销窗口证据的锚点。

## 7. M2：Placement CAS、数据库 fencing 与 handoff

### 交付

- `owner_leases` expand migration及数据库单行 epoch CAS。
- placement authority 可以多实例请求，但同一行并发 CAS 只有一个成功；无需额外应用层共识服务。
- handoff_id、drain_deadline、transfer_to、committed watermarks 和 recovery owner。
- `transferring -> active` 激活 CAS（FL-01）：执行主体为新 holder，前置谓词 `state='transferring' AND transfer_to=:self AND handoff_id=:handoff_id AND epoch=:new_epoch`；激活带超时，超时后由 placement recovery 回收——回滚为旧 holder `active`（同 handoff_id 失效）或强制再递增 epoch 指派新 holder，禁止 owner 永久停留在 `transferring`。
- `committed_watermarks` 在 handoff CAS 的同一数据库事务内由 placement 从 `stream_heads` 原子记录；新 holder 只能从该记录值接管，不得自行探测水位。
- 每笔领域事务按全局顺序锁定所有 touched owner lease/head，提交前最终校验并持锁至 COMMIT；draining 期最终 fencing 谓词采用参考架构 v2.3 明文谓词（`active` 或 `draining` 且同 handoff_id 且未过 drain_deadline），不得私自放宽。
- 外部调用移到 transactional outbox；接收方以业务键去重。

### 强制故障注入

- 两个 placement 实例并发转移同一 owner。
- draining 时旧 mailbox 有积压；旧 holder pause 超过 deadline。
- 新旧进程重叠、旧 epoch 在 COMMIT 前恢复运行。
- handoff 期间包含多 owner 命令。
- handoff 注入（新 holder 崩溃/激活 CAS 前 pause/placement 崩溃）后，owner 必须在 deadline 内到达 `active` 或回滚到旧 holder（IC-2 判据）；期间 `scheduled_jobs` lag 有界，且事后不得以清理积压 job/receipt 制造 drain 通过。

### 退出门禁

- 每次成功转移产生唯一、严格递增 epoch。
- handoff 线性化点前后都只有一个可提交 writer。
- 旧 epoch、错误 holder、过期 lease 的最终 CAS 均影响 0 行并回滚整个事务。
- 不存在停留在 `transferring` 超过激活超时的 owner 行；每次 handoff 的 `committed_watermarks` 记录与 handoff CAS 同事务可审计。

## 8. M3：窄行权威 aggregate 与约束

### 顺序

1. `player_resources` 与付费/奖励业务键。
2. `cities`、`city_buildings`、`research_tasks`。
3. `armies`、`marches`、`world_tile_state`。
4. alliance、battle、vision 等共享域。

### 每个纵切面的迁移模板

- expand 新表、索引、FK/CHECK/UNIQUE 和 version head。
- 从 JSON backfill，记录 source row/version 和 checksum。
- 新路径 shadow read 比对，但仍由旧路径唯一写。
- 在一个 release manifest 中原子切换该 aggregate 的唯一 writer。
- 持续对账；旧 reader/writer 为 0 后 contract 对应 JSON 字段。

### 退出门禁

- 无变化 tick 不重写权威大 JSON。
- 热点命令只触及声明的窄行、heads、receipt、job/event/outbox。
- duplicate reward、negative spend、double occupation、double active march 等由数据库约束最终阻止。

## 9. M4：定时任务、持久事件与 outbox

### 交付

- `scheduled_jobs` 稳定 claim、lease token/epoch、退避、DLQ。
- job `leased -> completed` CAS 与领域写入同事务；影响 0 行整体回滚。
- `stream_heads`、`stream_events`、`consumer_cursors`、`inbox_receipts`。
- transactional outbox 只承担短期 relay；目标 durable accept 后可清理，与 stream retention 分离。
- snapshot 使用 committed per-stream watermark，增量从 `seq > watermark` 开始。

### 退出门禁

- scheduler pause/steal/restart 不会重复推进或丢任务。
- 多 stream 事务按规范化 key 全序锁定，所有 heads/events 原子提交。
- consumer gap 阻止 cursor 前移；慢 consumer 不会导致错误 outbox 语义或无界磁盘增长。

## 10. M5：Session、授权、命令状态与客户端续传

### 交付

- credential_version、session_epoch、authz_epoch 和 refresh family rotation 状态机。
- 客户端只提交 intent；服务端逐引用授权并记录 expectedVersion 来源。
- 受管制命令类（支付、管理员、权限授予）的 commit 点 epoch 复核（FL-13 后半）：领域事务在 COMMIT 前重读 sessions 当前三类 epoch（普通读即可）并断言与 receipt 的 admission 快照一致，不一致则整体回滚并以 `AUTH_EPOCH_STALE` 终态化 receipt；普通命令不做 commit 点复核，其撤销窗口由声明 SLO 与证据覆盖，session/epoch 缓存超 SLO 一律 fail-closed。
- ACK/statusUrl 只改变 pending command UI，不写 projection、不推进 applied cursor。
- 客户端 pending commandId 跨崩溃持久化（FL-16）：客户端本地持久化未终态的 commandId 集合，重启后经 statusUrl 收敛 outcome 归属，不重复生成新 commandId。
- 多条 stream 共用一种 event/snapshot 应用机制。
- subscription generation、permission epoch、连续 visible seq；客户端丢弃旧 generation/epoch 包。
- `SUBSCRIPTION_INVALIDATED` in-band 控制事件（FL-18 残余）：服务端主动收缩权限后经 ordered stream 下发，客户端据此失效本地缓存并重订阅；发送方为权限收缩的提交事务经 outbox/stream 投递，属活性与缓存卫生要求而非泄漏门禁。
- 已启用 transport 的 commandId、错误、seq、cursor 和恢复语义等价。

### 退出门禁

- 冻结、重置、登出、refresh 重放和权限撤销在声明 SLO 内使旧请求失效。
- 撤销竞态矩阵验收（IC-1 判据）：admission 后-COMMIT 前撤销注入 ×100 交叉（撤销 × admission × commit × plan_attempt 重授权全时序），受管制命令类 stale-commit 计数 = 0；普通命令的 stale-commit 全部落在声明 SLO 窗口内并留有 receipt admission 快照对比证据。
- EVENT 先于或后于 ACK 均收敛；status query 不影响状态应用顺序。
- AOI 取消/重订阅时旧包可到达但不可应用，也不能泄露撤销后的实体信息。
- 客户端崩溃重启后，持久化的 pending commandId 全部收敛到唯一终态归属，不产生重复提交。

## 11. M6：Saga 与付费经济账务

### 交付

- Saga instance/step/reservation 持久状态、版本 CAS、dedupe key、deadline 和 escalated 终态。
- saga dedupe_key 作用域（FL-06）：唯一约束采用 `UNIQUE(realm_id, saga_id, dedupe_key)`（或等价地在 key 构造规则中钉死必含 saga_id），无关 saga 不得相互阻塞；命中 dedupe 的迟到/重复消息幂等返回原结果，不报错、不产生新副作用。
- `M_max` 只做 admission：要求原子可见的命令超限时拒绝或重塑分区；只有产品允许中间状态时才转 Saga。
- append-only economy journal/ledger，按 currency 守恒。
- chargeback 的 receivable/liability 策略、部分退款上限谓词（累计退款 ≤ 原始入账，数据库谓词强制）、外部 unknown result、FX rate snapshot 和估值元数据。
- 账务结构对齐参考架构 v2.3（FL-07）：`fx_snapshots` 独立表（journal 行以 `fx_snapshot_id` 外键引用）；fee/tax/rounding 各有显式 `account_class`，FX 舍入差额记入 rounding 账户使每笔兑换按 currency 逐项守恒；逻辑表目录与示例 DDL 同步（`reversal_of`、`fx_snapshot_id` 必须同时出现在两处）。
- ledger 保留、去标识化、法定义务和访问审计政策。

### 退出门禁

- 迟到 confirm 不能越过 compensating/completed。
- reserve/confirm/compensate 重放保持资源守恒。
- 充值后消费再拒付、部分退款、跨币种兑换均有可审计闭环。

## 12. M7：恢复、发布与容量准入

### 交付

- 签名 immutable release manifest、独立信任根和原子 release switch。
- `restore_runs`、唯一 execution lease、逐步幂等 guard、超时、evidence digest 和 reopen authorization。
- `restore_runs` 单例准入（FL-32）：创建 restore run 的 admission guard 为 `INSERT ... WHERE NOT EXISTS (SELECT 1 FROM restore_runs WHERE status NOT IN (终态集合))`，配非终态部分唯一索引双保险；执行主体为 restore coordinator 的 admission 事务，插入 0 行即拒绝并返回既存 run id。
- PITR × 外部消费 reconciliation（FL-38）：STREAMS_REPLAYED 含子步骤——比对各 consumer_group `acked_seq` 与 restored heads，超前部分写入显式 divergence 清单；divergence 非空即 TRAFFIC_REOPENED 被阻塞，只能走 escalated/manual 升级路径；恢复后 stream seq 分配起点 ≥ 崩溃前最大已发布 seq + 1，禁止 seq 重用。
- 单机与多机分别声明 process crash、power loss、disk failure、replica/failover 的 durability profile；不具备的能力明确 NOT_APPLICABLE 或 NOT_PROVEN。
- 用 `容量合同-v2.3.schema.json` 生成联合 workload 结果：合同必须申报 release manifest 绑定的 `deploymentProfile` 并覆盖其 mandatory profile（requiredGateIds/StageClasses/FaultScenarioIds/ResourceCenterTypes），percentiles 携带必填 `mean`/`sampleCount`（可选 failedCount/droppedCount）供 evaluator 唯一复算 rho；由确定性 evaluator 判定所有 gate，NOT_APPLICABLE gate 仅在携带完整审批链（reason+approvedBy+审批证据 digest）时可与整体 PASS 共存。
- 判定器 mutation suite 含 IC-5 fixture：单 gateway stage + 单自命名 gate + `faults:[]` + 自填 `mMax` 的结构合法合同，evaluator 期望输出 NOT_PROVEN 且 reasonCode 指明 mandatory profile 缺失。
- 告警、自动降级和人工 runbook 明确半径；支付、管理员和安全审计失败默认 fail closed。

### 退出门禁

- 完整 restore 演练可以从签名请求走到有证据的 traffic reopen，失败步骤稳定停在 RESTORE_BLOCKED。
- 并发 restore 注入（IC-4 判据）：两个 coordinator 并发触发 restore，`restore_runs` 恰一行进入非终态；step 交叉注入被各步 `restore_id+step` 前置谓词的 next-step CAS 拒绝（影响 0 行即回滚）。
- PITR 演练强制包含"外部 consumer `acked_seq` 超前于恢复 cut"fixture（IC-3 判据）：断言 divergence 清单非空即 reopen 被阻塞、恢复后 seq 不重用（分配起点 ≥ 崩溃前最大已发布 seq + 1）。
- reconnect_storm × authz cache miss 联合注入（IC-6 判据）：两 fault 必须联合注入而非分别注入，声明 SLO 内 stale-approve 计数 = 0，前台 SLO 不崩。
- release rollback 回到另一份完整兼容 manifest，不出现前后端/config/schema 混装。
- 目标并发档位的所有 correctness gate 为 PASS；任一字段或证据缺失保持 NOT_PROVEN。
- burst/fault 后 backlog 在产品 deadline 内排空，前台 SLO 与最终 checksum 同时满足；drain 证据须通过 finalState 守恒计数复算（cleanup/delete 计数为 0）。

## 13. 发布决策规则

| 决策 | 必要条件 |
|---|---|
| 允许继续开发 | 当前阶段测试通过，上一阶段不变量持续成立 |
| 允许内部单人测试 | 基础正确性 smoke 通过；未证明能力明确标注，不对外承诺 |
| 允许小范围外测 | C/D/R 全部适用 gate PASS；容量合同对应实际 release/hardware/workload；restore 演练有效 |
| 允许扩大容量 | 新档位联合负载重新测量，不能从低档线性外推 |
| 必须停止/回滚 | 任一守恒、fencing、receipt、cursor、restore、release identity 或最终 checksum 失败 |

## 14. 明确不做的捷径

- 不以增加 PM2 worker、机器或 busy timeout 代替单写者与事务证明。
- 不把当前 `owner_locks` 改名后宣称完成 epoch fencing。
- 不以一组平均 QPS/p95 宣称容量通过。
- 不删除尚未实现的健康或恢复门禁来获得绿灯。
- 不在生产运行目录执行手工 Git checkout、数据库临时修补或逐文件回滚。
- 不把流程文档、表名或测试数量当作可运行机制本身。
- 不以清理积压 job/receipt 制造 drain 通过（IC-2 教训）：容量合同 finalState 的 cleanup/delete 守恒计数必须为 0 且由 evaluator 以 accepted-to-terminal 守恒方程复算，不信任 `backlogDrained` 布尔值。
- 不以自选 scope 的容量合同满足发布门禁（IC-5 教训）：合同的 stage/gate/fault 覆盖必须满足 release manifest 绑定的 deploymentProfile mandatory profile，自命名单 gate、空 faults 的结构合法合同一律 NOT_PROVEN。
