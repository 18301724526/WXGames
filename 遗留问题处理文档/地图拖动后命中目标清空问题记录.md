# 地图拖动后命中目标清空问题记录

记录日期：2026-06-14

## 现象

账号 `test1` 导出的前端操作日志 `wxgame-oplog-test1-20260613-204928Z.json` 显示：

- `20:49:15`，`/game/state` 返回 `worldMap.tileCount=39`、`territoryCount=6`。
- `seq 7048`，世界地图运行时同步到 `baseHitTargetCount=45`、`hitTargetCount=45`。
- `seq 7054`，第一次拖动开始时仍有 `hitTargetCount=45`。
- `seq 7057`，拖动移动后立刻同步为 `baseHitTargetCount=0`、`hitTargetCount=0`。
- 后续点击只命中背景 `worldMapDrag`，runtime 内部 `worldMap:tapHit` 记录 `action=null`、`hitTargetCount=0`，所以城池点击和地块行军 HUD 都不会出现。

## 结论

这是前端 Canvas 世界地图 runtime 的命中目标生命周期问题，不是后端世界状态丢失，也不是多人同步权威状态问题。

后端在该日志段只返回一次 `502` 心跳错误，随后 `/game/state` 成功返回真实地图数据。问题发生在前端拿到有效 state 之后，拖动快照刷新把 runtime 的稳定 hit target 集合覆盖为空。

## 架构边界

- 后端和多人同步仍只负责世界事实、可见性、行军任务、城市归属等权威状态。
- 前端 `WorldMapRuntime` 负责 camera、baked layer、hit target 稳定集和 HUD 坐标转换。
- renderer 可以发布本帧绘制产生的 hit targets，但 snapshot/actor-only/快速拖动帧不一定包含完整地图 hit targets。
- 完整渲染可以提交空 hit target 集合，因为地图真的不可渲染或状态清空时必须能清空交互面。
- snapshot/拖动帧遇到空 hit target 集合时，不得覆盖已有稳定目标，只能保留上一帧稳定目标并应用当前 drag offset。

## 修复策略

新增 `WorldMapRuntimeHitTargetPolicy`：

- `collectRendererHitTargets(renderer)`：只收集 renderer 当前暴露的地图层和 actor 层 hit targets。
- `resolveBaseHitTargets()`：纯策略决定是否提交本帧 source targets。`preserveOnEmpty=true` 的 snapshot/拖动帧遇到空地图层时，不把稳定地图/城池/地块 targets 清空；如果 actor 层有最新 targets，则替换上一帧 actor targets，避免行军单位点击框残留。

接入点：

- `WorldMapRuntime.syncHitTargetsFromRenderer(options)` 使用该 policy。
- `WorldMapRuntimeRenderPipeline` 的 snapshot render path 传入 `{ preserveOnEmpty: true }`。
- `CanvasGameShellWorldMapLayerBridge.refreshWorldMapLayerFromSnapshot()` 和 actor-only refresh 传入保留策略。
- `CanvasGameAppWorldMapRuntimeBridge` 的 snapshot refresh path 传入保留策略。

## 性能约束

- 保留策略是 O(n) 分层数组拷贝，不触发网络请求，不重算地图，不生成世界数据，不参与多人同步权威状态。
- 拖动路径仍然优先使用已有 `baseHitTargets + dragLayerOffset`，避免在 H5 拖动中做全量重渲染。
- 30fps 基准和高配 60fps 的目标不依赖每次拖动都重新构建 hit targets。
