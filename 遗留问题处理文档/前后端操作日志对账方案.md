# 前后端操作日志对账方案

## 背景

大地图拖拽、点击城池、选择行军目标等行为大量发生在前端本地状态内。后端现有 `api_logs` 只能看到已经发到服务器的请求，无法解释一次点击为什么没有打开城池 HUD、为什么没有出现行军 HUD、或者点击是否被拖拽判定吞掉。

## 当前代码事实

- 前端新增 `ClientOperationLog`，位于 `frontend/js/debug/ClientOperationLog.js`。
- `ClientOperationLog` 只记录在浏览器本地内存和 `sessionStorage`，不会主动向服务器发送日志。
- 前端记录入口包括：
  - `H5CanvasInputController`：记录 pointer down/up、tap、drag move 采样，以及 tap 是否被拖拽/抑制吞掉。
  - `CanvasGameShellInputRouter`：记录 tap 命中的 action、runtime 是否处理、disabled/tutorial/空命中等分支。
  - `WorldMapRuntime`：记录大地图 hit target 同步、背景点击推导出的行军目标、拖拽开始/结束和 camera 摘要。
  - `CanvasActionController`：记录 action begin/end/error 和关键 UI state 摘要。
  - `GameAPI`：记录已有 API 请求/响应/错误的 requestId、path、action 和状态，不新增请求。
- 后端继续使用本地 `api_logs`，并在 body 内附加 `operationLog` 摘要。
- `ObservabilityService` 的 recent events 现在保留 `requestId` 和 action target 摘要。

## 对账方式

1. 前端日志用 `window.ClientOperationLog.getRecent()` 或 `window.ClientOperationLog.exportText()` 查看。
2. 后端日志用现有玩家日志接口或 SQLite `api_logs` 查看。
3. 通过 `GameAPI` 自动生成的 `X-Client-Request-ID` 对齐两端：
   - 前端事件里是 `requestId: api-N`。
   - 后端 `operationLog.requestId` 或 observability `requestId` 中是同一个值。
4. 如果前端有 `input:tapHit`、`action:begin`，但没有 `api:request`，说明动作停在前端本地。
5. 如果前端有 `api:request`，后端没有对应 requestId，说明请求没有到达服务器或被网关/连接层截断。
6. 如果两端都有 requestId，但结果不同，看后端 `statusCode`、`operationLog.success`、`operationLog.error`。

## 性能边界

- 高频 `dragMove` 和 camera 变化做采样记录。
- 日志窗口有上限，默认保留最近有限条目。
- 前端不主动上传操作日志。
- 后端只记录已有请求和响应，不改变业务执行路径。

## 典型排查目标

- 点击城池没有打开 HUD：看 `input:tapHit` 是否命中 `openWorldSite`，再看 `action:end` 后 `selectedSiteId` 是否变化。
- 点击地图没有出现行军 HUD：看 `worldMap:backgroundTarget` 是否推导出 `selectWorldMarchTarget`，再看 `action:end` 后 `worldMarchTarget` 是否存在。
- 拖动后疑似坐标错乱：看 `worldMap:dragEnd` 的 camera、bakedCamera、dragLayerOffset，以及后续 `worldMap:hitTargetsSynced` 的 hit target 数量。
- 轻点被判拖拽：看 `input:pointerUp` 中 `skipTap` 和 `dragMoved`。
