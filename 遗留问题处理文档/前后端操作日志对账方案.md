# 前后端操作日志对账方案

## 背景

大地图拖拽、点击城池、选择行军目标等行为大量发生在前端本地状态内。后端 `api_logs` 只能看到已经发到服务器的请求，无法解释一次点击为什么没有打开城池 HUD、为什么没有出现行军 HUD、或者点击是否被拖拽判定吞掉。

因此操作日志必须同时满足两件事：

- 日常运行时不把每一次点击、拖拽都变成服务器通信。
- 线上复现后必须能显式导出或上传一次诊断快照，让工程侧按账号、时间、requestId 对账。

## 当前代码事实

- 前端 `ClientOperationLog` 位于 `frontend/js/debug/ClientOperationLog.js`。
- 默认记录在浏览器内存和 `sessionStorage`，不会因为点击、拖拽、行军选择自动上传服务器。
- 前端可以通过 `window.ClientOperationLog.getRecent()` 查看最近日志。
- 前端可以通过 `window.ClientOperationLog.exportText()` 导出本地 JSON 文本。
- 前端可以通过 `window.ClientOperationLog.upload({ reason: '复现说明' })` 显式上传一次诊断快照。
- `frontend/app.js` 在 `GameAPI` 创建后注册上传器：`ClientOperationLog.upload()` 会调用 `GameAPI.uploadClientOperationLog()`。
- `GameAPI.uploadClientOperationLog()` 调用 `POST /api/client-operation-logs`，带 `Authorization` 和 `X-Client-Request-ID`。
- 后端 `LogService` 维护 `client_operation_logs` 表，保存玩家、设备、原因、条目数、完整诊断 payload 和时间。
- 后端 `GET /api/client-operation-logs?limit=N` 可以查询当前登录玩家最近的诊断快照。
- 普通业务请求仍写入 `api_logs`，并在 body 内附加 `operationLog` 摘要。
- `ObservabilityService` 的 recent events 保留 `requestId` 和 action target 摘要。

## 前端记录入口

- `H5CanvasInputController`：记录 pointer down/up、tap、drag move 采样，判断 tap 是否被拖拽判定吞掉。
- `CanvasGameShellInputRouter`：记录 tap 命中的 action、runtime 是否处理、disabled/tutorial/空命中等分支。
- `WorldMapRuntime`：记录大地图 hit target 同步、背景点击推导出的行军目标、拖拽开始/结束和 camera 摘要。
- `CanvasActionController`：记录 action begin/end/error 和关键 UI state 摘要。
- `GameAPI`：记录已有 API 请求/响应/错误的 requestId、path、action 和状态。

## 对账方式

1. 玩家复现后，显式上传前端日志：

   ```js
   window.ClientOperationLog.upload({ reason: 'city-click-no-hud-after-drag' })
   ```

2. 后端查询该账号最近的前端诊断快照：

   ```http
   GET /api/client-operation-logs?limit=5
   ```

3. 后端查询普通 API 日志：

   ```http
   GET /api/player/logs
   ```

4. 通过 `X-Client-Request-ID` 对齐两端：

   - 前端事件里是 `requestId: api-N` 或 `client-oplog-N`。
   - 后端 `operationLog.requestId` 或诊断快照 `requestId` 中是同一个值。

5. 如果前端有 `input:tapHit`、`action:begin`，但没有 `api:request`，问题停在前端本地状态或动作分发层。
6. 如果前端有 `api:request`，后端没有对应 requestId，问题在网络、网关或请求到达前。
7. 如果两端都有 requestId，但结果不同，看后端 `statusCode`、`operationLog.success`、`operationLog.error`。

## 性能和边界

- 高频 `dragMove` 和 camera 变化做采样记录。
- 日志窗口有上限，前端默认保留最近有限条目。
- 只有显式调用 `upload()` 才上传诊断快照。
- 上传接口只保存诊断数据，不改变业务状态。
- `POST /api/client-operation-logs` 不写入普通 `api_logs`，避免大 payload 重复存储。

## 典型排查目标

- 点击城池没有打开 HUD：看 `input:tapHit` 是否命中 `openWorldSite`，再看 `action:end` 后 `selectedSiteId` 是否变化。
- 点击地图没有出现行军 HUD：看 `worldMap:backgroundTarget` 是否推导出 `selectWorldMarchTarget`，再看 `action:end` 后 `worldMarchTarget` 是否存在。
- 拖动后疑似坐标错乱：看 `worldMap:dragEnd` 的 camera、bakedCamera、dragLayerOffset，以及后续 `worldMap:hitTargetsSynced` 的 hit target 数量。
- 轻点被判拖拽：看 `input:pointerUp` 中 `skipTap` 和 `dragMoved`。
